- SRP（OO的第一步是实现类的SR）
    - 决定把什么放到一个类里面去
        - 这是一个组织问题？
        - 需要容易修改，废话
        - 又说一遍设计定义，相比实现的多么完美，保持可变性更重要
    - 创建符合SRP的类
        - 为啥SRP重要
            - 容易修改的程序由容易复用的类组成，使用的时候就是把这些类像搭积木一样搭起来（为啥）
            - 一个类有很多职责就不容易复用，主要关注点在于类的各个职责交织在一起存在风险：不能只考虑这部分不考虑另外一部分
            - 有两种方式面对这样的问题：都不够有吸引力
                - 复制一份
                - 封装下这个类，暴露出你需要使用的方法，这个只是并没有解决根本问题
            - 太多职责就会导致修改的可能性比较多，增加修改的时候把依赖他们的代码被改坏
        - 怎么判断是否符合SRP
            - 把它的每个方法转成这样的一个问句？『hi，类xxx，什么是的xxx方法名』，通过拟人的方式判断这个方法是否和类xxx的职责，看这个说不说的通-- **也就是问他这个是干什么用的**
            - 用一句话表达类的职责，如果无法表达说明内容太多了
        - 什么时候开始做设计的决定
            - 权衡点：如果今天不做代价大不大
                - 一个类的设计不符合上面好的设计标准，但是以后不会有人用了，现在可以不管，等到新需求出来的时候再考虑
                - 怎么考虑呢
                    - 会不会被复用呢（也就是依赖）？太多职责，复用比较困难
                    - 代码会不会被效仿呢？如果烂代码被效仿了那不是尴尬了
    - 编写拥抱可变性的代码
        - 依赖行为，不依赖数据（行为就是message呗）
            - 隐藏类中的实例变量，也就是说引用的时候通过方法引用
            - 隐藏数据结构：数组、map，封装他们，提供出具体的行为，让数据、map只在一个地方出现，DRY
        - 强制每个地方使用SR
            - **好的设计不是一开始就知道，可以通过好的实践发现一个好的设计。其中之一就是把每个地方都保持SR---经典**
            - 抽取方法中额外的职责：例如：迭代+函数，可以抽出一个函数来
                - 暴露之前隐藏的职责，让类变得更加清晰
                - 避免需要注释，如果需要那就抽出一个方法来
                - 鼓励复用
                - 方便移动到其他类中
            - 抽出类中额外的职责
                - 根据抽取方法中的额外职责，可以发现类的额外职责，把这些职责挪到其他地方去，不要让他们干扰原来类的职责
                - 怎么决定把这个职责封装在一个新的类中？
                    - 考虑可见性问题
                        - public别人都会用到，相当于在整个应用级别创建新的概念
                        - 只对当前类可见是最理想的
                - 只有当整个应用的上下文中有需要的时候才会真正定义出这个类
- 管理依赖（依赖于SRP）
    - 为啥会有依赖
        - 把所有的都放在一起，大脑hold不住，所以要任务分解。分解成不同的对象，不同的对象负责不同的行为，通过消息来通行。这里消息就是调用其他对象的行为。
        - 所以管理依赖的前提是 **定义好每个对象的行为**
    - 理解依赖
        - 依赖场景
            - 依赖其他类：名称
            - 依赖其他类的消息名称（或者方法）
            - 依赖其他类的消息的参数类型
            - 依赖其他类的消息的参数顺序
            - 链式依赖，违反了LoD，怎么办？
            - 测试依赖代码太紧了，导致代码一改测试也跟着改，怎么办？
        - 对象直接的耦合
            - 根据上面的场景依赖的越多耦合越重
            - 越重越像一个东西，考虑合在一起吧
            - 分还是合是个问题？
            - 定义好职责，是不是就解决了一大半？
    - 编写松耦合的代码（少依赖的代码）
        - 依赖注入
            - 设计好依赖的消息，然后具体的依赖构造的时候注入
        - 隔离依赖：在无法避免依赖的情况下，把依赖的访问缩小在一个较小的范围，使得依赖变化的时候，受影响的范围最小化
            - 隔离对象的创建：隔离对类的名称的依赖，也就是把创建对象放到一个单独的函数或者一个类中，factory
            - 隔离容易变化的方法的调用：把对具体某个对象的调用放到一个方法中，然后调用这个方法，同时这个方法和调用方要有相近的语义
            - 移除参数顺序的依赖
                - 参数对象，关键字参数，显示定义默认值（常量表示出来）
    - 管理依赖的方向（按稳定性排序，通过依赖反转改变依赖顺序，做到只依赖比自己稳定的）
        - 依赖反转
        - 依赖选择的原则： **依赖比自己少修改的**
            - 有些类比另外一些类因为需求的修改而修改的情况来的少
            - 具体类比抽象类更容易修改
            - 修改一个被很多类依赖的类会导致修改的扩散
        - 需要理解修改可能性
            - 语言特性>语言的标准库>框架>自己的代码（自己的代码也会有排序）
        - 识别抽象和具体类
        - 避免依赖有很多职责的类
        - 知道重要的依赖
            - 被很多类依赖
                - 修改可能性小，做抽象
                - 修改可能性大，要避免
            - 被很少类依赖
                - 没关系
- 创建弹性接口（怎么暴露自己的接口）
    - 尽管你做到了srp，做到依赖注入，还需要决定我暴露的接口，不然还是会存在各个类交织在一起的问题，导致类难以复用
    - 暴露的接口特点v私有的接口特点
        - 揭示类的主要职责v实现类的职责
        - 被外部调用v不被外部调用
        - 不会随意改变v随时改变
        - 被依赖是安全的v被依赖是不安全的
        - 测试case中会直接引用到v测试case中不会调用（测试共有方法）
    - 如何找到定义公有的接口
        - 一门艺术
        - 使用时序图表达业务对象和对象之间的消息，重点是消息
        - 有经验的人可以从测试入手，其实就是tdd
        - 消息应该是表达what，而不是how（本质还是和职责相关的名词）
        - 寻找类的无关上下文，类的上下文指类知道的其他对象的事情，通过尽量减少上下文可以识别更有意义的接口
        - 使用消息发现新的对象，通过判断消息和已有的对象是否匹配，如果不匹配说明需要新的对象来承载这个消息
        - 构建基于消息的应用
    - 创建接口的一些原则
        - 显式接口
            - 多谢表达what，而不是how
            - 选择一个尽量不会变得名字
            - 参数个数少，精炼
        - 只使用其他对象的public方法
        - 最小化上下文
            - 别人使用你的接口的时候，尽量减少使用时的其他信息
            - 使用有问题的接口时，把它的问题封装在一个单独的地方，让自己的收到最小的影响
    - LoD（使用public接口的误区）
        - a.b.c.d不是a.a.a
        - 本质是知道了对象的实现，改变成本高，不容易推导，不容易复用，不是好的代码模式
        - 通过delegate来避免违反LoD
        - 不是铁律，取平衡，修改的成本和移除的成本
        - 如果违反了，可能是某些对象缺少了消息
    - 接口是消息的实现，特征是稳定，代表类的职责，并用最小的成本提供最大的收益。
    - 关注消息能够揭示出那些不被重视的对象。
    - 接口应该提供what而不是how
- duck type
    - 上面说的都是单个类的接口，也就是说在使用的时候你要知道具体某个类或者对象，是不是可以做到使用的时候不用知道呢，不知道的话，使用的地方就不用随着使用类的变化而变化了，那就是接口了
    - **一个思维还是基于消息（行为）的设计，不是看哪个类，而是行为，从行为到类**
    - 不就是多态么
    - 一些可能需要的地方
        - 需要反射的地方
        - 需要知道类型的地方
- 如何使用继承
    - 本质：知识复用，不是代码复用
        - 一般来说是这样的场景：父类抽象出了通用的逻辑，但是不同的子类有部分特化
    - 有个问题比较好
        - 如果我做错了，将会发生什么？
        - 一个决策完成至少考虑两点成本：
            - 实现成本
            - 错了以后，修改成本
    - 识别什么时候用
        - 类中有一些表示不同类的概念的变量，比如style，type，category等等，并且会用他们来做条件来执行不同的逻辑，同时其他行为大部分一致，这样时候可以考虑继承
        - I know who you are, so I know what you do?当需要这样的时候，就该考虑继承了，去避免这样的问题
    - 怎么做：
        - 从具体中提升抽象，而不是具体中下沉具体，因为后者会容易把具体的内容当作抽象的
        - 抽象或者继承的设计应该让子类容易实现
        - 如果只有两个类的时候不值得开始做继承
    - 场景
        - 模板设计模式
        - hook message，从语言机制上都是利用了overwrite特性（golang没有）
- 通过模块共享角色行为
    - golang的接口和嵌入式字段
    - 怎么写好继承的代码
        - 识别反模式
            - 对象中使用了名称类似type/category的变量，用来判断给自己发送哪个消息。难维护呗，使用继承吧
            - 给目标对象发送消息的时候，需要知道目标对象的类型，然后发送具体的消息。使用duck type吧
        - 坚持抽象
            - 父类中不能有这个子类用这个方法不用这个方法的情况
            - 子类中不能出现一个方法，这个方法是抛出一个不需要实现的情况
            - 以上两种情况，都说明子类和父类的抽象不一致了
        - 遵循里式替换原则，否则就不是is-a关系
        - 使用模板方法的设计模式
        - 避免强制子类需要调用父类某个方法的情况，这样会导致子类依赖父类的行为，本来继承就是为了能够自动把消息委托给父类，这样一来这个好处就没有了。除非你是在使用decorate设计模式。
        - 创建浅、窄的继承结构
- 使用组合构造对象
    - 结论
        - 优先组合，大部分情况都考虑组合，只有在非常确定的情况下考虑继承
    - 继承
        - is a关系
        - 抽象的概念不会变多
        - 自动的把消息委托给父类，代价是有类层次结构
        - 好的继承
            - 性价比很高，因为越高层的类，修改影响越大，因此只需要修改这里就能获得很大收益
            - 复用性强，OC原则，通过子类的继承就能扩展逻辑
            - 可以效仿性强，容易复制、容易理解
        - 坏的继承
            - 原因
                - 错误理解问题
                - 使用者使用的时候需要基础你的类
    - 组合
        - has a关系
        - 一个对象包含很多部分，但是不仅仅是这些部分之和，音乐的比喻很棒，很多音符，但是合在一起以后可以表达情感
        - 抽象概念比继承要多
        - 收益和继承刚好相反
    - duck type
        - behaves like a
        - 不同的对象扮演相同的角色。
        - 识别方法：
            - 一个对象扮演这个角色，但是不是主要职责
            - 有很多不想干的对象也需要扮演同样的角色
