# grokking simplicity：taming fp
- 问题
    - 本质还是任务分解么，按照把复杂和简单的分离，避免简单的逻辑复杂化，data比calculation简单，calculation比action简单
    - 效率问题:
        - 作者说拷贝是比较省事的,浅拷贝+深拷贝.
- FP视角下，代码中的核心concept：
    - action
        - 执行效果依赖次数，时间
        - 怎么做
            - 如何保证执行顺序，执行一次，有序
            - 最少化
                - 使用不可变的数据结构或者Copy on Write实现不可变,让读可变数据变成读不可以变数据,这样就变成了calculation
                - 避免调用外部函数或者被外部函数调用的时候改变数据的不变性，调用的时候参数做拷贝，返回的时候返回值做拷贝
            - 限制action只和外部交互
            - 最小化
            - 简化对时间和顺序的依赖
    - calculation
        - 只和输入有关系
        - 怎么做
            - 静态分析，数学工具，测试策略
    - data
        - 记录事件的数据，每次事件一份数据（性能堪忧）
            - 怎么做
                - 如何组织数据使得访问高效，记录长时间不坏的规则，扑捉主使用数据的重点的原则
- FP的思维
    - 区分三个概念
    - 函数抽象
- 分层模式
    - 直接的实现
        - 函数签名表示要解决的问题,实现(解决问题)的时候使用相同抽象层次的代码(语言特性/通用业务逻辑/特定业务逻辑等等), <font color=#DC2D1E>**是不是说不要出现跨层**</font>
        - 签名表达意图
        - body使用相同的抽象函数，通过调用图检测
        - 另外不要信息过载
        - 通过函数调用图可以发现问题:(每个函数处于什么层,函数的调用关系代表层之间的调用)
            - 层之间的交互问题?
                - 跨层调用?
                - 通过全局视角查看
            - 层的实现问题?
                - 通过层视角来看
                - 发现那函数
            - 函数的实现问题?
                - 通过函数视角来看
                - 调用的函数是不是都在同一层
                - 是不是有混合基础的语言特性和函数调用.
                    - 抽取函数,包含基础的语言特性
                    - 还是有些...
    - 抽象边界
        - 好处：
            - 应对不确定性
            - 减少信息负载
                - 关注主要问题
                - 减少模块耦合
                - 代码容易读写
    - 什么时候停止抽象，当你觉得OK的时候，这个地方需要品位了
- timeline
    - 画法基础
        - 有先后顺序的action放在一个时间线上
        - 同时发生或者没有现有顺序关系的action放在不同的时间线上
    - 怎么画
        - 识别action
            - 需要考虑执行环境的线程模型
        - 画出每个action
            - 顺序执行，还是并行执行。顺序的画在同一个时间线下，乱序或者在不确定某个时间执行的画在新的时间线下
        - 简化
            - 中间不会插入其他action的，合在一个框框里面
            - 如果一个时间线执行完以后，多出一个时间线，那么可以合并这两个时间线
            - 虚线表示不同时间线的顺序
    - 作用：方便查看一段代码多线程执行时的不同顺序，找到可能不符合预期的执行顺序（bug）
    - 原则
        - 时间线越少越好
        - 时间线越短越好
        - 不同时间线共享的资源越少越好
        - 有资源共享时，需要协调不同时间线访问资源的顺序
            - 锁、队列
        - 把控制时间作为一等概念
            - 控制时间线，cut，waitgroup，once等等
- 架构
    - reactive arch
        - 解耦原因和效果：也就是原本我因为很多原因修改了一个数据，而有一些逻辑依赖这个数据的变化，因此也需要触发这些逻辑。结果就是如果有很多地方修改了这份数据，这些逻辑都要调用一遍，响应式做法是封装这份数据，让这些逻辑监听这份数据的变化，相当于解耦了原因（触发数据变化的操作）和效果（依赖逻辑），因此也就不用每个修改数据的地方都得调用一遍那些依赖这份数据的逻辑
            - 观察者模式
            - 坏处：解耦以后不直观，不知道哪些逻辑依赖这份数据
        - 把操作步骤当作流水线（极限）
            - 参考js里面的promise
        - 简化时间线，一个时间线的action变少了，但是时间线可能变多了。同时，共享变量的情况减少了
        - 定义：反转“先做x，后做y”，“当x发生时，出发y”
        - 极限是pipeline，每个操作作为流水线一环，一部一部出发
    - 洋葱架构
        - 交互层
            - 对外提供的接口，对内使用的基础设施，action
        - 领域层
            - 尽量是做到calculation
        - 编程语言层
        - 权衡点
            - 可读性
            - 性能：技术债
            - 紧急业务：记得技术债
        - 和函数式很契合，高阶函数天然支持装饰者模式
- 最重要的三个点
    - 尽可能的从action中抽取calculation
    - 使用高阶函数做抽象
    - 封装并发的代码，不同时间线中的action，明确他的语义，包括不同顺序执行的语义和多次执行的语义。
