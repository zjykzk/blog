# unit testing principles practices and patterns
- 单侧目标（why）
    - 软件项目的 <font color=#75C940>**维护性扩展性**</font> 发展，迭代
        - 为啥：项目越大，代码建模的概念越多，概念之间的关系也越复杂，如果记不住，所以需要工具把它记录下来
    - 副作用：代码质量的强负反馈，不容易测试说明代码太耦合了，但不是强正反馈，容易测试不代表代码质量高，太多抽象实现耦合性低有时候也会提升认知负担
    - 成本，不是越多越好，代码负债，不是资产
        - 修改代码会修改测试
        - 代码修改以后跑测试
        - 单侧错误失败
        - 维护单测
    - 那么什么的单测是好的
        - 代码覆盖率
            - 行，分支覆盖率都不是越高越好，高覆盖率不保证：
                - 验证所有结果
                - 所有调用的外部库返回的结果都会处理
            - 它是一个强负反馈，低了说明很多代码没测试，比较危险，弱的正反馈，高了不一定好
            - <font color=#75C940>**所以，对核心代码保证高覆盖率，不定义一个高统一标准**</font>
        - 好的几个性质
            - 集成在开发过程中
            - 只针对重要的代码
                - 代码按照分类主要是
                    - 基础代码
                    - 外部服务，依赖代码，比如：数据库
                    - 胶水代码，比如：组装代码
                - 重点是基础代码，特别是领域模型代码
            - 用最小的 <font color=#DC2D1E>**维护成本**</font> 获取最大价值
                - 识别有价值的测试和低价值测试：四个原则
                - 写有价值测试：需要写代码技巧
- 什么是单测（what）经典学派，底特律学派
    - 验证一个行为单元，或者功能单元
        - 伦敦学派：代码单元，一个类，一个方法
    - 快速执行
    - 和其他测试隔离
        - 为什么会产生这个问题，是因为有依赖，三类
            - 共享依赖：各个测试共享，会互相影响
            - 私有依赖：不共享
                - 又区分：可以修改和不可修改
            - 进程外依赖：
                - 和共享依赖会有重叠，比如数据库
            - 共享依赖和进程外依赖比较像，进程外但是不能修改，但是都需要mock
        - 经典学派只会替换共享依赖，而伦敦学派还会替换可变的私有依赖
        - 通过test double 来隔离依赖，来自电影替身演员
        - 伦敦学派的最大问题是和实现细节耦合，在架构比较明确的情况下使用比较好，比如数据库依赖
    - 满足不了这些的性质的就是集成测试
        - 端到端测试是指集成了全部或者大部分共享依赖
    - 结构
        - aaa结构，arrange，act，assert
        - assert要精简，太多说明抽象不够
        - 用sut来命名需要被测试的对象
        - 命名使用业务行为的语义命名，所以不要包含被测试的函数名，另外是测试的是基础代码，不是业务代码
        - 使用可以提升可读性的assert库
- 怎么区分什么是好的测试，单测，集成测试，端到端测试
    - 保护代码， <font color=#75C940>**发现修改问题**</font> ，取决于
        - 测试执行到的代码量
        - 以上代码量的复杂度
        - 以上代码的领域含义的大小
        - <font color=#75C940>**结论是：执行尽可能多的代码，覆盖率**</font>
    - 重构代码时，不用修改测试代码， <font color=#75C940>**避免误报**</font> ，所以项目开始的时候不明显
        - 如果不满足就可能存在误报
        - <font color=#75C940>**结论：测试不要依赖实现细节，mock的时候特别注意**</font>
        - <font color=#75C940>**备注：这个没得权衡，要么是，要不是，必须得是，都失败了，还不解决吗**</font>
    - 快速反馈
    - 容易维护
    - 总结
        - 乘法关系，只要一个很坏，那就是坏，呼呼
        - 前面三个性质不能同时兼得
        - 金字塔测试，越往上前面两个越好，后面两个越差
- 怎么写出好的单测
    - 从代码划分视角看，先把生产代码做划分，然后写业务相关的测试
- mock 什么时候用
    - 维护成本高，稳定（需要被观察到）外部共享依赖，eventbus
    - 数据库不是，不不是文档外面依赖，是实现细节，尽量稳定了
    - 有时候还是spy比较好
- stub什么时候用
    - 获取数据的依赖场景，不会校验参数
    - 不用严格准守，本质还是那几条性质
- 三种测试风格
    - 基于输出测试：最优，发现问题，重构不修改视角看，可维护性都最好
    - 基于状态测试
    - 基于交互测试：最差，三个视角都最差
- 金句
    - oo通过封装可修改的操作让变得易懂，fo通过最少化可修改操作让代码变得易懂
- 函数式架构和洋葱架构
    - 函数式是洋葱的极限情况
        - 领域层不允许有修改
        - 领域层之外的是修改shell
        - 缺点： <font color=#DC2D1E>性能不好</font> 修改要复制，为了维护不可变性 <font color=#DC2D1E>基础代码比较大</font>
- 代码划分
    - 外部视角（怎么测试）
        - public，private api
        - 外部可观察行为和实现细节
        - public api和可观察行为保持一直
    - 内部视角（怎么写测试）
        - 领域代码，富含业务知识，业务复杂度告，这个代码的测试最有价值
        - control 胶水代码，包含比较多的合作组件，组件交互多，编排流程代码
        - 尽量让代码留在这两个方向
            - 两者都少的代码不用测试，收益太少了
            - 两者都有的代码要拆分，太复杂了，测不动
    - controller怎么写的权衡点
        - 领域模型容易测试
        - 牺牲controller的简洁性
        - 保护性能
- 集成测试
    - 涉及进程外依赖的测试？
        - 被管理的：拥有全部控制权，数据库
        - 没有被管理的：消息队列
    - 测试的内容是胶水代码，controller部分代码
    - 测试内容就是进程之外的依赖是否处理正确，参数和返回值，就是协议了
    - 测试范围
        - 测试最长的happy path，覆盖尽可能多的依赖
        - 补充单元测试不到的地方
    - 怎么mock
        - 最佳实践
            - 只控制之外的依赖
            - 这个依赖应该一般是最外层的依赖
            - 单测不要用mock
            - 记得校验函数调用次数和不被调用的函数
            - 只mock自己的类型
    - 最佳实践
        - 明确领域模型的边界
        - 减少应用的层数，测试的代码会很碎
        - 避免循环依赖，增加认知成本
- yagni vs ocp
    - 有抽象才需要考虑ocp
    - 机会成本：将来会用到，不确定比较大
    - 代码越少越好
- 反模式
    - 单测私有方法：再抽象下
    - 为了测试暴露私有状态
    - 泄露领域知识到测试中：测试中执行领域逻辑，为了验证结果
    - mock具体的类
