# 编程大师访谈录
- 尽可能的准确定义系统和外界之间的接口，以及系统自身各个主要部件之间的接口。
- 把问题的解决方案组织成容易操控的结构，其中每个组件都可以用简单的方式说明。
- 学习数学，思考，写作，阅读
- 动手前深思熟虑
- 不要迷恋一个想法，固守某样东西
- 不要假设你知道的东西别人不知道
- 系统设计在于平衡
- 尽量推迟决定时间，不要过早绑定。设计要比你自认为需要的程度更加灵活。快速让某样东西工作起来，然后还能弃之不用。
- 从小的实验入手学习，而不是大的。
- 思考数据结构，花很长时间思考，编程前思考整个流程。程序就像机械装置，一段代码和一段代码之间的工作方式像一个齿轮一个齿轮之间的。 确定数据结构之后开始写小段代码，不断改善和检测，编码过程中修改是局部的，有问题可以马上发现。需要速度，快速执行，编辑，调试。
- 熟悉代码细节，数据结构很重要（计算机早期）
- 盖茨对代码熟悉程度要求严格
- 产品思维，从用户角度思考程序和产品。两类工作：系统设计师，透过基本的工具看到需要构建什么、需要使用哪些技术以及该如何使用（硬件变好了，需要那么抠门么）；还有一类是好的技术人员，把需要做什么很精确的定义清楚，然后交给刚刚毕业的人来做
- 佩奇对解决问题带来的成就感痴迷
- 平衡：各个模块之间保持平衡，易于维护，让代码保持简单，这是优雅的第一步
- 杰出程序员：面向用户，面向数学问题
- 易于理解，便于维护，是否简单，是否有不必要的复杂性，容易修改
- 编写容易理解的代码。
- 编程本身不是目的。
- 建立基本框架，添加新功能。
- 阅读优秀代码和书籍。
- 严谨，一致而清晰的结构。高度模块化和分层。早期确定子程序命名规范。
- 拆分注意：粒度，不大不小（额。。。），同一模块应该处理的功能一样
