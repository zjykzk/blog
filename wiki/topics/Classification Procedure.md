# Classification Procedure

分类不是把东西分堆，而是把一组混乱对象，按某个问题目标，切成若干互斥、可解释、可使用的类别。

它的核心问题不是“有哪些类别”，而是：**我为什么要分类，分类之后要改变什么理解、判断或行动。**

## Core Formula

一个可用分类至少包含五个要素：

```text
分类 = 对象 + 目标 + 维度 + 边界 + 用途
```

- **对象**：现在要分的到底是什么，例如人、需求、bug、风险、知识、系统问题或方法论。
- **目标**：为什么要分，是为了理解、决策、排优先级、管理、迁移知识、找根因，还是形成行动方案。
- **维度**：按什么标准分，例如来源、功能、阶段、层级、责任主体、因果位置、抽象程度、稳定性或行动策略。
- **边界**：每一类和另一类的判别标准是什么；如果一个对象同时像两类，按什么主规则处理。
- **用途**：分完之后能做什么；如果所有类别后续动作都一样，这个分类通常只是命名游戏。

## Principle: Classify by the Problem, Not by Surface Features

同一批对象会因为目标不同而需要不同切法。

需求如果为了分派实现责任，可以按页面、接口、数据、权限分类；如果为了排优先级，就可能要按核心业务闭环、合规风险、体验改善和技术债分类；如果为了减少返工，则应按问题定义不清、业务规则复杂、跨系统依赖强和展示为主等差异来分类。

因此分类前应先问：**我分类之后，是为了看见什么差异？**

## Common Dimensions

### 1. 层级分类

层级分类适合理解复杂系统，避免把表现层、机制层、结构层和治理层混在一起。

例如系统问题可以分为页面报错、接口失败等表现层；状态流转、并发控制、缓存策略等机制层；模块边界、数据模型、服务依赖等结构层；需求变更、发布流程、责任机制、监控反馈等治理层。

### 2. 因果位置分类

因果位置分类适合根因分析。它把触发因素、近因、中层机制、深层设计原因和组织原因区分开，防止只修这次爆出来的症状。

### 3. 生命周期分类

生命周期分类适合分析流程、项目、需求和知识生产。它按产生、澄清、设计、实现、验证、上线、反馈等阶段定位问题发生的位置。

### 4. 责任主体分类

责任主体分类适合协作和治理问题。它把问题映射到产品定义、架构设计、开发实现、测试覆盖、运维监控、业务使用或管理决策等可改变位置。

### 5. 功能分类

功能分类适合拆系统、工具和方法论。它关注每个部分在整体里起什么作用，例如定义问题、收集信息、建模分析、判断优先级、形成行动和反馈修正。

### 6. 稳定性分类

稳定性分类适合架构和知识管理。稳定概念、场景经验、操作技巧和临时信息应被不同方式保存与复用。

### 7. 抽象层次分类

抽象层次分类适合学习和表达。它把“怎么用”“有什么规则”“内部模型如何运转”“为什么这样设计”“这个思想可迁移到哪里”分开。

### 8. 行动策略分类

行动策略分类适合决策。问题可以被分成立即修复、暂时规避、继续观察、建立监控、重构治理或放弃处理等类别。

## Operating Flow

### 1. 写清对象

先写一句：我现在要分类的是 `X`。

如果对象没有被锁定，分类会滑向泛泛的目录整理。

### 2. 写清目的

再写一句：我分类是为了 `Y`。

目的决定主维度。理解、决策、责任分派、根因分析和行动优先级通常需要不同切法。

### 3. 列原始样本

不要一开始就造分类。先列 10–20 个具体案例，记录它们什么时候发生、由谁提出、为什么出现、影响了什么、是否可预防。

分类应从样本中长出来，而不是先拍脑袋定格子。

### 4. 找候选维度

观察样本之间真正有解释力的差异：变化来源、影响范围、可预见性、责任主体、发生阶段、修复成本或暴露的问题层级。

### 5. 选择主维度

一次先选最服务目标的一个主维度。多个维度可以作为标签、矩阵或后续细分，但不要把不同维度混成同一层级。

### 6. 写判别标准和处理策略

每一类至少要写清两件事：

- **判别标准**：什么情况下算这一类，边界情况如何处理。
- **处理策略**：这一类对应什么不同动作、资源、优先级或检查方式。

分类只有走到处理策略，才从“看起来有条理”变成“能改变下一次行为”。

## Quality Checks

一个分类做完后，应检查：

1. 是否尽量不重叠、不遗漏；如果天然重叠，应改成标签体系而不是强行唯一归类。
2. 类名背后是否有清楚判别标准，而不只是好听名词。
3. 不同类别是否对应不同后续动作。
4. 同一层级是否只使用一个主维度，避免把业务模块、技术组件、优先级和问题性质混在一起。
5. 粒度是否合适：太细会变成命名个案，太粗会失去解释力。
6. 分类是否允许随理解、样本和目标变化而演化。

## Common Failure Modes

- **用名词代替标准**：只说“技术类、业务类、管理类”，却没有说明判别规则。
- **混合维度**：把高优先级、页面需求、后端需求、合规需求放在同一层。
- **分类太细**：每个类只有一个样本，无法形成可复用处理方式。
- **分类太粗**：所有问题都叫“沟通问题”，没有解释力。
- **没有后续动作**：分类完只是整理得好看，但无法指导下一步。

## Descriptive and Generative Classification

分类可以分成两类：

- **描述型分类**：回答这些东西表面上有什么不同，例如前端问题、后端问题、数据问题。
- **生成型分类**：回答这些东西为什么会变成这样，例如边界定义不足、状态模型不完整、反馈链路过长、责任接口不清或历史债务累积。

更有穿透力的分类通常是生成型分类。它不是按现象分，而是按产生机制分。^[inferred]

## One-Sentence Compression

> 好分类不是把对象放进漂亮格子，而是按当前问题目标切出能解释原因、判断优先级并导出行动差异的结构。
