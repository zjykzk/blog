# Software Development Thought Lineage

这页关心的核心问题，不是某一条具体方法怎么落地，也不是现代软件工程的总定义，而是：不同软件开发思想是如何在历史上一个接一个出现、彼此修正，并逐步形成今天这组主流方法栈的。

如果要区分它和相邻页面：

- `[[wiki/topics/Modern Software Engineering]]` 提供更高层的方法论总纲
- `[[wiki/topics/Software Methodology]]` 讨论潘加宇《软件方法》这条具体方法线
- 本页讨论的是跨流派的历史脉络、问题转移与前后关系

软件开发中的主流思想，可以理解为一条不断回应复杂性失控、变化加速与验证成本上升的演化链。每一代方法都不是凭空发明新口号，而是在修正上一代方法暴露出的结构性问题。

如果把这条主线压缩成一句话：

> 软件开发思想的演化，本质上是在寻找一种方式，让系统在复杂性、变化和不确定性面前，仍然能持续被理解、被修改、被验证。

## Core lineage

可以把主干脉络粗略写成：

`软件危机 -> 结构化方法 -> OOAD -> 迭代开发 -> 敏捷 / XP -> TDD / 简单设计 / 重构 -> 测试策略体系化 -> DevOps / 可观测性 -> 现代软件工程 -> Spec-Driven Development / AI-Assisted Development`

这条线不是单一学派替换另一学派，而是问题重心不断迁移：

- 先解决开发失控
- 再解决系统结构
- 再承认变化不可避免
- 再建立快速反馈与验证体系
- 最后把开发理解成持续学习系统，并在 AI 时代把约束前移

## The recurring questions

如果把软件开发思想继续向上压缩，会发现几乎所有流派都在围绕五个母问题演化：

1. 如何分解复杂性？
2. 如何应对变化？
3. 如何验证正确性？
4. 如何确保多人协作不失真？
5. 如何建立持续学习闭环？

不同流派的差别，主要在于它们优先回答哪一个问题、默认世界是什么样、以及它们认为最有效的约束放在哪一层。

## Comparative table

| 思想 | 解决什么问题 | 核心假设 | 代表实践 | 局限 | 被谁修正 |
|------|--------------|----------|----------|------|----------|
| **结构化方法 / 瀑布** | 大型软件开发失控、缺乏流程、阶段混乱 | 软件可以像传统工程一样被前期分析、分阶段控制 | 需求→设计→实现→测试；阶段评审；文档驱动 | 过度相信前期确定性；变化成本高；反馈太晚 | 迭代开发、敏捷 |
| **结构化分析与设计** | 如何把复杂系统拆解为可理解部分 | 复杂性主要靠功能分解和流程分解来管理 | 数据流图、模块分解、结构图 | 对行为变化与领域概念支持不足；容易把系统看成过程拼装 | OOAD |
| **OOAD（面向对象分析设计）** | 代码结构与现实世界概念脱节，系统难演化 | 用对象、职责、关系、抽象来建模，能更稳定地应对复杂性 | 类、对象、封装、继承、多态、UML、职责分配 | 容易过度抽象、过度建模、僵化继承层次 | 敏捷、XP、组合优于继承、演化设计 |
| **设计模式** | 局部设计问题反复出现，经验难复用 | 某些设计问题有稳定可复用的结构解法 | Strategy、Observer、Factory、Adapter 等 | 容易被教条化，变成“为了模式而模式” | 简单设计、重构文化 |
| **架构中心主义** | 系统规模上升后需要更高层结构稳定性 | 先建立稳定架构骨架，局部实现才能可控演化 | 分层、模块边界、接口隔离、架构视图 | 容易演化成 BUFD；架构脱离真实反馈 | 敏捷架构、演化架构、现代软件工程 |
| **迭代开发** | 一次性规划错误代价太高 | 对问题的理解会随着开发推进而提高，因此必须分批逼近 | 增量交付、短周期迭代、阶段性回顾 | 仍可能停留在“按计划切小块”，不一定真正重视反馈 | 敏捷 |
| **敏捷（Agile）** | 需求变化快、前期规划经常失真 | 开发是学习过程，不是纯制造过程；要缩短反馈回路 | 用户故事、短迭代、backlog、站会、回顾、持续交付 | 容易只剩流程仪式，没有工程支撑；“快”可能变成无序 | XP、DevOps、现代软件工程 |
| **XP（极限编程）** | 团队虽然想拥抱变化，但代码结构承受不住变化 | 适应变化必须靠工程实践维持代码可修改性 | TDD、重构、结对编程、持续集成、简单设计、小版本发布 | 对团队纪律要求高；在大组织中全面落地难 | 测试策略体系、DevOps |
| **TDD** | 修改没信心、设计腐化、行为边界模糊 | 先定义行为，再实现代码，设计会在测试压力下长出来 | Red-Green-Refactor、测试先行、快速回归 | 易被误解成“测试越多越好”；对集成和系统问题覆盖不足 | 测试策略、契约测试、系统测试 |
| **简单设计（Simple Design）** | 设计过度复杂、抽象先于真实需求 | 在当前需求下保留最少但足够的结构，设计应从变化中长出 | 4 Rules of Simple Design、消除重复、提升表达性、最少元素 | 对远期结构考虑不足时，可能导致局部最优 | 演化架构、领域驱动设计 |
| **重构（Refactoring）** | 代码随着修改不断腐烂 | 设计不是一次完成，而是持续改善内部结构 | 小步修改、保持行为不变、清理坏味道 | 没有测试保护时成本高；容易变成无边界美化 | TDD、测试策略 |
| **SOLID / Clean Code** | 模块耦合高、职责混乱、难以理解和替换 | 好设计来自清晰职责、依赖控制和可读表达 | SRP、OCP、DIP、命名、函数控制、边界隔离 | 容易教条化，脱离业务与反馈语境 | 简单设计、现代软件工程 |
| **DDD（领域驱动设计）** | 复杂业务系统中，代码结构无法承载领域复杂性 | 软件复杂性核心在领域复杂性，模型应与业务语义对齐 | Bounded Context、Ubiquitous Language、聚合、领域事件 | 成本高；对小系统可能过重；容易只学术语不建模 | 轻量建模、事件驱动实践、现代产品工程 |
| **测试策略（Pyramid / Quadrants）** | 单一测试手段不足以覆盖真实系统风险 | 不同问题应在不同成本层级尽早发现 | Unit / Integration / E2E、契约测试、冒烟测试 | 容易被形式化理解成比例口号；分层边界在现实中常模糊 | 可观测性、生产验证、DevOps |
| **持续集成（CI）** | 团队代码合并后经常出问题 | 集成必须频繁进行，问题应尽早暴露 | 自动构建、自动测试、主干频繁合并 | 只解决“集成时刻”的质量问题，不覆盖部署与运行 | CD、DevOps |
| **持续交付 / DevOps** | 开发完成不等于可安全上线，开发与运维割裂 | 交付应是持续流动，运行反馈必须回流开发 | CI/CD、自动部署、监控、日志、Tracing、回滚、IaC | 容易工具化；如果没有产品与工程反馈文化，只剩流水线 | SRE、现代软件工程 |
| **SRE / 可观测性** | 系统上线后问题难发现、难定位、难权衡可靠性与速度 | 运行中的系统必须被测量，可靠性是可经营目标 | SLI/SLO、错误预算、Tracing、监控告警、事故复盘 | 容易偏运维侧，忽略需求和设计前置问题 | 现代软件工程、产品工程闭环 |
| **Lean / Flow** | 交付慢、等待多、浪费大 | 软件交付是价值流，应减少浪费、缩短流动时间 | 小批量、限制 WIP、价值流分析、快速反馈 | 过度强调流动时，可能忽视技术深度与架构负债 | DevOps、现代软件工程 |
| **用户中心设计 / UX 驱动开发** | 做出来的功能不等于用户真正需要 | 软件价值取决于是否帮助用户完成真实任务 | 用户研究、原型、任务流、可用性测试 | 易与工程实现脱节；若无状态与系统视角，容易停留在界面层 | 前端工作流、现代产品工程 |
| **前端工作流思想** | UI、状态、异步行为、反馈路径复杂，需求失真常最先暴露在前端 | 前端不是“做页面”，而是校准用户意图、界面行为、系统行为的一致性 | 从用户任务出发、状态建模、交互验证、原型测试、反馈闭环 | 若只盯前端局部，容易忽略后端、组织和业务系统约束 | 现代软件工程、跨职能产品团队 |
| **现代软件工程** | 单点实践很多，但缺少统一视角管理复杂系统 | 开发本质上是经验主义问题求解：观察、假设、验证、修正 | 小批量实验、证据驱动、反馈闭环、可观测性、跨层协同 | 抽象层高，若没有具体工程实践支撑会显空泛 | 正在吸收并统摄前面各流派 |
| **Spec-Driven Development** | AI 时代代码生成很快，但意图、边界和约束不清 | 先明确规格与验证边界，再实现，能降低高速度偏航 | Specify → Plan → Implement → Validate、spec、contract、验证先行 | 如果 spec 写得空泛，会变成新文档负担；对探索型问题未必适用 | 还在发展中，可能与 TDD / DDD / 产品 discovery 深度融合 |
| **AI-Assisted Development** | 人工编码速度成为瓶颈，但实现细节越来越可外包 | 人的价值从“逐行实现”转向“定义问题、约束、评估结果” | Prompt/spec、agent workflow、自动生成、自动验证、人机协作 | 容易制造表面正确的伪完成；没有验证体系时放大错误 | Spec-Driven、现代软件工程、验证驱动开发 |

## Major transitions

如果把上表继续压缩，可以看到六次大的思想转向：

1. 从“写代码”到“设计结构”
2. 从“前期规划”到“迭代学习”
3. 从“实现功能”到“持续可修改”
4. 从“写完再测”到“全程验证”
5. 从“开发活动”到“全链路学习系统”
6. 从“人逐行实现”到“人先定义约束，系统再生成实现”

## Grouped by problem focus

### 结构治理

- 结构化方法
- OOAD
- 设计模式
- 架构中心主义
- SOLID / Clean Code
- DDD

这些流派主要回答：系统应该怎样组织，才能不失控。

### 变化适应

- 迭代开发
- 敏捷
- XP
- 重构
- 简单设计

这些流派主要回答：当需求和认知持续变化时，怎么继续前进。

### 验证与反馈

- TDD
- 测试策略
- CI
- DevOps
- SRE / 可观测性

这些流派主要回答：我们如何知道自己没有偏航，或者已经偏航。

### 语义与意图对齐

- 用户中心设计
- 前端工作流思想
- DDD
- Spec-Driven Development
- AI-Assisted Development

这些流派主要回答：我们做出来的东西，是否真的是要做的那个东西。

## How to use this page

更有用的理解，不是去问“应该信哪一个流派”，而是先判断当前面对的问题属于哪一层：

- 如果问题是系统太乱，先看结构治理类思想
- 如果问题是需求总变，先看变化适应类思想
- 如果问题是改了就坏，先看验证与反馈类思想
- 如果问题是做出来却不对，先看语义与意图对齐类思想

成熟的软件工程，通常不是信某一个学派，而是知道什么时候调用哪一层方法，并知道这些方法之间的前后修正关系。
