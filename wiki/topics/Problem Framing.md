# Problem Framing

一个问题可以先被看作：输出不符合预期。

这个定义虽然简单，但它迫使我们先把“预期”说清楚，而不是直接跳进解决方案。

## Core elements

当前笔记里提到的问题要素包括：

- 输入
- 输出
- 限制条件
- 角色
- 障碍
- 预期

## Problem-solving direction

解决问题时，至少要看三件事：

- 核心要素是什么
- 当前差异在哪里
- 应该选择什么策略

[[wiki/concepts/Iceberg Model]] 提醒 problem framing 不要停在事件层：如果一个“输出不符合预期”的问题反复出现，就要继续追问它背后的模式、结构和心智模型。^[inferred]

## Problem structure before framework choice

[[wiki/concepts/问题解决铁三角]] 补充了一个重要顺序：遇到问题时可以启动框架思维，但框架选择必须受“问题结构”约束。源文强调结构是客观存在的，而人对结构的理解是主观逼近；因此 problem framing 的任务不是先挑一个熟悉框架，而是先问：这个问题由哪些部分构成、关系如何、现象到本质的层级是什么。

如果跳过这一步，框架会从工具变成投射：人把熟悉的结构套到问题上，而不是让问题本身的纹理约束方法论。^[inferred]
