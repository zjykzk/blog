# Prompt Frequency

一个值得单独抽出来的判断是：在语义不变时，越接近训练语料中常见表达的 prompt，模型通常越容易发挥出能力。

## Why it matters

这条判断的重要性在于，它把很多人对 prompt 的直觉反过来了。

常见误区是：为了“更专业”或“更强控制”，把 prompt 写得更复杂、更书面化、更罕见。但从文章总结的实验看，问题往往不是模型不会，而是提问方式偏离了模型最熟悉的语言分布。

所以 prompt 质量，不只是“信息够不够”，还包括“表达是不是自然”。

## Core idea

这里的核心不是“简单表达更好”，而是“高频表达更好”。

两者不能完全等同：

- 高频表达不一定更简单
- 低频表达也不一定更难

真正关键的是：模型在训练中见过多少类似表达。

如果一句话的词汇搭配、句法组织、任务表述方式更常见，模型更容易把它映射到已有能力轨道上；如果表达虽然语义相同，但非常生硬、绕或罕见，模型表现就可能下滑。

## Evidence from the article

文章总结的实验覆盖了多类任务：

- 数学推理
- 常识推理
- 机器翻译
- 工具调用

结论方向基本一致：高频表达版本整体优于低频表达版本。

更重要的是，改善不只体现在最终答案上，也体现在推理过程质量上。换句话说，prompt 的表面改写，会改变模型进入 reasoning 轨道的难易程度。

## Practical pattern

如果输出不理想，可以先做这类改写：

1. 保持任务语义不变
2. 去掉生硬、绕、过度书面化的说法
3. 改成更直接、更自然、更像真实人类常说的话
4. 比较不同表述下的结果稳定性

一个实用准则是：

- 先追求自然表达
- 再追求结构清楚
- 最后才是额外控制项

而不是一上来就堆很多术语、约束词和形式化口吻。

## Examples

下面这些例子不是在追求“更短”，而是在追求“更像自然任务表达”。

### Example 1: 摘要任务

较低频的写法：

> 请基于如下文本语料执行信息蒸馏，提炼其核心论旨、关键支撑点与潜在隐含假设，并以结构化格式予以呈现。

较高频的写法：

> 请读这段内容，帮我总结它的核心观点、主要论据，以及作者没明说但暗含的假设。

后者并没有减少任务要求，但表达方式更像真实人类会说的话。

### Example 2: 代码修改任务

较低频的写法：

> 请在不改变既有外部行为语义的前提下，对该函数实施最小扰动式重构，并显式说明修改依据。

较高频的写法：

> 请在不改变对外行为的前提下，最小化修改这个函数，并说明你为什么这样改。

这里真正保留的是约束本身，而不是那种过度形式化的包装。

### Example 3: 调研任务

较低频的写法：

> 请对该议题展开系统性调研，综合多源证据形成结论，并标注证据与判断之间的映射关系。

较高频的写法：

> 请调研这个问题，综合几个可靠来源给出结论，并说明每个结论分别是基于哪些证据。

这说明更自然，不等于更松散；完全可以自然表达，同时保持判断结构。

## Counterexamples and boundaries

这条规律不能被误读成几个极端版本。

### 不是“越短越好”

过短会丢边界。

例如：

> 帮我看看这个。

这虽然很常见，但信息不足。高频表达有帮助，前提是任务语义仍然完整。

### 不是“越口语越好”

过度口语化也可能损失精度。

例如：

> 随便搞一下，大概那个意思就行。

这类说法虽然自然，但把质量标准冲掉了。自然表达不等于放弃约束。

### 不是“不要结构”

很多任务仍然需要清楚结构。

更好的做法不是去掉结构，而是用自然语言表达结构要求。例如：

> 请分三部分回答：先讲结论，再讲原因，最后讲你不确定的地方。

这依然是结构化 prompt，只是没有用很僵硬的命令腔。

## Rewrite template

当你怀疑 prompt 太低频时，可以按下面的模板重写。

### Template 1: 任务重述

把：

> 请对 X 执行 Y，并输出 Z。

改成：

> 请帮我处理 X，目标是 Y，最后给我 Z。

### Template 2: 约束保留

把：

> 在满足 A、B、C 约束条件下进行处理。

改成：

> 处理时请注意这几点：A、B、C。

### Template 3: 输出结构

把：

> 以结构化形式返回，并显式标注各部分内容。

改成：

> 请按这个结构输出：1）... 2）... 3）...

### Template 4: 推理要求

把：

> 请展示你的推理过程并进行逐步求解。

改成：

> 请一步一步说明你是怎么得到答案的。

## Practical checklist

写完 prompt 后，可以快速过一遍：

- 这是不是我真实会对一个人这样说的话？
- 有没有为了显得专业而加入的生硬词？
- 约束是否还在，但表达已经更自然？
- 输出格式是否清楚，但没有写成官样命令？
- 如果结果不好，我是否先尝试重写表达，而不是立刻换模型？

## Implication for workflow

这条判断不只是 prompt 写作技巧，也影响 AI 工作流设计。

如果我们把 prompt 看成在塑造模型进入某种 latent-state trajectory 的入口条件，那么“自然且高频的表达”本质上是在降低进入正确轨道的摩擦。

这意味着：

- prompt engineering 不只是加规则，也包括把入口语言调到模型熟悉的分布
- 上下文管理不只是信息选取，也包括表达分布的选择
- workflow 里的 instruction 模板，不该默认越正式越好，而该默认越自然越好
