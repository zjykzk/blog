# Mechanism Models Map

机制模型地图收拢的是 wiki 里那些能回答“为什么这个结果会被稳定生成”的页面。

它不是普通思维模型清单。普通模型可能只是一个比喻、分类或判断原则；机制模型必须能说明：结果、变量、约束、激励、反馈和时间如何连在一起。核心入口是 [[wiki/concepts/Mechanism Model]]。

## 机制模型的通用骨架

```text
结果：反复出现的结果是什么？
变量：哪些因素一变，结果就会明显变？
约束：什么限制了参与者的选择空间？
激励：系统实际奖励、惩罚、容忍什么？
反馈：什么被放大，什么被抑制？
时间：结构如何累积、固化、恶化或复利？
杠杆：如果要改变结果，最值得动哪一层？
```

一句话格式：

```text
在 X 约束下，A/B/C 等关键变量通过 D 激励与 E 反馈，随着时间累积，持续生成 Z 结果。
```

## 1. 系统生成类机制

这一组模型用于从事件下潜到结构。

- [[wiki/concepts/System]]：系统不是一堆部件，而是一组关系在边界和时间中反复生成整体行为。
- [[wiki/concepts/Iceberg Model]]：把可见事件下潜到模式、结构和心智模型，防止把重复问题当成孤立事故。
- [[wiki/concepts/Feedback Loops]]：结果反过来影响原因，形成增长、稳定、振荡或失控。
- [[wiki/concepts/Leverage Points]]：越靠近结构、信息流、规则、激励、目标函数和心智模型，越可能改变生成结果的机制。
- [[wiki/topics/Thinking in Systems]]：把反复问题从事件层推进到模式、结构、心智模型和杠杆点。

适用问题：为什么同一类问题总回来？为什么局部修补没用？为什么系统越努力越糟？

## 2. 深度思考与本质类机制

这一组模型用于把“解释”压成可迁移结构。

- [[wiki/syntheses/深度思考 高阶思维与本质理解]]：高阶思维负责审题，深度思考负责沿生成机制下钻，深度理解负责形成可解释、可迁移、可行动的结构。
- [[wiki/sources/概念 因果 规律 机制与知识 Source Guide]]：把机制定位为因果的内部解释，把规律定位为因果稳定显现后的概括，并区分因果成立、因果出现与规律成立。
- [[wiki/sources/知识论的秩 Source Guide]]：把知识论解释为世界阻力、心智取形和公共追问构成的负反馈环，用来压低自洽幻觉、原始刺激和任意相信。
- [[wiki/sources/知识论概念解剖 Source Guide]]：把知识论解释为信念取得资格的机制：信念必须被世界约束、理由结构、公共追问和可错边界共同筛选。
- [[wiki/sources/知识论圆桌 Source Guide]]：把知识论解释为对象性、可错性和可信任性之间的动态平衡，尤其强调怀疑、证据和信任分配的反馈环。
- [[wiki/sources/知识结构概念解剖 Source Guide]]：把知识结构解释为节点、关系、权重、场景触发和可重组路径组成的调用机制。
- [[wiki/sources/知识图谱的秩 Source Guide]]：把知识图谱解释为钉住对象、织出关系和沿路追问之间互相修正的反馈环。
- [[wiki/concepts/Analysis]]：分析不是拆碎，而是沿对象自身结构切开，并能重新组合成行动理解。
- [[wiki/concepts/Understanding]]：理解不是拥有一句定义，而是形成一团能类比、抽象、迁移和重建的概念云。
- [[wiki/topics/Problem Framing]]：先审查问题边界、目标和前提，避免在错误问题里高效求解。
- [[wiki/topics/Mental Models]]：把经验压成可复用的判断结构，但需要现实反馈校验。

适用问题：我是否抓住本质？这个解释能不能解释多个现象？它能不能迁移到新场景？

## 3. 真实结构诊断类机制

这一组模型最接近“拆机制”的操作模板。

- [[wiki/concepts/Mechanism Model]]：用结果、变量、约束、激励、反馈和时间把混乱叙事压成结构句。
- [[wiki/concepts/Hard Constraint]]：行动必须先通过资源、时间、规律和他人能动性的现实校验。
- [[wiki/concepts/Human Agency]]：能动不是情绪宣言，而是在约束中主动选择目标、调用工具、制造反馈并承担结果。
- [[wiki/concepts/Narrative]]：叙事把事实组织成能预测、赋义、协调和行动的因果线；叙事也可能遮蔽真实结构。
- [[wiki/concepts/Cynefin Framework]]：先判断上下文类型，再决定是分类处理、专家诊断、实验探测，还是先行动再感知。

适用问题：真实结构是什么？关键变量是哪几个？这个系统实际奖励什么？哪个反馈在放大问题？

## 4. 工程与软件系统机制

这一组模型解释工程系统为什么可靠、失控、返工或演化困难。

- [[wiki/concepts/engineering-thinking]]：工程思维把模糊意图变成可靠、可测试、可修复、可演化的结构。
- [[wiki/syntheses/Reality-Refutable Engineering Systems]]：成熟工程系统允许现实通过反馈、申诉、问责和重设计触发器反驳自己的模型。
- [[wiki/topics/Modern Software Engineering]]：现代软件工程是一种经验主义、证据驱动的问题求解活动，而不是单纯流程仪式。
- [[wiki/topics/Testing Strategy]]：测试通过分层、质量门和 AI 时代验证机制增加对变化正确性的信心。
- [[wiki/syntheses/Quality Engineering Three Generators]]：质量工程通过“承诺清楚、偏差显形、修正闭环”控制系统承诺和真实结果之间的偏离。
- [[wiki/concepts/Conceptual Integrity]]：系统所有部分像从同一个世界观中长出来，用户、程序员和机器面对的是同一套概念结构。
- [[wiki/concepts/Software Analysis Three Generators]]：软件分析可压到组织、交换、边界三根生成器。
- [[wiki/concepts/Software Design Three Generators]]：软件设计可压到责任、协作、抗变三根生成器。
- [[wiki/syntheses/软件设计作为系统诊断]]：把软件设计视为从代码事件下潜到重复模式、生成结构、心智模型和杠杆点的系统诊断。
- [[wiki/sources/设计模式 设计原则与系统思维 Source Guide]]：把设计原则/设计模式放进 Iceberg Model 的层级中，并区分信息流与状态流转这两类软件系统生成机制。
- [[wiki/sources/Galls Law Source Guide]]：把复杂系统能否工作追溯到是否从一个已经工作的简单系统演化而来。

适用问题：为什么越改越难？为什么测试没有带来信心？为什么需求变化其实是早期问题定义失败的延迟暴露？

## 5. 组织与责任机制

这一组模型解释组织里责任、信息、权力和激励如何生成行为。

- [[wiki/concepts/Accountability]]：公开、署名、承担风险让社会和组织能把责任、权益和杠杆路由到人。
- [[wiki/concepts/Institutional Friction]]：组织在对话、决策、所有权和共享现实上退化时，会产生协调损耗。
- [[wiki/concepts/Organizational Memory]]：组织记忆把个人工作变成共享工作，再变成可被未来决策调用的制度记忆。
- [[wiki/concepts/Organizational Ontology]]：把对话解释成决策、承诺、风险、假设、依赖和开放问题。
- [[wiki/topics/Technical Management]]：技术管理要处理授权、梯队、激励、信息流和责任边界，而不只是追执行。
- [[wiki/syntheses/Decision Modes for Engineering Work]]：不同问题类型需要不同决策模式，不能只换答案不换决策结构。

适用问题：为什么组织奖励表演而不是负责？为什么问题被反复转移？为什么信息没有变成行动？

## 6. AI agent 与 harness 机制

这一组模型解释 agent 系统的能力、失控、修复和演化来自哪里。

- [[wiki/topics/AI Harness]]：harness 是连接模型、上下文、工具、权限、状态、恢复和反馈的运行时秩序层。
- [[wiki/concepts/Coding Agent User Harness]]：开发者通过上下文、工具、规则、检查和模板塑造 agent 行为。
- [[wiki/concepts/Verification Loop]]：测试、视觉检查或 evaluator 给 agent 外部反馈，防止错误复利。
- [[wiki/concepts/Harness Ratchet]]：把重复 agent 失败沉淀为规则、hook、检查或流程改造，而不是一次性坏结果。
- [[wiki/concepts/Feedforward and Feedback Controls]]：前馈控制塑造行动前条件，反馈控制检查行动后结果。
- [[wiki/concepts/Context Information Density]]：长程 agent 质量取决于有限上下文中的决策相关信息密度，而不是历史越多越好。
- [[wiki/concepts/Skill Self-Evolution]]：skill 可以通过边界任务、指令/代码变体、对照执行和可追溯评分自我改进。

适用问题：为什么 agent 这次能做对、下次又错？为什么 prompt 优化不如改 harness？为什么验证变成 AI coding 的瓶颈？

## 7. 记忆与状态机制

这一组模型解释组织和 agent 为什么需要共享、可检查、可纠正的状态。

- [[wiki/concepts/Company Brain]]：公司大脑是权限化的组织记忆基底，不只是工具访问数据。
- [[wiki/concepts/Factual Memory]]：组织必须能回答什么存在、发生了什么、证据在哪、谁拥有、何时改变。
- [[wiki/concepts/Interaction Memory]]：保存人们的意图、争论、承诺、假设和未决问题。
- [[wiki/concepts/Action Memory]]：记住什么时候该唤醒工作、走哪条路径、使用什么 guardrail、结果如何。
- [[wiki/concepts/Semantic File System]]：文件关系、所有权、来源、新鲜度和权限与文本内容同等重要。
- [[wiki/concepts/Context Graph]]：连接事实、证据、责任人、决策、承诺和权限，让公司事实变成可推理上下文。

适用问题：为什么每个工具都有自己的局部真相？为什么组织反复忘记已经做过的判断？为什么 agent 不能安全代表公司行动？

## 8. 学习与认知机制

这一组模型解释理解、迁移、认知负荷和学习停滞如何生成。

- [[wiki/concepts/Cognitive Load Theory]]：学习受工作记忆带宽、内在负荷、外在负荷和图式形成约束。
- [[wiki/syntheses/ICAP 与知识类型的适用边界]]：ICAP 是学习深度刻度，需要结合知识类型、学习阶段、先验图式和认知负荷使用。
- [[wiki/concepts/AI Learning Tutor Loop]]：AI tutor 应先建立知识地形、暴露分歧、诊断理解，再修补错误。
- [[wiki/concepts/Knowledge as Network]]：知识不是塔，而是局部有序的网。
- [[wiki/concepts/Cognition Three Channels]]：认识依赖接收、替代经验和想象推演三根生成器。
- [[wiki/concepts/三个自我模型]]：成长要改写内核自我的预测模型、先验假设和更新规则。

适用问题：为什么我以为懂了但不会用？为什么输入很多却没有结构？为什么 AI 如果只代做会破坏学习反馈？

## 9. 财富、杠杆与世界观机制

这一组模型解释回报结构、资产、风险和能动性如何改变人生策略。

- [[wiki/concepts/Heavy-Tailed Distribution]]：少数极端值占据巨大份额，平均值会失去直觉意义。
- [[wiki/concepts/Additive World]]：回报主要由投入次数相加，适合保底、补短板和稳定交付。
- [[wiki/concepts/Multiplicative World]]：回报由当前动作与既有存量共同决定，要求寻找可复利、可放大的长板。
- [[wiki/concepts/Permissionless Leverage]]：代码和媒体让判断不经资本或层级许可也能被复制放大。
- [[wiki/concepts/Specific Knowledge]]：特定知识来自真兴趣、实践和创造/技术边缘，难以被标准培训直接复制。
- [[wiki/topics/Wealth Creation]]：财富创造是拥有可扩展资产，为社会提供想要的东西，而不是单纯出租时间。

适用问题：为什么努力不等于上限？为什么有些动作只加法增长，有些动作进入复利？为什么杠杆会放大判断而不是替代判断？

## 使用顺序

遇到一个复杂问题时，可以按这个顺序调用机制模型：

```text
1. 用 [[wiki/topics/Problem Framing]] 审题：我问的是不是正确问题？
2. 用 [[wiki/concepts/Iceberg Model]] 下潜：事件下面有没有模式、结构、心智模型？
3. 用 [[wiki/concepts/Mechanism Model]] 压句：结果、变量、约束、激励、反馈、时间是什么？
4. 用 [[wiki/concepts/Feedback Loops]] 找环：什么在放大，什么在纠偏？
5. 用 [[wiki/concepts/Leverage Points]] 找动作：最值得改哪一层？
6. 用 [[wiki/syntheses/Reality-Refutable Engineering Systems]] 校验：现实能否反驳这个解释？
```
