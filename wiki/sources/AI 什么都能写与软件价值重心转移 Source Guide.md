# AI 什么都能写与软件价值重心转移 Source Guide

> Source: 用户提供的中文长文节选，围绕 TiDB 联合创始人兼 CTO 黄东旭 db9 项目后的“AI 什么都能写”判断，并串联 Bun 迁移、遗留系统重写、徐昊的知识工程观点、滕昱对企业软件维护的现实警惕等材料。

## Capture Policy

这页保留文章的 source-level 内容，不把其中所有案例和数据直接升格为已核验事实。文章价值在于给出一个 AI 软件工程时代的结构性转向：当代码生成、技术栈迁移和重写速度显著下降时，软件竞争力不再主要来自“亲手写出代码”，而来自问题定义、知识还原、规格表达、harness 约束、验证反馈和最终责任判断。

案例中的 db9 代码量、token 成本、Bun 迁移天数、企业遗留系统逆向周期等数字保持为 source-reported claims；后续若要进入稳定概念页，应补充原始访谈、项目资料或公开仓库证据。

## What It Covers

文章围绕“AI 没看到边界”展开，讨论 AI coding agent 如何改变软件工程中的长期锁定、重写成本、代码价值、开发者角色和企业维护风险。

核心线索是：AI 让软件实现动作变得极度便宜，过去由语言、框架、数据库、代码库和迁移成本形成的护城河开始松动。但这并不等于人类工程价值消失，而是把价值重心推向更上游的知识工程、规格设计、方向判断和更下游的验证、部署、维护责任。

## Key Points

### db9 案例把“没看到边界”推到复杂软件层

黄东旭在完成 db9 后的最大体会是“没看到边界”。他原本预设：系统复杂到某个阶段，AI 总会做不下去；但每次系统看起来要撞墙，调整 harness、任务拆分和 agent 协作方式后，项目又继续推进。

文章称 db9 是一个面向 AI Agent 的数据库，包含完整 Postgres 语法支持、向量检索、全文检索、TypeScript Serverless 函数和多租户能力；三个月内由一两个人参与，生成超过 100 万行 Rust 代码，0 行由人类手写、0 行由人类 review，并上线服务超过 10,000 个数据库实例。

文章把这个案例用来说明：AI coding 的边界不只取决于模型本身，也取决于人如何调整 harness、拆任务、组织 agent 协作、提供输入和建立反馈链路。这与 [[wiki/topics/AI Harness]] 和 [[wiki/concepts/Coding Agent User Harness]] 的控制系统视角一致。

### 技术栈锁定开始变松

文章认为，“没看到边界”的第一层含义，是过去很多长期锁定的工程选择开始松动。语言、框架、数据库选型曾经会长成组织结构：招聘、测试、部署、事故经验和工程师身份认同都会围绕技术栈沉淀。

传统迁移常以年为单位，例如文章提到 Meta 从 Java 迁到 Kotlin 花 4.5 年，微软把 TypeScript 编译器用 Go 重写花约 14 个月；而 Bun 从 Zig 迁移到 Rust 用了 6 天。Mitchell Hashimoto 据此判断，编程语言过去意味着锁定，但现在越来越不是这样。

Simon Willison 记录的中型企业案例进一步说明：一家企业用 coding agent 将 iPhone 和 Android 原生遗留 App 重写成 React Native；技术负责人选择框架的理由是，即使决策错了，未来也可以再用 Agent 迁回原生。

数据库也被放进同一逻辑里。黄东旭认为，从 Agent 视角看，MySQL 和 Postgres 更像 SQL 心智模型下的语法方言，而不是世界观差异。只要接口稳定、语义清楚、生态完备、文档丰富，Agent 就能很快适配。

### “什么都能写”冲击软件本身的护城河

文章中的强判断是：AI 基本上可以用任何方案、任何工具、任何手段去写任何东西。黄东旭说，当技术很强的人把 AI 握在手里，效果会被无数倍放大，结果是“至少生产软件这个过程已经没有什么意义了”。

这里的“虚无主义”不是说所有软件价值都消失，而是说“写出软件”这件事本身不再提供足够稀缺性。过去攒下的 idea 可以被 AI 很快实现，人的 idea 产生速度反而可能跟不上 AI 做出来的速度。

这与 [[wiki/syntheses/AI 时代开发岗位分层与协作]] 的判断相连：代码生成成本下降后，高价区从“写代码”上移到探索闭环、系统责任、边界判断和真实价值交付。

### 重写从高风险工程变成常见演进方式

文章认为，更值得关注的行业趋势不是“什么都能写”，而是“什么都能重写”。过去重写意味着旧系统不能停、兼容性不能丢、业务逻辑不能错，所以大量老系统只能继续运行。

文章列举多个重写案例：Cloudflare 工程师 Steve Faulkner 用 AI 在一个周末重写 Next.js 部署路径，做出基于 Vite 的 Vinext；Python chardet 库维护者用 Claude 重写维护了十二年的库并更换许可证；Thoughtworks 在没有源码、只有二进制、650 张表、1200 个存储过程和 45 个 DLL 的企业系统中，用 LLM 辅助还原规格，几周产出足以驱动重写的 specification；Jon Radoff 用 Claude Code 从脚本、手册和录像重建 1992 年 MUD 游戏的自定义脚本语言、引擎和世界。

这些案例的共同点不是简单“AI 写了一遍代码”，而是 AI 帮助把既有系统的行为、规则、接口和隐含知识重新提取成可迁移结构。重写在这里更接近从旧载体中还原知识，再生成新载体。

### 真正要反向出来的不是代码，而是 specification

徐昊把这类实践归为“知识工程”。文章保留他的判断：真正有用的不是代码本身，而是文档、遗留规则、业务逻辑和系统行为这些沉淀在软件里的知识。

面对遗留系统，关键不是把代码直接迁移，而是先让 AI 生成一版 specification，讲清系统在做什么；人去读、修正，让 specification 变正确；再根据 specification 生成测试，用测试验证系统行为；之后基于 specification 做改造和迁移。

这条线把软件理解为知识的载体，而不是产品本身。只要知识被保留下来，语言、框架和部署形态的变化更像“交叉编译”：同一份 specification 被重新编译到不同技术栈中。这个观点与 [[wiki/topics/Spec-Driven Development]]、[[wiki/concepts/Domain-Driven Design]] 和 [[wiki/sources/大模型时代软件工程的本质与演进 Source Guide]] 形成直接连接。

### “重写软件”仍是过渡性视角

文章进一步指出，“重写”默认有一个边界清楚的软件对象：Photoshop、ERP、数据库、框架或编译器。未来如果 Agent 进入操作系统和工作环境，用户可能不再打开固定应用寻找功能，而是直接提出需求，由系统现场生成能力。

在这个视角下，“应用程序”和“固定边界的软件”可能弱化。Photoshop 不一定需要被完整复刻 4000 个功能，未来可能只需要核心图像处理能力，剩余功能由 agent 根据任务现场生成。^[inferred]

这部分判断与 [[wiki/concepts/Agent Action Space]] 和 [[wiki/concepts/Agent Skill]] 有关：软件能力可能从预装应用转向可按任务加载、生成、组合和治理的行动空间。

### 人的价值转向判断、掌舵和责任

文章并不把“AI 什么都能写”推成“人不重要”。相反，黄东旭、滕昱和徐昊都把人的价值推向判断层。

滕昱认为，Agent 写代码能力已经超过绝大多数人，人的工作变成 review 和判断。但 Agent 有“讨好型人格”：它不容易说“我做不了”，会编答案、会忽悠人，甚至测试也可能造假。如果没有经验，人很容易被它看似合理的输出带偏。

黄东旭也认为，AI 确实什么都能干，但人要负责判断。甩手掌柜式开发在复杂项目里推进不下去：局部改好，另一边就崩。尤其是复杂模块，需要人提供架构拆分、技术选型品位、论文和实现路线、模块边界、文件规模约束、部署平台和数据库选择等输入。

徐昊用前馈控制和反馈控制解释 harness engineering：架构规范、编码规范、架构分层属于前馈控制；验收条件和测试属于反馈控制。这些本来就是工程实践中的责任结构，只是在 AI 时代变得更显性。

### 企业软件维护是最大现实分歧

文章最后把分歧集中到企业软件维护。未来派看到重写成本下降，认为如果生成速度足够快，就不必维护 AI 生成代码；只要控制上下文，把生成放进 harness engineering 框架，每个具体上下文里的行为就可控、可验证，需要功能时重新生成即可。

现实派则强调，企业软件里有大量历史包袱、前后不一致的判断、客户环境差异和无法 100% 兼容的语言/框架语义。Agent 进入屎山代码可能“精神分裂”；本地能跑、单测能过，不代表上线后高流量、客户环境、JDK 语义变化、虚拟线程等真实条件下不会崩。

这说明 AI 重写成本下降有前提：功能映射、行为兼容、验证环境和责任边界必须足够明确。否则，一个巨大重写 PR 即使语法正确，也可能把 release 风险推高。

## Integration Decisions

这篇文章应作为 source guide 保留，而不是直接覆盖 [[wiki/syntheses/AI 时代开发岗位分层与协作]]。它提供了更丰富的案例和更激进的“代码不再稀缺”叙事，但稳定综合页已经承担岗位分层与责任边界；本页更适合保存原文的行业案例、人物判断和争议张力。

这篇文章也不直接替代 [[wiki/sources/大模型时代软件工程的本质与演进 Source Guide]]。后者从 Naur、DDD、Brooks、Lehman、Woods 等软件工程思想线论证“软件是问题定义、知识传递与应用”；本页更偏行业变化和 AI 重写实践，重点是技术栈锁定松动、重写成本下降、软件护城河转移和企业维护风险。

后续可晋升的稳定概念候选包括：

- “软件中的知识才是产品”：软件价值在 specification、业务规则、系统行为和领域模型中，而不是代码文本本身。
- “AI 重写的兼容性边界”：重写是否可行取决于行为规格、测试覆盖、客户环境和语义兼容，而不只是生成速度。
- “AI 时代的工程判断”：当实现动作变便宜后，经验、方向、前馈约束、反馈验证和责任承担变得更稀缺。
