# 四层次原因分析法 Source Guide

> Source: 工作和生活中经常会遇到一些线性问题，怎么去分析导致问题的原因呢？

## Capture Policy

这页保存一篇中文方法文章的 source 层内容，重点保留“四层次原因分析法”的教学顺序、图示结构、例子、适用边界和操作口诀。它不是把该方法直接升格为稳定概念页；稳定层可以继续通过 [[wiki/concepts/Analysis]]、[[wiki/topics/Problem Framing]]、[[wiki/concepts/Iceberg Model]]、[[wiki/concepts/Mechanism Model]] 和 [[wiki/topics/Thinking in Systems]] 承接。

## What It Covers

这篇文章回答一个具体问题：工作和生活中遇到线性、因果较明确的问题时，怎样把原因分析透，而不是停留在表象、拍脑袋归因或简单怪人。

文章把原因分析压成四层：

1. **拆解问题**：先把模糊大问题切成能继续追因的小问题。
2. **直接原因**：用可测量、可验证的近端因素解释子问题为什么发生。
3. **流程原因**：追问系统为什么允许这个直接原因发生，寻找流程、标准、制度和防御漏洞。
4. **人的原因**：在流程和管理设计之后，再分析无意识犯错、能力不足、态度/价值观冲突等人因。

这条路径和 [[wiki/sources/拆解 让复杂问题变清晰 Source Guide]] 相邻：前者更广泛讨论“拆解让复杂问题变清晰”，本页则把其中的“四层次原因分析法”展开成更完整的操作型 source guide。

### 图示结构：从问题分类到直接原因，再越过流程与人因边界

配图把方法画成从左到右的分析链：

```text
问题 y
  ↓
第一层次分析：问题分类、分层、分维度
  - 20/80 原则
  - MECE
  - A 类 / B 类 / C 类
  - A1/A2/A3、B1/B2、C1/C2 等继续分层
  - 归纳法用于从具体问题回到分类结构

  ↓
第二层次分析：直接原因（识别的可测量因子）
  - X1 / X2 / X3
  - X11、X12、X13、X21、X22、X31、X32 等候选因子
  - 红框突出 X12、X13、X22 等被验证或聚焦的因子
  - 演绎法用于从候选因子推出可能原因
  - 假设验证法贯穿前两层，用事实或数据验证真因

  ↓  （图中以竖向虚线分隔技术直接原因与管理/人因层）
第三层次分析：管理原因（流程）
  - 无管控流程
  - 管控流程无效
  - 流程有效，但存在“未按要求操作”
  - 可归结为流程原因

  ↓
第四层次分析：人的原因
  - 能力
  - 人的无意识犯错、态度等
```

图示的重要信息不是“所有问题都必须完整画树”，而是：原因分析先在可测量的技术因子上收敛，再穿过系统边界去看管理流程，最后才进入人的原因。这样可以避免一开始就把问题归咎于个人。

### 方法总纲：像剥洋葱一样从表象到根因

文章把“四层次原因分析法”定义为一种从表象到根因的结构化分析框架。它的动作像剥洋葱：不是一次性给出根因，而是逐层降低模糊度。

四层分别是：

```text
表面问题
  → What 切块（分类拆解）
  → Why 找技术原因（可测量验证）
  → 流程漏洞（系统防御）
  → 人因短板（最后防线）
```

文章强调，这套方法最适合线性、因果明确的问题。遇到非线性反馈、黑天鹅或群体心理效应时，不能把它当成万能框架，而应结合 [[wiki/topics/Thinking in Systems]]、因果循环图、增强回路、调节回路和杠杆点分析。

### 第一层：What 型原因——把大问题切小块

第一层的目标不是解释“为什么”，而是先把“到底是哪一块出了问题”切出来。

核心原则是：**切分要服务于追溯原因，而不是为了切分而切分。**

文章把这个动作比作切蛋糕：模糊的大问题必须被拆成具体子问题，但切分维度需要服务于后续 Why 型原因分析和问题解决。如果某个切分维度不利于继续追因，或者不能支持解决问题，就要换一个维度再切。

常见切分维度包括：

- **按流程切**：“会议效率低”可以切成会前准备差、会中跑题、会后无跟进。
- **按公式切**：“利润下降”可以切成收入减少和成本增加。
- **按模块切**：“客户投诉多”可以切成产品质量差、物流慢、客服态度差。
- **按最适合问题的维度切**：目标是切出能继续深挖的小问题，最好还能切出二八结构，找到关键少数。

第一层分析必须严格遵循 MECE：不重复、不遗漏。这里的 MECE 不是形式洁癖，而是避免漏掉重要子问题、重复分析同一块问题。

文章也提醒：如果问题很小，不需要切分，这一层可以略过；不要为了“看起来结构化”而强行切。

### 第二层：Why 型原因——找到可测量、可验证的直接原因

第二层开始寻找技术层面的直接原因，也就是可测量、可验证的近端因素。

核心原则是：**原因必须可测量、可验证，拒绝拍脑袋归因。**

文章建议组合使用：

- 5Why 的溯因逻辑；
- 鱼骨图或树图的结构化发散；
- 投票法、评分法等相对主观的聚焦方式；
- 假设验证法，用数据或客观事实验证候选因子。

完整动作是：

```text
先发散识别大量影响因子
  → 聚焦最可能的影响因子
  → 采集数据或事实
  → 验证哪些因子是真因
```

文章给出的对比实验例子是：同一产线 A 班次不良率高，通过对比 B 班次操作参数，发现 A 班温度设定低 5℃。在设备、原料相同而操作参数不同的条件下，温度、次数、错误率等可测量指标可以支持真因验证。

这里要避免停在“人”的层面。例如“操作工没检查”是结果，不是技术层面的直接原因。它可能仍要继续追问检查为什么没有发生、系统为什么允许漏检。

文章用经典 5Why 例子说明第二层到哪里可以先止血：

```text
问题：机器停了
  → 为什么？保险丝烧了
  → 为什么？负载过大
  → 为什么？轴承缺润滑
  → 为什么？润滑泵故障
```

到“润滑泵故障”时，技术层面已经可以解决：换泵。但文章同时强调，这种解决只处理了当前问题；如果不继续分析为什么润滑泵故障没有被预防，解决效果不可持续。

第二层不要求严格 MECE。面对未知原因，很难保证无遗漏。如果本轮遗漏了真因，后续问题没有解决，再进行第二次第二层分析。

### 第三层：流程原因——系统防御漏洞在哪里

第三层要回答的问题更深：**为什么系统允许这个直接原因发生？**

文章的核心原则是：**重复发生的问题，本质是流程防御的系统性漏洞。**

第二层找到的是一次性问题的直接答案。第三层寻找的是系统、流程、制度、作业方法或标准中的漏洞。它关心的不是“这个人为什么这次没做”，而是“这个漏洞为什么能存在”。

典型流程问题包括：

- **有标准但没执行**：可能是检查机制失效。
- **无标准**：可能是知识没有沉淀。
- **标准过时**：可能是更新流程缺失。
- **标准不好执行**：可能是制定标准时没有考虑现场实施环境。

文章延续润滑例子：

```text
发现轴承润滑不足
  → 为什么没预防？点检表无润滑项
  → 为什么？设备手册未纳入此要求
```

这一层要刻意避免跳过流程直接怪人。文章引用戴明博士的判断：“94% 的问题来自系统，而非个人。”这里的核心意图是把反复问题放回系统防御中，而不是把每次事故都当成个人注意力或态度问题。

### 第四层：人因分析——最后才分析人性弱点

第四层不是一上来追责，而是在流程和防御条件已经被审视之后，继续分析人因。

核心原则是：**先反思管理设计是否给了人不犯错的条件，再谈态度。**

文章认为，人为错误常常是多层防御同时失效的结果，例如：

```text
操作工未检查
  = 无防错提醒（流程）
  + 未培训检查要点（能力）
  + 考核只看速度（激励）
```

这和“瑞士奶酪模型”的精神相近：事故并不是单个洞导致，而是流程防错、能力匹配、激励机制等多层防御同时出现漏洞。人因分析的目标不是追责，而是补全奶酪上的漏洞。^[inferred]

文章把人因进一步分为三类：

#### 无意识犯错：流程防错性不足

无意识犯错对应流程没有做到“想犯错误都难”。文章认为这类问题不需要过度下钻原因，因为再怎么追问也可能只能得到“人会犯错”。更有效的动作是防错设计、提醒机制、强制校验、物理或流程约束。

#### 体力、能力、技能不足：选拔或培训体系漏洞

能力不足对应系统没有确保人具备能力。可继续追问：

```text
培训不足
  → 为什么？培训内容未覆盖实际场景

疲劳分神
  → 为什么？排班不合理 / 无休息提醒
```

这里的根本动作仍不是单纯责怪个人，而是检查选拔、培训、排班和支持系统。

#### 态度或价值观冲突：激励机制或文化缺陷

态度问题需要管理干预，但也不能被滥用。文章给出的例子是：

```text
故意简化步骤
  → 为什么？考核只强调速度
```

如果系统实际奖励速度、容忍绕流程，那么“态度差”可能只是表象；更深层是激励机制在奖励错误行为。

文章要求先问一句：**是否给了员工不犯错的条件？** 动不动扣“态度差”的帽子，是管理的懒惰。

第三、四层不要求 MECE，重点是可执行性和可实施性。

### 如何剪裁层次

文章强调，这套方法不是每次都要机械走完四层，而要根据问题剪裁：

- **简单问题**：到第二层即可，例如技术升级、技术改造可以直接解决的问题。
- **复杂问题**：走完四层，例如不良事件分析、重复事故分析、组织流程问题。
- **紧急问题**：先用第二层处理直接原因止血，再补第三、四层的系统修复。

这对应一个操作节奏：先控制近端风险，再修系统漏洞。对工程、管理和生活问题都适用，但前提是不要把止血误认为根治。

### 适用边界：线性问题优先，非线性问题要换工具

文章明确指出，“四层次原因分析”有前置条件：最适合线性、因果明确的问题。

遇到以下情况时，需要突破该框架：

- 非线性问题：多个原因循环作用，不能只沿单链条下钻。
- 黑天鹅事件：无先例可循，历史经验和标准诊疗路径不足。
- 群体心理效应：行为受模仿、恐慌、舆论、预期等群体反馈影响。

对于非线性问题，文章建议先用四层次分析拆解单个线性子问题，再用因果循环图梳理子问题之间的关联。也就是说，四层次分析可以作为局部诊断工具，但全局关系仍需要系统动力学方法。

### 四句操作口诀

文章最后把方法压成四句话：

```text
拆问题：切分要“MECE 无遗漏”；
找技术因：验证要“数据说了算”；
查流程：漏洞要“修墙补系统”；
析人因：管理要“将心比心设计”。
```

这四句话分别对应四层的常见误区：

- 第一层防止乱切和漏切。
- 第二层防止拍脑袋归因。
- 第三层防止只修当下、不修系统。
- 第四层防止用“态度差”替代管理设计。

## Open Questions

- 四层次原因分析法与 Iceberg Model 的层级关系如何精确对应：直接原因是否等同于事件/模式之间的近端因子，流程原因是否等同于结构层？
- “人的原因”如何在实践中避免滑向追责？是否需要一个固定问题清单来强制先检查流程防错、能力匹配和激励机制？
- 对非线性问题，什么时候先拆局部线性链条，什么时候应该直接进入因果循环图？
- 第二层的“主观评分/投票聚焦候选因子”如何避免群体偏见？是否需要明确的证据等级？
