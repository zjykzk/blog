# 面向 LLM 的清晰表达 Source Guide

> Source: 当前对话中关于“对 LLM 表达清晰”与“如何判断是否显式化”的中文教学片段

## Capture Policy

本页保留一段中文教学对话的 source 层内容。它不是把内容压缩成普通 prompt 技巧，而是保存“清晰 = 减少模型猜测成本”的解释、六要素模板、显式化检查表、最小反问测试和最小验收测试，供之后连接到 [[wiki/topics/Prompt Frequency]]、[[wiki/topics/Context Management]]、[[wiki/concepts/Expression Three Channels]] 和 [[wiki/concepts/Understanding]]。

## What It Covers

面向大语言模型的清晰表达，关键不在于模型能否读懂自然语言，而在于让模型在当前上下文里，以最低歧义、最低猜测成本，生成想要的可验证结果。

这类清晰可以压缩为：任务边界清楚、判断标准清楚、输出形式清楚、可用信息清楚。对人类说话时，许多信息可以靠常识、关系和语气补全；对 LLM 下指令时，模糊处会被模型补成“看起来合理”的内容。因此，清晰的核心不是让模型“懂”，而是减少模型自由解释的空间。

## Preserved Content

### 清晰不是说得自然，而是减少自由度

不清晰的指令是：

> 帮我优化一下这段内容。

模型无法知道“优化”指什么：逻辑、文风、结构、压缩、说服力、口语化、专业化、保留原意，还是允许重写。

更清晰的版本是：

> 请重写下面这段文字，目标是让它更适合放进产品说明文档。  
> 要求：  
> 1. 保留原意，不新增事实。  
> 2. 语气专业、简洁。  
> 3. 删除重复表达。  
> 4. 输出一版改写文本，再列出 3 条主要修改理由。

这个版本通过目标、限制、输出物和验收依据收窄模型自由度。

### 对 LLM 表达清晰要明确六件事

#### 1. 任务是什么

不要只说“处理一下”“看看”“优化一下”。任务动词要明确，例如总结、改写、对比、提取、分类、审查、生成、翻译、反驳、找漏洞、给方案、写代码、设计测试。

更清晰的表达是：

> 请从下面文章中提取 5 个核心观点。

而不是：

> 看看这篇文章。

#### 2. 输入材料是什么

LLM 容易把上下文里不相关的东西也当成材料，因此要说明“看哪里，不看哪里”。

材料内回答可以写成：

> 只基于下面这段材料回答，不要使用外部知识。

综合回答可以写成：

> 请结合你的一般知识，但如果材料中没有证据，请标明“材料未说明”。

这能控制模型是在做 source-grounded answer，还是在做一般知识综合。

#### 3. 目标读者是谁

同一内容给不同读者，清晰标准不同。

> 请解释给一个没有编程背景的产品经理。

和：

> 请解释给一个有 5 年 Go 经验、但不熟悉 React 的工程师。

会产生完全不同的抽象层级、术语密度和例子类型。目标读者是调节解释粒度的关键参数。

#### 4. 好答案的标准是什么

LLM 不知道用户心里的“好”是什么，因此要把判断标准外显。

例如：

> 一个好答案应该：  
> - 先给结论；  
> - 再给理由；  
> - 不超过 800 字；  
> - 包含一个具体例子；  
> - 明确指出适用边界。

这相当于给模型一个局部评分器。

#### 5. 输出格式是什么

如果结果要直接可用，就不要让模型自由决定格式。

不清晰：

> 帮我整理一下。

清晰：

```markdown
请按以下格式输出：

## 核心结论
...

## 支撑理由
1. ...
2. ...
3. ...

## 风险与边界
...

## 下一步行动
...
```

格式越明确，模型越不需要猜“该怎么组织答案”。

#### 6. 不要做什么

负约束对 LLM 很重要。常见约束包括：不要扩写、不要新增事实、不要改动专有名词、不要使用项目符号、不要给泛泛建议、不要解释基础概念、不要写代码只给设计方案。

很多 prompt 不清晰，不是因为没说要什么，而是没说不要什么。

### 最清晰的表达结构

一个完整模板是：

```text
任务：你要做什么。

背景：为什么要做，当前上下文是什么。

输入：下面这些内容是你要处理的材料。

目标：最终结果要服务谁、解决什么问题。

要求：
1. ...
2. ...
3. ...

限制：
- 不要 ...
- 必须 ...
- 如果不确定，请 ...

输出格式：
...
```

例如：

```text
任务：请帮我审查下面这段产品需求是否表达清楚。

背景：这段需求会交给工程师估算和实现。

目标：找出会导致误解、漏实现或测试困难的地方。

要求：
1. 指出模糊词。
2. 标出缺失的业务规则。
3. 给出改写建议。
4. 不要评价产品方向，只评价需求表达。

输出格式：
- 问题：
- 为什么会误解：
- 建议改写：
```

### LLM 场景下清晰的核心

人类表达清晰，常常是为了让对方理解。对 LLM 表达清晰，更准确地说是为了：

- 减少模型需要补全的空白；
- 减少模型可以自由解释的范围；
- 减少模型输出无法验收的可能；
- 增加结果可检查、可复现、可修改的程度。

一句话定义是：

> 面向大语言模型的表达清晰，就是把隐含意图、上下文、标准、边界和输出形式显式化，让模型不靠猜也能完成任务，并且让结果可以被检查。

### 如何判断是否显式化

判断是否显式化，可以问：如果把这段话交给另一个不了解意图的人，或一个新开的 LLM 会话，它还能不能知道该怎么做、做到什么程度、什么不能做、结果怎么算好。

#### 1. 任务是否显式化

不显式：

> 帮我看看这个。

显式：

> 请审查这段需求是否足够清楚，重点找会影响工程实现和测试验收的歧义。

检查问题：有没有说清楚“要它做什么动作”？

#### 2. 对象是否显式化

不显式：

> 优化这段。

显式：

> 优化下面这段产品说明文案，不改动事实，只提升结构和可读性。

检查问题：有没有说清楚“处理哪一段、哪一类内容、哪一个文件、哪一个问题”？

#### 3. 目标是否显式化

不显式：

> 写得更好一点。

显式：

> 改成适合放在 README 开头的介绍，让第一次打开项目的人能在 30 秒内理解它解决什么问题。

目标不是“更好”，而是更容易理解、更容易决策、更容易实现、更容易测试、更适合发布、更适合新手阅读，或更适合专家审查。

#### 4. 读者是否显式化

不显式：

> 解释一下这个概念。

显式：

> 解释给一个懂后端但不熟悉前端框架的工程师。

读者决定术语密度、例子类型、抽象层级、是否需要背景知识，以及是否跳过基础概念。

#### 5. 判断标准是否显式化

不显式：

> 帮我改好。

显式：

> 好版本应满足：  
> 1. 保留原意；  
> 2. 少于 300 字；  
> 3. 先给结论；  
> 4. 每个技术词都有解释；  
> 5. 不新增原文没有的事实。

如果没有标准，模型只能猜用户品味。

#### 6. 边界是否显式化

不显式：

> 自由发挥一下。

显式：

> 可以调整结构和语气，但不要新增事实，不要删除任何限制条件，不要改动专有名词。

边界包括能不能新增、删减、重写、推断、使用外部知识、改变语气、改变格式。

#### 7. 输出形式是否显式化

不显式：

> 整理一下。

显式：

> 请按表格输出，列为：问题、原因、修改建议、优先级。

输出形式可以是一段话、列表、表格、JSON、Markdown、PRD、commit message、测试用例、diff 或代码块。

#### 8. 不确定性处理是否显式化

不显式：

> 你看着办。

显式：

> 如果材料里没有证据，请标注“材料未说明”，不要自行补充。

常见约束包括：如果不确定，请列出假设；如果信息不足，请先问我；如果材料未说明，请不要编造；如果有多种解释，请分别列出。

### 最小反问测试

写完 prompt 后问自己：模型还需要问我什么，才能开始做？

例如：

> 帮我优化这篇文章。

模型会需要继续追问：优化什么、给谁看、保留多长、要不要改结构、能不能新增内容、风格是什么、输出一版还是多版、要不要解释修改理由。

这些没说，就是未显式化。

### 最小验收测试

再问自己：模型输出后，我能不能判断它做得对不对？

例如：

> 帮我写得更有质感。

很难验收。

改成：

> 把这段改成更克制、具体、少形容词的风格。  
> 验收标准：  
> - 删除空泛形容词；  
> - 每段不超过 80 字；  
> - 至少保留一个具体事实；  
> - 不使用“极致、赋能、领先”等营销词。

这样结果就可以检查。

### 极简判断法

判断是否显式化，只看四个问题：

1. 它知道要做什么吗？
2. 它知道对什么做吗？
3. 它知道什么算好吗？
4. 它知道什么不能做吗？

如果四个问题都能回答，基本就是显式化了。

## Integration Decisions

这页应当作为 [[wiki/sources/表达清晰圆桌 Source Guide]] 的 companion source guide，而不是替代它。圆桌讨论的是一般表达清晰：如何让理解在关系、责任和复杂性中可进入、可检查。本文讨论的是特殊接收者为 LLM 时的操作化版本：如何通过显式化任务、输入、读者、标准、边界、输出和不确定性处理来降低猜测成本。

这页也补充 [[wiki/topics/Prompt Frequency]] 的边界：自然高频表达有助于模型进入熟悉分布，但自然表达不能替代任务边界、判断标准和验收条件。好的 prompt 同时需要自然、明确、可检查。

它还连接到 [[wiki/topics/Context Management]]：上下文管理不只是放入哪些材料，也包括告诉模型哪些材料可用、哪些不可用、缺证据时如何处理。对 LLM 的清晰表达，本质上是在有限上下文里安排信息、约束和验收器。
