# 架构文档与图示之道 Source Guide

> Source: 第13讲：架构文档与图示之道 - 如何有效传递你的设计思想

## Capture Policy

这页保存一份中文课程材料的 source 层内容。它不直接压缩成“架构文档”概念页，而是保留原材料中的教学顺序、案例、模板、图示类型、ADR 模板、文档系统机制和工作坊设计，供以后从 [[wiki/topics/UML Diagrams in Software Development]]、[[wiki/topics/Requirement to Architecture Mapping]]、[[wiki/concepts/engineering-thinking]] 与 [[wiki/maps/Software Design Map]] 继续提升。

## What It Covers

这份材料讨论的中心问题是：架构师如何把设计思想清晰地传递给不同受众。它的基本立场是：架构设计如果不能被 CEO、产品、开发、运维、新人等不同角色理解和使用，就无法真正产生组织价值。好的架构师不仅要设计得好，还要解释得好。

材料从一个预算评审失败故事开始：技术 VP 准备了 120 页架构文档、45 张复杂架构图、30 页术语解释、15 页压测数据和 10 页引用，但董事会主席只问“5000 万能带来什么商业价值、风险是什么、多久见效、现有系统有什么问题”。技术团队准备了技术细节，却没有准备业务价值表达，导致预算被驳回。这被用来说明技术人的“知识诅咒”：知道得太多，反而忘记别人不知道什么。

## Preserved Content

### 核心命题：文档是架构的放大器

架构文档不是设计之后的附属说明，而是设计思想进入组织协作的放大器。架构沟通失败通常不是因为技术方案完全错误，而是因为文档没有回答当前受众真正关心的问题。

材料最后把这一点压缩成三条原则：

1. 受众第一：为不同受众准备不同层次的文档，从受众的需求出发，而不是从技术人员已有的知识出发。
2. 图形化思考：一图胜千言，但必须是正确的图；应使用标准符号，如 C4 和 UML，降低理解成本。
3. 持续演进：文档不是一次性产物，要像代码一样持续维护、审查、发布和测试。

本周行动建议有三个可选入口：

- 为当前项目创建一个一页纸概述。
- 使用 PlantUML 或 Draw.io 为系统绘制 C4 图。
- 为最近一个架构决策创建 ADR。

### 受众分层：文档先回答“为谁而写”

架构文档写作的第一步不是画图，也不是列技术栈，而是明确受众。不同角色的关注点、厌恶点和文档目标不同：

| 受众 | 关心什么 | 讨厌什么 | 文档目标 |
|---|---|---|---|
| 高管 / 投资人 | 商业价值、ROI、风险、时间线 | 技术细节、术语、复杂图表 | 获取支持与资源 |
| 产品 / 业务 | 功能影响、用户体验、上线时间 | 实现细节、技术约束 | 对齐目标与期望 |
| 开发团队 | 如何实现、接口定义、代码结构 | 模糊需求、矛盾设计 | 指导开发与测试 |
| 运维团队 | 如何部署、如何监控、如何扩缩容 | “魔法”般黑盒、缺失运维文档 | 确保系统稳定运行 |
| 新人 / 外包 | 快速上手、整体理解、关键概念 | 过时文档、矛盾信息 | 降低学习成本 |

这个表的隐含判断是：同一个架构方案需要被翻译成多种组织语言。对 CEO 是资源投入与回报，对产品是用户价值与排期，对开发是接口与模块，对运维是部署与稳定性，对新人是上手路径。^[inferred]

### 四层文档金字塔

材料提出四层架构文档金字塔：

```text
塔尖：一页纸概述（1页）
  ↓
执行摘要（3-5页）
  ↓
详细设计文档（20-50页）
  ↓
底层：技术规格说明（不限）
```

#### 一页纸概述（One-Pager）

一页纸概述的受众是高管、投资人和非技术合作伙伴，目标是在 30 秒内讲清楚核心价值。它应该回答五个问题：

1. 我们要解决什么业务问题？
2. 为什么现在要解决，紧迫性是什么？
3. 解决方案的核心思想是什么，能否用简单比喻说明？
4. 需要什么资源，包括人、钱、时间？
5. 预期成果是什么，量化指标是什么？

材料给出的示例结构：

```markdown
# [项目名称] 一页纸概述

## 业务问题
当前用户下单到收货平均需要3.2天，比竞争对手慢1天，导致15%的潜在客户流失。

## 解决方案
构建智能物流调度平台，通过算法实时优化配送路径，将平均配送时间缩短至1.8天。

## 核心价值
1. 用户体验：配送时间缩短44%
2. 商业价值：预计提升转化率10%，年增收5000万
3. 成本节约：优化路线降低15%的物流成本

## 资源需求
- 预算：800万（含人力、硬件、软件）
- 时间：6个月（3个月MVP，3个月完善）
- 团队：15人（架构2，后端6，前端3，算法2，测试2）

## 成功指标
- 关键指标：平均配送时间从3.2天降至1.8天
- 业务指标：客户满意度从75%提升至85%
- 效率指标：每单配送成本降低15%
```

#### 执行摘要（Executive Summary）

执行摘要面向技术总监、产品总监和跨部门负责人，目标是在 5 分钟内理解架构全貌和关键决策。它在一页纸概述基础上增加技术路线、高层设计和关键取舍，但仍避免陷入实现细节。

#### 详细设计文档（Detailed Design）

详细设计文档面向开发团队、测试团队和技术负责人，目标是指导具体实现，成为开发过程中的“宪法”。它应该表达架构边界、模块职责、接口协议、核心流程、数据设计、异常处理、测试策略和已确认的设计约束。^[inferred]

#### 技术规格说明（Technical Specifications）

技术规格说明面向特定技术角色，例如 DBA、SRE、安全专家。它提供深度技术细节，包括 API 文档、数据库 Schema、部署脚本、监控配置、安全策略等。

### C4 模型：架构沟通的通用语言

材料将 C4 模型称为架构师的“瑞士军刀”，用于修复传统架构图的三个常见问题：

1. 一图流：试图在一张图上展示所有细节。
2. 符号混乱：不同人用不同符号表示同一类事物。
3. 视角缺失：只展示静态结构，不展示动态行为。

C4 模型用四个层次和一致符号降低沟通成本：

```text
层次1：系统上下文图（System Context）
  回答：系统与外部世界的关系

层次2：容器图（Container）
  回答：系统由哪些主要部分组成

层次3：组件图（Component）
  回答：每个容器内部如何组织

层次4：代码图（Code）
  回答：关键组件如何用代码实现
```

#### 层次 1：系统上下文图

系统上下文图的目标是让业务方理解系统边界和外部依赖。它只显示系统与外部角色、外部系统的关系，不显示内部细节。

绘图要点：

1. 使用标准 C4 符号，如人、系统、外部系统。
2. 每个元素只有一个名称标签和一个简短描述。
3. 只显示系统与外部的关系，不显示内部组件。
4. 用颜色或形状区分系统类型。

材料中的电商订单系统示例包含客户、客服、运营、订单系统、支付系统、物流系统、库存系统，以及下单、查询订单、配置促销、调用支付接口、创建物流订单、扣减库存等关系。

#### 层次 2：容器图

容器图的目标是让技术负责人理解系统高层结构。材料中的订单系统容器包括：

- 订单 Web 应用：React，提供订单管理界面。
- 订单移动应用：React Native，提供移动端订单功能。
- API 网关：Spring Cloud Gateway，负责路由、认证、限流。
- 订单服务：Spring Boot，处理核心订单逻辑。
- 支付服务：Spring Boot，处理支付相关逻辑。
- 订单数据库：MySQL，存储订单数据。
- 消息队列：Kafka，负责异步消息传递。

容器图绘制时需要：

1. 定义明确系统边界。
2. 每个容器有明确技术选择。
3. 展示容器间通信协议。
4. 区分应用、数据库、消息队列等容器类型。

#### 层次 3：组件图

组件图的目标是让开发团队理解模块划分和依赖关系。材料以订单服务为例，列出：

- 订单控制器：REST API 端点，处理 HTTP 请求。
- 订单服务组件：业务逻辑，协调订单流程。
- 订单领域组件：领域模型，承载订单核心业务规则。
- 库存客户端：外部服务适配器，调用库存系统。
- 支付客户端：外部服务适配器，调用支付服务。
- 订单仓库：数据访问，负责数据库操作。
- 消息生产者：事件发布，发布领域事件。

组件图绘制时需要：

1. 展示容器内部关键组件。
2. 明确组件职责和依赖关系。
3. 遵循分层架构原则，例如 Controller → Service → Repository。
4. 区分内部组件和外部依赖。

#### 层次 4：代码图

代码图的目标是让开发人员理解关键设计模式和类结构。材料用 UML 类图展示订单领域模型核心部分：

- Order 聚合根：包含 orderId、customerId、status、items、totalAmount、shippingAddress、createdAt、updatedAt，以及 create、addItem、removeItem、submit、pay、cancel、calculateTotal 等方法。
- OrderItem：包含 itemId、productId、productName、quantity、unitPrice、totalPrice，以及 updateQuantity、calculateTotal 等方法。
- 值对象：OrderId、Money、Address。
- 枚举：OrderStatus，包括 DRAFT、SUBMITTED、PAID、SHIPPED、DELIVERED、CANCELLED。

代码图要点：

1. 只展示核心领域类，不展示所有类。
2. 明确类之间的继承、组合、聚合、关联。
3. 展示关键属性和方法。
4. 用包组织相关类。

#### C4 进阶技巧

材料提出三个进阶技巧：

1. 使用分层颜色编码：业务层蓝色，应用层绿色，基础设施层灰色，外部系统橙色。
2. 添加图例和注释：每张图都应包含图例、版本信息和关键设计决策注释。
3. 创建图集：把系统上下文图、容器图、服务组件图、部署图等组织成 diagram collection，说明文件、描述和受众。

示例图集结构：

```markdown
# 订单系统架构图集

## 1. 系统上下文图
- 文件：order-system-context.png
- 描述：展示系统与外部系统的关系
- 受众：所有人员

## 2. 容器图
- 文件：order-system-containers.png
- 描述：展示系统由哪些容器组成
- 受众：技术负责人、架构师

## 3. 组件图
### 3.1 订单服务组件图
- 文件：order-service-components.png
- 描述：订单服务的内部组件
- 受众：订单服务开发团队

### 3.2 支付服务组件图
- 文件：payment-service-components.png
- 描述：支付服务的内部组件
- 受众：支付服务开发团队

## 4. 部署图

- 描述：系统在K8s中的部署方式
- 受众：运维团队、SRE
```

### 动态行为图：展示系统如何工作

静态结构图展示系统是什么，动态行为图展示系统如何工作。材料重点保存三类动态图：时序图、活动图和状态机图。

#### UML 时序图

时序图适合展示交互流程。材料中的下单流程包含以下参与者和步骤：

- 用户提交订单到前端。
- 前端通过 API 网关调用订单服务。
- 订单服务验证订单数据。
- 订单服务调用库存服务预扣库存。
- 库存服务返回扣减成功。
- 订单服务创建订单记录。
- 订单数据库返回订单 ID。
- 订单服务调用支付服务创建支付单。
- 支付服务返回支付信息。
- 订单服务发布“订单已创建”事件到 Kafka。
- API 网关和前端返回订单详情，用户看到订单创建成功并被引导支付。
- Kafka 事件可被支付服务消费，用于准备支付。

绘图要点：

1. 明确参与者和系统组件。
2. 编号关键步骤，便于讨论。
3. 区分同步调用和异步调用。
4. 必要时用 alt 片段展示异常流。

#### 活动图

活动图适合展示有分支和循环的复杂业务流程。材料中的订单状态流转活动图覆盖：

- 用户提交订单。
- 判断库存是否充足。
- 库存充足时创建待支付订单并发送订单创建通知。
- 等待用户支付，验证支付结果。
- 支付成功后更新为已支付、扣减实际库存、创建物流订单、变为已发货。
- 物流运输中，用户确认收货后变为已完成；物流异常时变为异常并通知客服处理。
- 支付失败则记录失败原因。
- 支付超时则取消订单并释放预扣库存。
- 库存不足则返回错误给用户。

绘图要点：

1. 使用标准开始 / 结束符号。
2. 明确判断条件和分支。
3. 展示循环和重复流程。
4. 保持图整洁，避免过度复杂。

#### 状态机图

状态机图适合展示有状态对象的状态变化。材料中的订单状态机包含：

- 待支付：支付成功转已支付，用户取消或超时未支付转已取消。
- 已支付：商家发货转已发货，用户申请退款转退款中，支付后取消转已取消。
- 已发货：用户确认收货转已完成，用户申请退货转退货中，物流异常转异常。
- 退款中：退款成功转已取消，退款失败转已支付。
- 退货中：退货完成转已取消，退货取消转已发货。
- 异常：问题解决转已发货，无法解决转已取消。

绘图要点：

1. 明确所有可能状态。
2. 标注状态转移条件。
3. 使用嵌套状态表达复杂状态。
4. 标注初始状态和终止状态。

### ADR：架构决策记录

ADR（Architecture Decision Record）记录重要架构决策，包括：

- 决策上下文。
- 考虑过的方案。
- 决策结果和原因。
- 后果和影响。

材料给出的价值判断：ADR 保留组织记忆，避免“我们当时为什么这么决定”的失忆；促进透明决策，让决策过程可追溯和可审查；帮助新人快速理解系统设计理由；避免相同问题重复讨论。

#### ADR 模板

```markdown
# [简短描述性标题]

## 状态
[提议 | 已采纳 | 已弃用 | 已取代]

## 决策背景
[描述问题的上下文，为什么需要这个决策]

## 考虑过的方案
### 方案1：[方案名称]
[描述方案1]

### 方案2：[方案名称]
[描述方案2]

### 方案3：[方案名称]
[描述方案3]

## 决策结果
我们决定选择[选择的方案]，因为：
- [原因1]
- [原因2]
- [原因3]

不选择其他方案的原因：
- [方案1的问题]
- [方案2的问题]

## 后果
### 正面影响
- [正面影响1]
- [正面影响2]

### 负面影响
- [负面影响1]
- [负面影响2]

### 中立于影响
- [中性影响1]

## 相关决策
- [相关ADR链接]
- [相关文档链接]

## 验证指标
[如何验证这个决策是否正确？]

## 附录
[补充材料、参考资料等]
```

#### ADR 示例：订单服务数据库选型

材料中的示例决策是“订单服务数据库选型：MySQL vs PostgreSQL”。

背景：订单服务要存储订单、订单项、支付记录等数据。当前 1000 万订单，每天新增 10 万，预计 3 年增长到 1 亿订单。需求包括复杂查询、高可用、事务支持、团队熟悉度。

考虑方案：

1. MySQL 8.0：支持 ACID；团队有 5 年经验；读写分离、分库分表成熟；社区和工具生态丰富。
2. PostgreSQL 13：支持 ACID；JSONB 支持半结构化数据；查询优化器更强；并发控制更好。
3. MongoDB + MySQL 组合：MongoDB 存订单主数据，MySQL 存关系数据，获得扩展性和灵活 Schema，但增加复杂度和一致性问题。

决策结果：选择 MySQL 8.0，理由是团队熟悉度高、运维体系成熟、监控备份迁移工具链成熟、当前需求可满足、生产风险最低。

不选其他方案的原因：PostgreSQL 需要学习成本且运维经验不足；MongoDB 组合增加系统复杂度和最终一致性问题。

后果：

- 正面影响：开发效率高，运维风险低，招聘容易。
- 负面影响：复杂查询可能需要应用层配合，分库分表增加应用复杂度，JSON 支持不如 PostgreSQL。
- 中立影响：两种关系型方案性能与成本都能满足要求。

验证指标：P99 查询延迟 < 100ms；每秒 5000 笔订单创建；数据零丢失；99.99% 可用性。

#### ADR 管理结构和生命周期

推荐目录结构：

```text
docs/
├── adrs/
│   ├── 0001-record-architecture-decisions.md
│   ├── 0002-use-mysql-for-order-service.md
│   ├── 0003-implement-circuit-breaker.md
│   └── README.md
├── decisions-log.md
└── templates/
    └── adr-template.md
```

决策日志按时间倒序列出所有架构决策，例如 2024 Q1 的 Redis 分布式锁、API 网关迁移、日志收集迁移，2023 Q4 的服务网格和订单服务数据库选型。

ADR 生命周期：

```text
提议 (Proposed) → 评审中 (Reviewing) → 已采纳 (Accepted) → 已弃用 (Deprecated)
         ↓                    ↓
     已拒绝 (Rejected)   已取代 (Superseded)
```

### 构建活的文档系统

#### Docs as Code

材料主张把文档当作代码管理，让文档享受版本控制、审查和自动化构建：

```text
编写（Markdown） → 审查（PR Review） → 合并（Merge） → 发布（CI/CD）
```

推荐工具栈：

- 文档编写：Markdown；VSCode + Markdown 扩展。
- 版本控制：Git、GitHub/GitLab/Bitbucket；每个文档修改一个分支；PR/MR 审查。
- 静态站点生成：MkDocs、Docsify、Docusaurus、GitBook。
- CI/CD：GitHub Actions/GitLab CI；自动部署到 S3/GitHub Pages；链接检查。
- 图表工具：PlantUML、Mermaid、Draw.io。

#### 文档测试（Doc Tests）

材料主张像测试代码一样测试文档，尤其测试文档中的代码示例。示例伪代码是执行文档中的订单创建 API 调用，并断言响应状态码为 201。

这个做法的核心不是某个 Python 示例，而是把文档中的可执行声明纳入验证循环，避免文档漂移。^[inferred]

#### 文档健康度检查

文档健康度检查维度包括：

1. 完整性：关键部分是否有文档。
2. 准确性：文档是否与实际情况一致。
3. 及时性：文档最后更新时间。
4. 可读性：文档是否清晰易懂。

评分：A 优秀，B 良好，C 及格，D 不及格。改进计划优先修复 D，再把 C 提升到 B，最终让所有文档达到 B 以上。

### 架构文档工作坊实战

#### 面向高管的架构汇报

准备高管架构汇报时，先明确目标：主要目标是获得预算批准，次要目标是建立技术团队信誉，关键信息是风险可控、ROI 明确、时间合理。

示例材料包：

```markdown
# 面向高管的架构汇报材料包

## 1. 电梯演讲（30秒）
"我们计划用6个月时间，投资500万，重构订单系统。这将把订单处理时间从3秒降到300毫秒，支撑未来3年10倍业务增长，预计每年带来2000万额外收入。"

## 2. 一页纸概述（1页）
[一页纸概述]

## 3. 执行摘要PPT（5页）
1. 现状与问题（业务视角）
2. 解决方案概述（价值主张）
3. 投资与回报（ROI分析）
4. 风险与应对（风险可控）
5. 下一步行动（明确请求）

## 4. 附录（备查）
- 详细架构图
- 技术选型依据
- 团队组成
- 详细时间线
```

10 分钟演讲脚本结构：

1. 开场：感谢时间，说明汇报主题和预算请求。
2. 问题陈述：用业务损失和实际风险说明现状，例如高峰期超时导致订单损失。
3. 解决方案：说明微服务重构或其他架构方案如何分别解决性能、扩展性、可维护性问题。
4. 投资回报：说明投入、年度收益和 ROI。
5. 风险与时间：说明业务中断风险，以及灰度发布、回滚预案、充分测试、阶段计划。
6. 结尾与行动号召：明确请求批准并进入问答。

#### 面向开发团队的新系统架构工作坊

开发工作坊议程建议为 2 小时：

```markdown
第一部分：业务上下文（15分钟）
  - 我们解决什么问题？
  - 用户故事和用例

第二部分：架构全景（30分钟）
  - 系统上下文图
  - 容器图
  - 技术选型决策

第三部分：核心流程（30分钟）
  - 关键流程时序图
  - 数据流图

第四部分：开发指南（30分钟）
  - 代码结构
  - 开发规范
  - 测试策略

第五部分：Q&A（15分钟）
```

互动设计包括现场绘图、问答穿插、分组讨论和现场编码。其目标是让工作坊成为共同理解系统的活动，而不是单向宣讲。^[inferred]

#### 新人入职工作坊

新人文档包分成四个时间尺度：

- 第一小时：安装开发环境、克隆代码库、运行本地开发环境。
- 第一天：阅读系统概述、查看系统上下文图和容器图、运行示例应用。
- 第一周：阅读详细设计文档、了解核心业务流程、参与代码审查、完成第一个小任务。
- 第一个月：独立负责一个功能、参与架构讨论、更新相关文档。

新人任务可设计为“文档审查与更新”：阅读系统概述、架构设计文档和 API 文档；在本地运行系统验证文档准确性；找出至少 3 处文档与实际不符之处；提交 PR 修复。成功标准是找出文档问题、理解系统基本架构、成功提交并合并 PR。
