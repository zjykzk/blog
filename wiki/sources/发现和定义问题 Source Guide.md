# 发现和定义问题 Source Guide

## Capture Policy

这页保存 source 层文章内容，而不是直接把它升格为稳定概念页。文章中的 Problem / Issue 区分、四阶十六字框架、案例和提问清单应作为后续重写“发现和定义问题”主题文章的素材层，稳定抽象可再进入 [[wiki/topics/Problem Framing]]、[[wiki/topics/Critical Thinking]] 或后续综合页。

## What It Covers

这篇文章把人生和日常工作中的成长、流程改进、业务迭代都放进“发现问题 → 分析问题 → 解决问题”的全生命周期中，并强调其中最关键的前置能力是发现问题。

文章的核心贡献不是提供某个单点技巧，而是把“模糊的不对劲”推进为“可以被组织共同处理的议题”：

1. 先改变对问题的心态，把问题视为机会信号，而不是负面暴露。
2. 用 Problem / Issue 区分“被发现的麻烦”和“被定义好的问题”。
3. 用四阶段框架完成从异常到议题的转化：发现异常、确认本质、定义议题、对齐共识。
4. 用案例说明很多人失败在急着找答案，或把现象误当问题。
5. 用事实锚定、目标澄清、价值意志审视和问题陈述重构来提高 Issue 定义质量。

## Preserved Content

### 开篇：人生与流程都是问题全生命周期管理

文章用“成功的人生也不过是一辈子的缝缝补补，以及永无止境的自我进化”作为入口，把个人成长解释为持续经历“发现问题 → 分析问题 → 解决问题”的循环。

同一逻辑也适用于日常工作和流程改进：没有先天完好无缺的流程，流程是在业务运行中不断修补、迭代和演化的。流程改进的真实路径不是一次性设计出完美流程，而是在运行中发现偏差、分析原因、解决问题，并让系统螺旋式提升。

在这个全生命周期里，文章把“发现问题”放在最关键的位置：如果看不见问题，后续分析和解决都无从发生；如果看错问题，后续行动越快，偏离越快。

### 文章中的“发现问题的模型”图

原文配图标题为“发现问题的模型”，用金字塔结构把发现问题分为 why / how / what 三层：

```text
why：思维模式
  - 认知、思维和心态
  - 批判性思维
  - 环境氛围

how：方法论（方法、工具等）
  - 明确问题定义
  - 发现 / 确认真问题的方法与步骤

what：定义出具体的问题
  - 问题具体化
  - 问题背景
  - 问题描述及现状
  - 设定目标
  - 这些构成“项目定义阶段的内容”
```

这张图的结构含义是：问题定义不是直接写出 what，而是先由 why 层的心态、认知和环境决定能否看见问题，再由 how 层的方法论把异常筛成真问题，最后落到 what 层的具体问题陈述。^[inferred]

### 为什么发现问题的能力重要

文章引用叔本华、爱因斯坦和德鲁克来强化一个判断：发现正确问题比寻找正确答案更重要。

- 哲学思考需要勇气面对任何问题，也需要把似乎理所当然的事物重新当作问题来认识。
- “提出一个问题比解决一个问题更重要”的重点在于：问题一旦被错误提出，答案再精巧也会服务错误方向。
- 德鲁克强调最困难的工作不是找到正确答案，而是发现正确问题。

文章据此形成一个强判断：现实生活中会解决问题的人很多，会发现问题的人少得多。人生成长从发现自己的问题开始，企业发展也从发现自身不足开始；但人和组织往往最不愿意发现自己的问题，因为问题暴露会触发能力、责任、面子和权力关系上的不适。

### Problem 与 Issue：从麻烦到议题

文章把 Problem 与 Issue 作为“发现和定义问题”方法论的基石：

```text
Problem 是被发现的“麻烦”。
Issue 是被定义好的“问题”。

发现 Problem → 确认 Problem → 定义成 Issue → 解决 Issue = 消除 Problem
```

Problem 更接近日常中被感受到的异常、麻烦、不对劲、痛点或偏差。Issue 则是经过确认、边界收缩、指标化和共识化之后，可以进入分析和解决阶段的问题陈述。

这个区分的意义在于：发现 Problem 并不等于已经具备了解决对象。Problem 仍可能只是现象、症状、误解、局部视角或暂时波动。只有当它被确认、定义并对齐为 Issue，组织或个人才能围绕同一个对象投入资源。

文章用一句改写罗丹的话强调方法意识：生活中不是缺少 Problem，只是缺少发现 Problem 的方法。发现之后，还需要把它转化成 Issue，才能真正进入分析解决阶段。

### 四阶十六字框架

文章把发现和定义问题的完整过程压缩为四个阶段、每阶段四个字：

```text
发现异常 → 确认本质 → 定义议题 → 对齐共识
```

这四步形成一条从“感觉不对劲”到“组织可行动问题”的转换链：

1. 发现异常：从日常工作中识别“不对劲”的信号。
2. 确认本质：筛选真正需要解决、值得解决、可以解决的 Problem。
3. 定义议题：把模糊麻烦转化为清晰、可衡量、可操作的 Issue。
4. 对齐共识：让相关方理解并认同这个 Issue，避免后续执行内耗。

### 第一阶段：发现异常

发现异常的核心任务，是从日常工作中识别“不对劲”的信号。文章把这个阶段分为心态转换和具体方法两层。

#### 把问题从负面词汇变成机会信号

如果一个人或组织把“问题”视为负面词汇，就容易隐藏问题：暴露问题可能意味着自己能力不行，也可能意味着别人能力不行。在传统文化中的内敛克制、折中调和倾向下，问题还可能被包装成“不说破”的默契。

文章主张改换心态：无论怎样预防，每天仍然会出现各种问题。问题是常态，也是机会。真正危险的不是问题本身，而是对问题视而不见。丰田和大野耐一的“没有问题才是最大的问题”在这里被用来说明：组织越成熟，越不能把“没有问题”当作好消息。

#### 用“更……”思维挑战现状

“更……”思维是一种不把现状视为理所当然的触发器：

- 不能更轻松吗？
- 不能更减少浪费吗？
- 不能更快吗？
- 不能让顾客更满意吗？

这种提问的价值在于，它不用先证明现状有错，而是先打开改进可能性。它把现状从“默认合理”改写成“可以被挑战的暂时结果”。

#### 培养批判性思维

文章把批判性思维理解为类似“零基思考”的能力：回到本质，基于事实重新思考，不把现有规则、观点或现象看成天然正常。

牛顿从苹果落下发现万有引力，弗莱明从霉菌周围细菌死亡发现青霉素，都是从平常现象中看见异常结构的例子。

文章还保留了一个作者自己的工作案例：2022 年做制度汇编时，不同部门提供的制度表头格式五花八门，手动点击表格属性或复制标准表头都很慢。作者估算按原方式处理需要一周，于是追问“能不能有更快的方法”，随后找到用代码批量修改的方法。这个案例说明：如果没有先把“传统方法处理表头格式效率低”识别为问题，就不会主动寻找代码或 AI 辅助方案。

文章用一句话概括这种态度：不要用战术的勤奋掩盖战略的懒惰。发现问题，就是战略的勤奋。

#### 审视环境氛围

发现问题不只是个人能力，也受组织环境影响。如果一个人经常问“为什么”却被理解为不尊重他人；如果提出问题后受到惩罚；如果领导喜欢听喜不听忧，组织就会压制问题发现。

文章列出“系统性失明”的警戒信号：

- 把质疑当挑衅。
- 让提问题的人背锅。
- 奖励报喜鸟。
- 迷信既往经验。

当这些现象中出现两项以上，就需要警惕组织已经形成抑制问题暴露的环境。此处与 [[wiki/concepts/Iceberg Model]] 和 [[wiki/topics/Thinking in Systems]] 的系统诊断视角直接相连：看不见问题，常常不是因为个体不聪明，而是反馈回路被惩罚结构切断。^[inferred]

#### 通过量化差距发现 Problem

文章给出一个最基础的问题定义：

```text
问题 = 目标 - 现状
```

这个定义的关键不是公式本身，而是它强迫人先说清楚“应有状态”。如果没有目标或应有状态，现状就很难被判断为问题。很多争论之所以停留在感受层，是因为各方没有共享同一个目标参照系。

#### 戴上“有色眼镜”：基于原则和公理发现问题

文章把某些原则和方法称为“问题探测器”。一旦现状不符合这些原则，就可以被视为待确认 Problem：

- 精益五大原则：不符合就是问题。
- 七大浪费 / 八大浪费：缺陷、过度加工、动作浪费、等待、库存、搬运、过量生产，以及人才浪费，存在就是问题。
- 顾客满意原则：顾客不满意就是问题。

文章特别强调职能部门中的办公室浪费：管理工作的等待、协调不力、生产要素闲置、应付工作、低效反复发生、管理无“理”可依。这些隐性问题可能不会出现在财务报表里，却每天消耗组织。

### 第二阶段：确认本质

确认本质的核心任务，是从发现的异常中筛选出真正需要解决的问题。文章明确反对一发现异常就立即把它当作“问题”处理。

确认清单包括五个问题：

1. 是问题吗？对发现的现象进行符合性判断。有些从发现者视角看起来是问题的现象，从全局视角看可能不是问题。
2. 能改变吗？有些问题不在当前行动者可控范围内，例如天气不好。
3. 会自己消失吗？有些问题随时间自然消退，强行干预反而可能制造新问题。
4. 是本质吗？它是症状还是根因？解决它能否达成目的？
5. 值得解决吗？资源是否允许，优先级是否足够高？

文章提出两个关键追问：

- 这个问题解决了，能否达成我们的目的和期望？
- 这个问题背后，是否隐藏着真正应该解决的问题？

#### 迪士尼排队问题：时间问题还是情绪问题

表面 Problem 是排队时间太长。但进一步确认后，真正需要解决的可能不是排队时间本身，而是排队时的情绪问题。

两种定义对应完全不同的方案：如果定义为“时间太长”，行动会集中在缩短等待时间；如果定义为“等待体验差”，行动可能转向让等待过程更可接受、更有趣、更可预期。

#### 呼叫中心困局：客户来电是信号，不是终点

一个呼叫中心收到大量客户电话，客户抱怨等待时间过长。负责人原本想安装新电话系统来缩短等待时间。

进一步追问后发现，关键问题不是“如何更快接电话”，而是“客户为什么必须打电话”。很多电话来自产品说明书不清楚导致的安装问题。修改说明书后，电话量减少了 45%。

文章的启示是：客户来电不是“需要处理的缺陷”，而是“缺陷已经发生的信号”。与其提高缺陷处理效率，不如让客户根本不需要打电话。

#### 分蛋糕：问题核心不是精确平分，而是双方接受

A 和 B 要平分一个蛋糕，表面问题是如何精确地把蛋糕分成两份。真正的问题核心则是如何让双方都接受分配结果。

解决方案是：A 切蛋糕，B 先选。这个方案改变了问题定义：它不追求外部测量意义上的绝对均分，而是设计一个让切分者有动机公平切分的机制。

#### 成本缩减：从零件成本回到总体成本结构

一家公司想缩减成本，反复讨论降低原料采购费、使用通用零件、精简品项，却始终找不到答案。后来回到根本成本结构，发现真正的问题是无法保留当前生产规模。降低规模，才是最有效的缩减成本路径。

这个案例说明：如果只盯住单个零件成本，会遮蔽总体成本结构。真正的论点常常要从更高层的系统结构中浮现。

### 第三阶段：定义议题

定义议题的核心任务，是把模糊的“麻烦”变成清晰的“议题”。在确认某个 Problem 值得解决之后，需要把它定义为 Issue，即问题陈述。

文章给出 Issue 的核心公式：

```text
Issue = 目标值 - 现状值，且需要明确衡量指标
```

这里的重点是把“问题 = 目标 - 现状”中的每一项具体化、可量化、可对齐。

#### 事实锚定：从“我觉得”到“事实是”

决策的起点不是“我觉得”，而是“事实是”。事实锚定要求先描述客观差距，而不是先下判断。

关键提问包括：

- 我们看到的差距具体是什么？请用事实和数据描述。
- 这个现象何时、何地、以何种频率和程度发生？
- 哪些是事实，哪些是推断或假设？

这一步与 [[wiki/concepts/Analysis]] 相关：分析不是堆观点，而是把混乱对象切成事实、关系、假设和待验证部分，再重新组合成可行动理解。^[inferred]

#### 设定衡量指标和目标值

定义 Issue 需要找到衡量这个问题的“指针”：

- 衡量指标：用什么数据衡量，如日活、转化率、响应时间、缺陷率。
- 目标值：什么状态算解决成功。

文章建议目标设定符合 SMART 原则：具体、可衡量、可实现、相关、有时限。

#### 写出清晰的 Issue 陈述

Issue 陈述应凝聚前面的事实锚定、指标设定和目标澄清，成为清晰、可操作的问题陈述。

文章把 Issue 陈述分为两块：

- 问题背景：为什么要开展这个项目？来龙去脉是什么？严重性和紧急性如何？涵盖范围是什么？
- 问题现状：用数据和事实说明趋势（Trend）、实际值（Actual）和与目标的差距（Gap）。必要时可配流程图或价值流图。

### 第四阶段：对齐共识

对齐共识的核心任务，是确保所有相关方理解并认同这个 Issue。组织内部的问题定义不是一个人的事，因为后续执行会牵涉多个角色、资源和责任边界。

文章强调：定义阶段让相关方参与会增加前期时间，但一旦决策做出，执行会更快，因为大家已经理解并认同问题本身。

组织层面定义问题有三个核心原则：

1. 复杂问题，先别谈方案，先谈目标。
2. 目标不清晰，所有人努力都是瞎忙。
3. 目标不对齐，团队一定会内耗、走偏、返工。

这部分与 [[wiki/maps/Management Map]] 的组织协作问题相连：问题定义不仅是认知动作，也是一种组织协调动作。没有共识的 Issue，会在执行阶段变成返工、推诿和方案争夺。^[inferred]

### 两个天坑

#### 天坑一：急于求成，盲目寻找答案

文章用“渔夫与鱼”的例子说明跳过第一、第二阶段的风险。渔夫发现捕到的鱼变少，立刻寻找答案：去更远海域、换更大的网、延长捕鱼时间。结果越努力越糟。

最后发现，真正的问题是近海鱼苗被“绝户网”捞光，生态系统正在崩溃。

启示是：在没有明确靶心之前，不要胡乱射箭。慢就是快，定义清楚“要解决什么”，比“怎么解决”更重要。

#### 天坑二：浅尝辄止，误把现象当问题

文章用“养花与根腐病”的例子说明停留在症状层面的风险。玫瑰叶子发黄，于是施肥，结果更黄；看到土壤潮湿，于是停水，结果更蔫。她一直在治疗“叶子发黄”和“土壤潮湿”这些症状。

真正的问题是根系腐烂，原因是花盆排水孔堵塞、长期积水缺氧。

启示是：烧退了不代表病好了。必须穿透表象，找到深处持续制造麻烦的“发动机”。这与 [[wiki/concepts/Mechanism Model]] 的问题意识一致：关键不是描述事件，而是找出稳定生成事件的变量、约束、反馈和时间结构。^[inferred]

### 高阶工具：把 Issue 定义得更好

文章把“好问题”定义为不仅清晰，而且能打开思维空间的问题。由于原文中的部分图片以不可用的 data:image 占位出现，本页保留文字层面的高阶工具。

#### 四步精炼法深度版

1. 事实锚定：不带预设地描述客观差距，悬置判断，先回到事实本身。
2. 目标澄清：明确并审视目标参照系。重新定义目标，是重新定义问题的最高杠杆点。
3. 价值与意志审视：确认解决意志的深度和来源。清晰的问题定义必须包含对“为何要解决”的理解。
4. 问题陈述重构：将前三步凝聚成清晰可操作的问题陈述。

这组步骤把普通的指标化问题定义推进到更深一层：目标本身是否正确、组织是否真的愿意付出代价、价值排序是否清楚，都会决定 Issue 是否真实可解。

#### 哲学思辨的高阶应用

文章给出四类思辨动作：

- 苏格拉底式追问：连续追问“为什么这很重要？”“我们假设了什么？”
- 视角转换：用至少三种不同利益相关者视角定义同一个问题。
- 时间缩放：拉远看“一年后这个问题还重要吗？”；拉近看“24 小时内能取得什么进展？”
- 边界挑战：主动思考“这个问题不包括什么？”

这些动作的共同作用，是防止问题定义被当前视角、当前情绪、当前组织边界或当前时间尺度锁死。

### 总结：从答案思维到问题思维

文章用查尔斯·吉特林的说法概括：“只要能把问题讲清楚，问题就已经解决了一半。”

完整路径可以压缩为：

```text
日常工作中的模糊感受
→ 发现异常：用“更……”思维、批判性思维、八大浪费识别 Problem
→ 确认本质：判断这是真问题吗，是症状还是根因，值得解决吗
→ 定义议题：事实锚定，设定指标和目标，写出 Issue 陈述
→ 对齐共识：让相关方理解并认同 Issue
→ 进入分析解决阶段
```

文章最后保留两句行动提醒：

- 从“找答案”到“问对问题”：别急着给方案，先问“这到底是个什么问题？”
- 从“我觉得”到“事实是”：用数据和事实定义问题，而不是凭感觉和经验。

最后一问是一个可复用的自检句式：

```text
我们当前的目标与现状之间，存在一个 X% 的差距，
它值得解决，因为我们希望达成 Y。
```

这句话把目标、现状、差距、价值理由同时压进问题陈述中，是从 Problem 进入 Issue 的最小表达模板。

## Integration Decisions

- 这篇文章应作为 source 层素材，与 [[wiki/sources/拆解 让复杂问题变清晰 Source Guide]] 互补：前者更强调从 Problem 到 Issue 的定义链，后者更强调复杂问题拆解、MECE、逻辑树和横向/纵向展开。
- “Problem 是被发现的麻烦，Issue 是被定义好的问题”是一个可提升的概念候选，但暂时先保留在 source 层；后续如果围绕“发现和定义问题”形成更逻辑自洽的文章，可再创建 synthesis 或 topic 页面承接。
- 文章中的“问题 = 目标 - 现状”和“Issue = 目标值 - 现状值 + 衡量指标”应连接到 [[wiki/topics/Problem Framing]]，但不能替代更广义的问题定义框架；复杂系统中的问题还需要边界、角色、约束、反馈和时间。
- 文章中的组织氛围、对齐共识和系统性失明部分，应该留意与管理和组织机制的关系：很多问题发现失败不是个人不会想，而是组织实际惩罚问题暴露。^[inferred]
- 文中引用的名人语录和案例多数作为文章论证材料保存；除非后续找到原始出处，不应在稳定概念页中当作严格来源证据推广。^[ambiguous]

## Open Questions

- 后续整合两篇旧文时，“发现和定义问题”应定位为个人成长方法、组织管理方法，还是更通用的思维方法？
- Problem / Issue 区分是否需要进一步吸收咨询、项目管理、精益、软件需求工程中的术语，以避免只停留在个人定义？
- “发现异常”和“确认本质”之间是否还需要一个“收集证据 / 观察模式”的中间层，以防过早从异常跳到根因判断？
- 对齐共识是否应增加责任机制：谁拥有 Issue，谁提供数据，谁有权改变目标，谁承担解决后的验证？
- 如果组织氛围惩罚问题暴露，个人应该如何在安全边界内提出问题？
