# AI Coding 信息论框架 Source Guide

> 作者：Cheer

## 导读

为什么 Prompt 写得再细，AI 还是会输出奇怪结果？为什么新项目 AI 很好用，历史业务却总是翻车？作者从信息论出发，用“信息”和“不确定性”的底层逻辑统一解释 AI Coding 中的 Prompt、RAG、记忆、Agent 与 Harness Engineering。

## 原文保留

### 01 引言

AI Coding 的概念更新很快：Context Engineering、RAG、Agent Memory、Harness Engineering、SDD 等新词不断出现。与其逐个追逐概念，不如先问两个问题：它在解决哪一层的问题？它对当前场景的信息流动有没有实质帮助？

文章从四个问题出发：

- 为什么 Prompt 写得很细，AI 还是会输出和预期相悖的结果？
- 为什么 AI Coding 在新项目上很好用，在历史包袱重的业务上却容易卡住？
- 对话式 Agent 能否只拿一份需求文档或链接，就完成整个需求交付？
- OpenClaw、Hermes 等 Agent 的记忆越多，AI 是更准确还是更容易走神？

作者不是要把 AI Coding 严格建模成完整概率系统，而是借信息论建立坐标系：哪些优化在对抗熵增，哪些只是在占满上下文，哪些能靠工程改善，哪些必须由人介入。

### 02 信息论视角

#### 熵：可能性空间的大小

熵衡量变量的不确定性。越不可预测的事件，发生时带来的信息越多。放到 AI Coding 中，“写一个登录函数”具有很高的熵：语言、框架、认证方式和错误处理都存在大量可能。

$$H(X)=-\sum_x p(x)\log p(x)$$

目标越模糊，可能性空间越大。

#### 条件熵：看完上下文后还剩多少不确定性

条件熵 $H(Y\mid X)$ 描述已知上下文 $X$ 后，目标代码 $Y$ 仍有多少不确定性。只说“写登录函数”时，模型要猜很多；补充 Python、FastAPI、JWT、现有 `UserService` 和 `ApiError` 后，它能排除大量方向。

$$H(Y\mid X)=H(X,Y)-H(X)$$

#### 互信息：上下文排除了多少可能性

互信息 $I(X;Y)$ 衡量观测上下文后获得了多少关于目标输出的信息。

$$I(X;Y)=H(Y)-H(Y\mid X)$$

上下文有用，不是因为它多，而是因为它排除了错误方向。RAG 的价值也不是把知识库塞给模型，而是找到与当前任务互信息最高的片段。精准召回会直接暴露接口、类型和业务约定，从而减少剩余猜测空间。

#### 一个有用的分解

$$H(Y)=I(X;Y)+H(Y\mid X)$$

在 AI Coding 中：

- $H(Y)$：目标代码的全部可能性，可通过任务拆解缩小。
- $I(X;Y)$：上下文排除的不确定性。
- $H(Y\mid X)$：模型看完上下文后仍需用通用经验填补的部分。

AI 最容易出问题的地方通常在最后一块。

### 03 理想模型的两个现实修正

#### 修正一：低熵不等于正确

LLM 训练时通过最小化训练数据分布下的交叉熵，使模型分布 $P$ 逼近数据分布 $Q$；推理时给定上下文产生概率分布，再由采样策略生成输出。模型在自己的 $P$ 下高度自信，只代表某种模式在训练分布中常见，不代表它符合当前私有业务。

模型可能稳定调用 `userClient.getProfile()`，因为这个名字符合公开代码习惯；但真实项目可能只有 `profileService.fetchByUid()`，且必须带灰度参数。输出语言上顺畅、内部一致，却在业务上错误。

真正关键的是模型先验 $P$ 与业务真实约束 $Q$ 的距离。这里的 $Q$ 包括私有仓库、业务约定、运行环境、团队审美、历史包袱和测试约束。历史业务越复杂、隐性规则越多，越不能期待模型靠公开先验补齐。

两类失败由此分开：

- 输出含糊、摇摆、前后不一致：剩余条件熵过大。
- 输出稳定、自信但业务错误：模型用错误先验填补了空白。

#### 修正二：上下文并非越多越好

理想条件下，增加相关上下文不会增加条件熵：

$$H(Y\mid X,Z)\le H(Y\mid X)$$

但真实 LLM 有上下文窗口、注意力限制、检索误差，也会被冲突信息带偏。上下文过多会让有用信号被冗余淹没，也会因无关或矛盾信息引入歧义。

评估 AI Coding 优化，应同时问：

1. 它能否增加上下文与目标输出之间的互信息？
2. 它的信息密度是否足够高？

RAG 只有在召回与任务强相关时才提高单位 token 的有效信息。错误上下文比低密度上下文更危险：旧文档可能让模型更自信地走向已经废弃的规则；冲突约束会迫使模型做概率折中；扩大任务边界则直接扩大 $H(Y)$，增加自由发挥空间。

负面原则因此很直接：不要给低密度、错误或冲突上下文；不要让模型猜业务决策；不要让 Agent 擅自扩大边界；不要跳过验证继续执行。

### 04 框架总览

框架有两个独立维度。

第一层是覆盖层：显性上下文消除了多少熵？

- 核心指标：$I(X;Y)/|Context|$，即信息密度。
- 主要手段：Rules、Skills、MCP、示例、约定显性化、RAG。
- 失败表现：输出含糊、摇摆、前后不一致。

第二层是填补层：剩余的熵被正确填补了吗？

- 核心问题：模型分布 $P$ 与业务真实约束分布 $Q$ 的交叉熵或错配程度。
- 主要手段：检索业务代码、显性化私有约定、测试和 review 校准。
- 失败表现：输出稳定，却一本正经地在业务上写错。

覆盖层决定 $H(Y\mid X)$ 的大小，填补层决定模型填补剩余空白的方向。任务级上下文解决“这次用什么接口、参考哪段代码”；项目级规则解决“状态机、错误码、兼容逻辑等长期业务约定”。遇到自信写错时，继续堆任务上下文未必有效，真正需要的是把项目真实规则写进 Rules、Memory、代码检索和验证机制。

### 05 用框架看 AI Coding 问题

#### 为什么 Prompt 很细仍与预期相悖？

人的意图是高维的，“优雅一点”可能隐含不影响接口、不加依赖、少改文件、保持团队风格、性能不退化等几十个判断，而 Prompt 只能表达其中一部分。反例、Few-shot、验收标准和现有代码片段的作用，都是把隐性意图转成可见上下文。Prompt 的目标不是写得漂亮，而是最大化互信息、减少模型必须猜的部分。

#### 为什么新业务容易、历史业务难？

新项目的真实约束通常更接近公开代码模式，模型先验 $P$ 与项目真实分布 $Q$ 重合度较高。历史业务的难点不只是代码量大，而是字段不能改名、旧客户端兼容、绕路工具函数、错误码、执行顺序等隐性规则多。这些不可见知识既无法提高互信息，又让 $P$ 与 $Q$ 的差距变大。真正困难的不是让模型写代码，而是让它知道哪些地方不能按通用经验写。

#### Agent 能否只靠需求文档完成交付？

复杂业务中，中间无人介入很难稳定跑通。数据处理不等式说明，多步处理不会凭空恢复源输入未包含的信息：

$$I(X;Z)\le I(X;Y)$$

Agent 可通过代码检索、测试、日志和失败反馈补一部分覆盖层缺口，但从未被表达、也未进入测试的业务意图属于填补层缺口。系统真正需要的能力，是识别哪些缺口可自动弥补，哪些必须升级给人。

#### 记忆越多越好吗？

记忆价值不在存储量，而在能否召回少量高密度、强相关、未过期的信息。应优先考虑：

- 存什么：错误经验、业务约定、架构决策和偏好边界，而非流水账。
- 怎么检索：少量强相关优于大量弱相关。
- 何时遗忘：过期 API、废弃模式和错误经验会把模型推向旧规则，造成填补层污染。

OpenClaw、Hermes 等 Agent 的记忆竞争力，本质是信息密度和先验校准，不是谁存得多。

### 06 工程实践与角色重塑

#### Harness Engineering：把真实约束变成反馈回路

Harness 是 Agent 外循环：

```text
Harness = 计划分解 + 状态管理 + 工具编排 + 验证门控 + 反馈回路 + 回退机制 + 人机交接点
```

模型内循环是在给定上下文后预测下一个 token；Harness 外循环决定何时行动、提供什么上下文、使用什么工具、如何验证、失败后如何把现实反馈送入下一轮。它不是简单增加上下文，而是在每轮行动后持续暴露真实约束。

Prompt Engineering 主要发生在单次调用前，Harness Engineering 管理多轮行动中的信息流、状态流和验证流。测试、lint、类型检查、沙箱和 review 把一部分隐性 $Q$ 转成可执行约束，但业务直觉、架构审美和潜在跨系统耦合仍不能完全自动化。

#### 产品、设计、研发的角色变化

传统开发是需求到 PRD、设计、技术方案、代码的多次转译。每次转译都可能损失信息，但人类角色会从链路外注入用户洞察、技术判断和隐性经验。AI 让代码生成成本下降后，关键角色变成能够高密度表达意图、提供验收标准、把隐性规则变成可执行检查的人。

产品、设计、研发开始交付相似产物：设计稿和原型可以直接生成代码，减少一次转译，但性能、兼容性和可维护性等约束会被推迟，谁承担这部分 $H(Q,P)$ 差距成为新的责任问题。

一人公司减少跨角色传递，意图互信息更高、转译损耗更低；代价是单人在某些专业领域的判断深度不足，特定领域的 $H(Q,P)$ 反而可能更大。新的权衡是信息传递成本与专业深度。

### 07 结尾

面对 Context Engineering、RAG、记忆、SDD、Harness Engineering 等概念，可以先问：

1. 它是在提高覆盖层的信息密度，还是在校准填补层的方向？
2. 覆盖层有没有被无关信息稀释？填补层的业务约定是否真正显性化？

上下文不足可以补，私有业务知识可以通过检索、记忆、测试和反馈缩小差距；意图本身从未表达，就必须让人介入。

> 让模型少猜一点，让真实约束多暴露一点。这是工程维度上对抗熵增的基本逻辑。
