### 1. 原始问题链

```text
用户：设计模式和设计原则区别

回答主轴：
设计原则是判断方向；设计模式是局部解法。
原则用于判断，模式用于实现。
原则更稳定，模式更场景化。
原则不告诉你具体代码形状，模式给你结构形状。
但模式必须接受原则校验，否则就是为了设计而设计。
```

```text
用户：这个和系统思维里面的『 Iceberg Model』有什么联系

回答主轴：
Iceberg Model 是“分析问题深度”的纵轴；
设计原则 / 设计模式是“软件设计问题”放进这根纵轴后的两层产物。

如果只看代码表面，你看到的是“用了什么模式”；
往下看一层，你会看到“它在解决哪类反复出现的设计问题”；
再往下看，你会看到“背后的结构性矛盾是什么”；
最底层，你会看到“设计者相信什么样的软件才是好软件”。
```

```text
用户：信息流和状态流转的区别是什么

回答主轴：
信息流关心“谁知道了什么，以及这些信息怎样影响判断”；
状态流转关心“系统里的某个对象/任务/资源从一个状态变成另一个状态，是否符合规则”。

信息流 = 知道什么、传给谁、何时传、是否失真。
状态流转 = 现在是什么、下一步能变成什么、谁触发、是否一致。
```

### 2. 设计原则与设计模式的区别

设计原则是判断方向。SOLID、KISS、YAGNI、DRY、关注点分离等原则回答的是：这个设计应该朝哪个方向变好，职责是否清楚，依赖是否稳定，边界是否隔离，有没有过度设计，重复是不是表达了同一个知识。

设计模式是可复用结构。Strategy、Observer、Factory、Adapter、Command 等模式回答的是：这个反复出现的局部设计问题，可以用什么结构来组织对象、接口、消息和变化。

设计原则更抽象，不直接规定代码形状；设计模式更具体，已经接近代码结构。原则用于评估和约束，模式用于组织和落地。二者不是并列关系，而是“原则生成模式 / 模式服务原则”。

例如：

- 开闭原则要求新增变化时尽量不改旧代码；Strategy、Factory、Observer 等模式常常把变化变成“新增一个实现”，而不是“到处改 if/else”。
- 依赖倒置原则要求业务核心不要直接依赖具体外设；六边形架构、端口适配器、Adapter 模式可以帮助实现这种方向。
- 接口隔离原则要求调用方只依赖自己需要的能力；Facade、Adapter、Port 等结构可以把接口修剪成更稳定的契约。

常见误区：

1. 把原则当口号，只说“要符合开闭原则”，却不指出变化点、边界和抽象稳定性。
2. 把模式当目标，为了模式而模式。
3. 以为用了模式就自然符合原则；糟糕的 Factory 可能只是把混乱搬到另一个地方，过度 Strategy 也可能违反 YAGNI。

最实用的判断顺序：

```text
先问原则问题：
这里真正的变化点是什么？
哪个职责混在一起了？
哪个依赖方向不稳定？
哪个边界泄漏了？
哪个状态流转不可控？

再问模式问题：
有没有一个成熟结构，正好能把这个变化点隔离起来？
这个模式会降低复杂度，还是只是增加名词？
不用模式会怎样？用了模式后未来哪类修改会更便宜？
```

### 3. 设计原则/模式与 Iceberg Model 的映射

Iceberg Model 可以把软件设计问题分成四层：

| Iceberg Model 层级 | 软件设计里的对应物 | 典型问题 |
|---|---|---|
| 事件 Events | 某段代码用了/没用某个模式；某个类很大；某处 if/else 很多 | 这里要不要用 Strategy？ |
| 模式 Patterns | 同类设计问题反复出现；经验被沉淀成设计模式 | 这类变化是不是反复出现？ |
| 结构 Structures | 职责、边界、依赖、信息流、状态流转 | 为什么每次改这里都会牵一大片？ |
| 心智模型 Mental Models | 对复杂性、变化、抽象、责任的基本信念 | 我认为“好设计”到底是什么？ |
| 杠杆点 Leverage Points | 改边界、改依赖方向、改反馈、改目标，而不是只换写法 | 动哪里能让后续变化成本下降？ |

事件层看到的是具体代码现象：这里用了 Strategy、Factory、Adapter、Observer、依赖注入、Repository。这一层如果停住，就容易变成“背模式”。

模式层看到的是反复出现的问题：算法可替换、对象创建复杂、接口不兼容、状态变化要通知多方、请求需要封装/排队/撤销。设计模式本身正是这一层的沉淀：对反复出现的局部设计问题的结构化回应。

结构层看到的是生成问题的原因：职责混在一起、变化点没有隔离、高层业务依赖底层细节、调用方被迫依赖不需要的方法、重复表达同一知识、提前为假想需求抽象。设计原则主要在这一层起作用。

心智模型层看到的是设计者对好软件的底层信念：软件最大的敌人是复杂性失控；需求一定会变化；抽象服务真实变化点；模块边界本质上是责任边界；依赖方向决定变化传播方向；测试困难常常不是测试问题，而是设计边界问题。

因此，Iceberg Model 帮助从“这个代码现象”下潜到“生成这个现象的结构”；设计原则帮助诊断这个结构好不好；设计模式帮助在某些场景下改造这个结构。

### 4. if/else 示例：从事件层下潜到结构层

当代码里有很多 if/else 时，事件层只看到“这个函数里 if/else 很多”。

模式层会问：这类 if/else 是否经常因为不同支付渠道、不同折扣规则、不同导出格式而变化？如果是，可能对应 Strategy、Factory、Plugin 等模式。

结构层会问：真正的问题是不是变化点没有被隔离、业务规则和流程编排混在一起、调用方直接依赖具体实现、新增规则需要改旧代码。

原则层会追问：

- 单一职责：流程编排和规则计算是否应该分开？
- 开闭原则：新增规则能不能不改旧流程？
- 依赖倒置：业务流程是否依赖抽象规则接口？
- 接口隔离：不同规则是否被迫实现无关方法？

底层心智模型是：变化会继续发生，所以要把稳定部分和变化部分分开。

完整链条是：

```text
事件：if/else 膨胀
模式：类似变化反复出现
结构：变化点和稳定流程混在一起
心智模型：变化必须被隔离
杠杆点：把变化点抽成可替换协作者
可能模式：Strategy / Factory / Plugin
```

### 5. 信息流与状态流转的区别

信息流关心“事实如何变成可用认知”。它问：谁能看到什么信息，信息什么时候到达，是否完整、准确、及时，是否经过节点被压缩/过滤/扭曲，谁根据这些信息做判断，反馈有没有回到能改变行为的人那里。

状态流转关心“对象如何沿时间合法变化”。它问：当前状态是什么，允许的下一个状态是什么，谁能触发变化，触发条件是什么，中间失败怎么办，能不能重复触发，是否需要事务、幂等、补偿、重试、最终一致，不变量是否始终成立。

| 维度 | 信息流 | 状态流转 |
|---|---|---|
| 核心问题 | 谁知道什么？信息怎么影响判断？ | 什么东西从一个状态变成另一个状态？ |
| 关注对象 | 信息、信号、反馈、可见性 | 状态、实体、任务、资源、业务对象 |
| 主要风险 | 信息缺失、延迟、失真、反馈不到位 | 非法跳转、重复执行、中间失败、不一致 |
| 典型工具 | 日志、监控、指标、事件通知、报表、trace、反馈机制 | 状态机、事务、幂等、补偿、重试、事件溯源 |
| 系统思维位置 | 结构层里的信息通路和反馈通路 | 结构层里的状态变化规则和时间约束 |
| 判断标准 | 信息是否到达正确的人/模块并支持正确决策 | 状态是否按合法路径推进并保持不变量 |

### 6. 导出订单报表示例

状态流转是任务自己的生命史：

```text
PENDING -> RUNNING -> SUCCESS
                  -> FAILED
SUCCESS -> EXPIRED
```

信息流是围绕任务发生的可见性与反馈路径：

```text
用户点击导出
系统返回 taskId
后台任务更新进度
前端轮询或订阅任务状态
失败原因记录到日志
告警通知开发
用户看到“导出失败，请缩小时间范围”
```

状态流转问题包括：任务能不能从 PENDING 正确到 SUCCESS / FAILED，失败是否会落到明确终态，重复点击是否幂等。

信息流问题包括：用户知不知道任务正在运行，失败原因有没有传给用户，开发知不知道失败发生了，监控有没有看到失败率升高，业务方知不知道导出慢是因为数据量过大。

状态正确但信息流可能很差：任务失败了，数据库里状态是 FAILED，但用户页面一直转圈，开发也没告警。

信息流很好但状态可能很差：所有人都收到“任务成功”的通知，但实际上文件上传失败，状态被错误标记为 SUCCESS。

### 7. 信息流与状态流转在系统思维和软件设计三根中的位置

在 [[wiki/topics/Thinking in Systems]] 中，信息流接近反馈通道：系统如何看见自己，决策者是否能看到真实状态，系统是否能根据结果修正行为。

状态流转更接近系统动力学：存量如何变化，流入流出如何改变状态，延迟、规则、反馈如何影响状态演化。

在 [[wiki/concepts/Software Design Three Generators]] 中，信息流更靠近“接线”：模块之间怎么传递信息，同步还是异步，事件通知还是直接调用，谁订阅谁，错误怎么返回，反馈怎么传回去。

状态流转直接属于“流转”：状态有哪些，状态之间怎么转，异常路径怎么走，是否幂等，是否补偿，是否最终一致。

二者会交叉：状态变化通常会产生信息，信息又会触发新的状态变化。例如支付成功这个状态变化会产生 `PaymentSucceeded` 事件；订单系统收到这个信息后，把订单从 `WAITING_PAYMENT` 改成 `PAID`。这里 `PaymentSucceeded` 作为事件属于信息流，订单状态变化属于状态流转。

### 8. 判断当前问题属于哪一类

如果问题是“没人知道 / 知道太晚 / 知道错了”，这是信息流问题：报警没发出来，用户不知道任务失败，开发看不到错误上下文，管理层拿到滞后指标，前端不知道后端任务进度。

如果问题是“东西变成了不该变的样子 / 卡在中间 / 重复变了”，这是状态流转问题：订单已取消还能支付，支付成功但订单仍待付款，任务一直 RUNNING 不结束，重复点击导致创建多个任务，库存扣了但订单没生成。

如果问题是“因为没人知道，所以状态没被正确推进”，两者都有。例如支付系统成功扣款，但通知消息丢了，订单没变成 PAID。消息丢失是信息流断裂，订单没有变 PAID 是状态流转停滞。

最简模型：

```text
信息流：
事实发生
  -> 被观察
  -> 被编码成信息
  -> 被传递
  -> 被理解
  -> 影响判断/行动

状态流转：
对象处于状态 A
  -> 触发条件出现
  -> 执行业务规则
  -> 状态变成 B
  -> 保持不变量/一致性
```
