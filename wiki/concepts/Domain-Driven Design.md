# Domain-Driven Design

DDD 不是一组面向对象技巧，也不是“把数据库表包装成实体类”。它更像一座折算器：先锁定软件要贴合的那段业务现实，再用共享语言和清晰边界，把这段现实压成可演化的软件结构。

压成一句话：

> DDD = 让软件由业务现实驱动，并通过共享语言与清晰边界，把这段现实稳定折算成代码结构。

## 三根

把 DDD 降到最少，它靠三根独立生成器撑着：领域、语言、边界。

### 领域（Domain）

先回答软件到底在替哪段现实负责。

- 不是先看页面、表结构、接口，而是先看业务规则、状态变化、价值交换
- 驱动软件的不是技术栈，而是那段现实里的约束和承诺
- 没有领域，DDD 会退化成“带业务名词的技术设计”

### 语言（Language）

再回答团队如何共同说清那段现实。

- 统一语言不是术语表，而是让讨论、文档、测试、代码里的词指向同一个业务结构
- 语言一漂，模型就散；词同义异和词异义同都会把领域模型弄坏
- 没有语言，DDD 会退化成开发者私有黑话

### 边界（Boundary）

最后回答模型在什么范围内成立。

- 同一个词在不同上下文里可以不是同一个东西
- 边界保护职责、一致性和演化范围
- 没有边界，DDD 会退化成大一统模型，最后重新长成泥球

## 七个常见构件，怎么从三根长出来

这些构件不是“DDD 七件套”，而是三根在代码层的不同投影。

| 构件 | 主生长轴 | 它主要在保护什么 | 一句话定义 |
|------|----------|------------------|------------|
| Entity | 领域 | 业务对象的一生与状态流转 | 被领域要求持续追踪的业务对象 |
| Value Object | 语言 | 语义精度、不变量、相等性 | 靠语义而不是身份存在的概念块 |
| Aggregate | 边界 | 一致性边界和唯一入口 | 一组规则的一致性收口点 |
| Domain Service | 领域 | 跨对象但仍属领域的动作 | 不归单体对象、但明确属于领域的动作 |
| Repository | 边界 | 聚合与持久化之间的取放边界 | 聚合边界对外的取放口 |
| Factory | 语言 | 对象创建时的语义和合法性 | 让领域对象按正确语义成型的构件 |
| Domain Event | 领域 | 已发生事实的表达与传播 | 领域事实穿过边界时的表达物 |

### 可以这样记

- Entity 管“一生”
- Value Object 管“定义”
- Aggregate 管“收口”
- Domain Service 管“动作”
- Repository 管“取放”
- Factory 管“出生”
- Domain Event 管“已发生”

## DDD 在整条链上的位置

DDD 不替代业务建模，也不替代软件设计。它站在中间，做一次关键折算：

`组织现实 -> 领域模型 -> 软件结构`

放到现有几层里看：

| 层次 | 主要回答什么 | DDD 在这里的角色 |
|------|--------------|------------------|
| 业务建模 | 组织怎样创造价值，哪里要改 | 给 DDD 提供现实来路 |
| 软件分析 | 系统该承接哪些交换，边界画到哪里 | 给 DDD 锁定待开发系统的外部职责 |
| DDD | 这段业务现实如何压成模型 | 把现实折成可编码的领域结构 |
| 软件设计 | 系统内部怎么拆、怎么连、怎么流转 | 把领域模型进一步落实成实现结构 |

所以 DDD 更像一座桥：

- 往前接业务建模与系统分析
- 往后接模块设计、接口设计和流转设计

## 为什么需要模型驱动设计

DDD 要解决的关键问题，是传统开发里分析结果和代码实现之间的“致命鸿沟”。

在分析阶段，开发者会充分理解问题域，发现领域词汇、实体及其属性，把它们作为系统的关键抽象。进入设计阶段后，这些分析元素会被精炼和转换：有些会直接成为实际类，有些会退化为其他对象的属性、同义词或辅助概念，设计者还会发明新的技术类来处理持久化、事务、性能、集成等实现机制。

问题不在于分析和设计有转换，而在于转换过程常常失控。传统方法容易让分析模型和代码模型分道扬镳，原因主要有四个。

### 角色隔离造成知识流失

瀑布式流程常把业务专家、分析师、设计师和程序员分成不同角色：业务专家讲需求，分析师把需求翻译成分析模型，再把文档交给程序员实现。

这条链路越长，领域知识损耗越严重。分析师不写代码，就缺少实现约束带来的反馈；程序员没有参与建模，也不会对分析模型负责。最后代码会按程序员自己的理解重建一套实际规则，而不是延续原来的领域模型。

### 分析模型和设计模型被刻意割裂

许多传统方法强调建立“纯粹”的分析模型，只整理业务概念，不考虑技术可行性、性能、事务完整性、存储和检索。

这种模型在沟通上可能很漂亮，但程序员落地时会发现它无法直接变成可持久化、可查询、可维护一致性的软件单元。于是团队在编码阶段另起炉灶，拼出一套 ad hoc 设计。系统能跑，但代码结构已经背离分析阶段的概念。

### 技术限制推动贫血模型

在 ORM 和面向对象持久化技术不成熟时，把富领域对象直接映射到数据库成本很高。开发者往往选择更容易落地的过程式风格：类只保存数据，业务计算散落在 service、DAO 或脚本中。

这就是贫血领域模型的问题：领域对象失去行为，只剩属性；真正的业务知识没有封装在对象里，而是分散在外部流程中。分析阶段得到的“富含业务规则的对象”在实现中被降级成了数据壳。

### 脱离代码约束后模型变成画板

一旦分析模型不再受代码实现约束，它就容易退化成缺乏严谨性的沟通图。分析模型越来越接近口头业务叙事，设计模型和代码则越来越接近技术细节。

两套模型平行发展：业务人员和分析师在图上讨论“应该是什么”，程序员在代码里定义“实际是什么”。时间一长，真正运行的业务规则只存在于代码角落，分析模型则变成过期的表达材料。

## Model-Driven Design 的答案

模型驱动设计的核心主张，是放弃“分析模型 / 设计模型”的二分法，寻找一个单一模型：

- 它必须能捕捉分析阶段发现的领域核心概念
- 它必须能作为实用软件实现的设计基础
- 它必须经得起代码、测试、持久化、一致性和性能约束的反馈

在这个范式下，代码不是模型的下游翻译件，而是模型的一种精确表达。修改代码就是修改模型；模型无法编码，说明模型还没有真正成立。

所以 DDD 不是让程序员照着业务术语命名类，而是让领域语言、对象职责、边界和实现机制持续互相校准，直到同一套模型同时能解释业务、指导设计，并支撑运行中的软件。

## 方法论含义

- 看到实体、值对象、聚合这些词时，先别急着套模式，先问它们在保护哪一根
- 讨论 DDD 失败时，优先检查是领域没锁准、语言没收敛，还是边界没切开
- DDD 不是第四根，它只是把“业务现实如何进入代码”这件事系统化了
