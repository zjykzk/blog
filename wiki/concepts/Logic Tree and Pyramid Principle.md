# Logic Tree and Pyramid Principle

逻辑树和金字塔原理共享树状层级结构，但承担不同任务：逻辑树主要帮助分析者把问题向下拆开，金字塔原理主要帮助表达者把答案向上收拢，并按照读者容易理解的顺序交付结论。

> 逻辑树向下拆问题，金字塔向上收结论。

## What It Is

### 逻辑树

逻辑树把一个模糊的总问题、对象或目标，沿明确的逻辑关系逐层拆成子问题。它把原本放在工作记忆中的结构外显出来，使分析者能够检查分支是否重复、遗漏、混层，以及叶子节点能否被调查、验证或执行。

逻辑树的基本要求是：

1. 父节点与子节点具有明确关系。
2. 同层节点尽量采用同一个切分维度。
3. 子节点合起来能够解释、覆盖或实现父节点。
4. 分解最终能够导出证据、判断或行动。

因此，逻辑树不是把相关词画成树，而是“问题分解结构、层间逻辑关系和同层分类规则”的组合。^[inferred]

### 金字塔原理

金字塔原理把已经形成的中心思想放在上层，再用分组后的理由、证据和细节向下支撑。它要求信息结构符合读者的理解路径：先知道结论，再知道理由；先看到分组，再进入细节。

金字塔原理不只检查内容之间有没有层级，还引入明确的读者模型：表达是否让读者以较低的认知成本理解主张、依据和细节。^[inferred]

## Shared Structure

两者都依赖同一种父子层级结构：

- 上层节点统摄下层节点。
- 同层内容按照一致维度组织。
- 下层共同支撑、解释或展开上层。
- 横向分组尽量满足 MECE，即相互独立、完全穷尽。
- 先呈现整体结构，再进入局部细节。

因此，同一组材料既可以成为逻辑树，也可以成为金字塔；区别取决于根节点是什么、当前要解决什么任务，以及什么证据允许这棵树停止生长。^[inferred]

## Core Distinction

| 维度 | 逻辑树 | 金字塔原理 |
|---|---|---|
| 根节点 | 问题、对象或目标 | 核心结论或中心思想 |
| 主要动作 | 分解和探索 | 归纳和组织 |
| 工作方向 | 从上向下打开可能性 | 从下向上提炼结论，再自上而下表达 |
| 主要用途 | 分析、诊断、寻找原因或方案 | 汇报、写作、说服、传递判断 |
| 叶子节点 | 可调查、可验证、可执行的问题 | 支撑结论的事实、理由和细节 |
| 完成标准 | 拆得完整、能验证、能找到关键原因或行动点 | 结论明确、论据充分、符合读者理解顺序 |
| 对未知的态度 | 可以保留待验证假设 | 应区分已验证结论与尚未确认的可能性 |

逻辑树允许在分析早期保留“可能原因”；金字塔式表达则不能把未经验证的可能性包装成结论。拆得完整不等于已经得到答案，表达得漂亮也不等于分析经过验证。^[inferred]

## How It Works

### 逻辑树先横向展开，再纵向深挖

横向展开用于看全貌，避免只盯住一个局部原因；纵向深挖用于继续追问最可疑的分支，避免停留在表面现象。

常见逻辑树可以按用途分为：^[inferred]

1. **构成树**：对象由什么组成，用于盘点全貌和建立指标结构。
2. **原因树**：结果为什么发生，用于形成和验证诊断假设。
3. **方案树**：目标可以怎样实现，用于打开方案空间并比较路径。

这三类树不能混用：现象树回答“发生了什么”，原因树回答“为什么发生”，方案树回答“准备怎样改变”。^[inferred]

### 金字塔把调查结果收成主张

分析完成后，事实和证据需要向上归纳为一个中心判断，再按理由分组。读者最终看到的结构通常是：

```text
核心结论
├─ 理由 A
│  ├─ 证据 A1
│  └─ 证据 A2
├─ 理由 B
└─ 理由 C
```

读者从上向下阅读时不断追问“为什么”；作者从下向上构造时不断追问“这些材料共同说明什么”。这两个方向共同检验上下层是否真正成立。^[inferred]

## Transformation Flow

逻辑树与金字塔原理可以组成一条完整工作流：

```text
模糊问题
→ 定义问题和边界
→ 用逻辑树横向展开可能性
→ 调查并验证证据
→ 删除或修正被证伪分支
→ 提炼核心结论
→ 用金字塔组织理由和细节
→ 按读者理解路径交付答案
```

因此，可以用逻辑树发现答案，用金字塔原理交付答案。反过来，构造金字塔时如果发现下层材料无法共同支撑上层结论，说明分析还没有完成，需要返回逻辑树继续拆解或验证。^[inferred]

## Quality Checks

### 检查逻辑树

- 根问题是否被写成一句清楚的话？
- 这棵树是在盘点构成、寻找原因，还是寻找方案？
- 同层节点是否采用同一个切分维度？
- 子节点是否共同回答父节点，而不只是“看起来相关”？
- 分支是否重复或遗漏？
- 叶子节点能否被证据验证、被责任人领取或转成行动？

### 检查金字塔

- 顶层是不是一个明确主张，而不是主题名？
- 下层内容是否真正支撑上层结论？
- 同组内容是否属于同一种逻辑关系？
- 事实、推断和待验证假设是否被清楚区分？
- 读者是否必须替作者重新组织信息才能理解？
- 删除某个下层理由后，上层结论是否仍然成立？^[inferred]

## Boundaries

树状结构擅长表达层级分解，但不擅长表达多因互动、双向因果、时间延迟、状态循环和反馈回路。复杂系统中的原因经常彼此增强，强行把它们塞进单父节点的树里，会制造虚假的独立性。^[inferred]

更稳的组合是：

```text
逻辑树负责切开
→ 证据检查负责验证
→ 系统图负责重新连接
→ 金字塔负责向读者交付判断
```

这与 [[wiki/topics/Categorical Thinking]] 的边界一致：分类负责把对象切开，系统思考负责把被切开的对象重新连接；与 [[wiki/concepts/五大思维三层架构]] 的协作关系也一致：结构化思维负责外化框架，逻辑思维检查推理链，批判性思维检查证据、假设和反例。
