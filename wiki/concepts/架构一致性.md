# 架构一致性

架构一致性要求代码实现、技术架构、数据架构、应用架构、业务架构、企业战略和客户价值之间保持可追溯、可校验、可反馈的关系。

它不是“代码和文档一致”这么窄的问题，而是整个价值到实现链路是否仍然对齐。

## 双向链路

《架构师启示录》配图把链路画成两个方向：

```text
信息流的传递方向：
客户价值 -> 企业战略 -> 业务架构 -> 应用/数据/技术架构 -> 代码实现

保持一致性的方向：
代码实现 -> 应用/数据/技术架构 -> 业务架构 -> 企业战略 -> 客户价值
```

前向链路负责把意图逐层转译成实现。反向链路负责检查实现是否偏离上层目标。

## 三类实现一致性

原笔记把代码一致性拆成三类：

### 与应用架构一致

代码中的功能、配置、交互和架构风格要能对应应用架构。

如果应用架构定义了应用边界和交互方式，代码却直接跨边界访问数据或绕过接口，就会破坏一致性。^[inferred]

### 与数据架构一致

代码中的模型、分布、流转、归属和集成要能对应数据架构。

这里的重点不是“数据库表设计对不对”，而是数据责任、数据副本、数据流转和集成关系是否符合整体数据架构。

### 与技术架构一致

代码和部署、技术组件、非功能需求之间要一致。

例如，一个组件被设计为高可用链路上的关键节点，代码和部署就不能只满足普通内部工具的可靠性要求。^[inferred]

## 降低信息失真

原笔记给出的解决方向是：

- 统一模型：把各个架构之间的模型联系表达清楚。
- 统一概念：不同架构中对同一事物使用同一个概念。
- 及时反馈：代码实现发现问题后，要反馈到前置架构。

这说明架构一致性不是一次性检查，而是一条反馈回路。实现层的现实摩擦必须能够反向更新设计层，否则架构文档会变成失真的上游叙事。^[inferred]

## 与架构决策可回溯性的关系

架构一致性依赖架构决策可回溯性。原笔记指出，可回溯性带来三个价值：

- 稳定性和可维护性：后续修改时有背景，不容易改错。
- 团队内部架构知识共享与共识。
- 拥有更多当前系统上下文，可以简化设计。

因此，架构一致性的操作面不只是画图，还包括 ADR、需求矩阵、接口契约、测试、监控指标、代码边界和评审机制。^[inferred]
