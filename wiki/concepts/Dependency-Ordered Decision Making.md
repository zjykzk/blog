# Dependency-Ordered Decision Making

依赖驱动的逐项决策，是先识别决定之间的前置关系，再逐个稳定上游决定，并让每次反馈重新约束后续决策空间的方法。

## What It Is

复杂计划不是一组相互独立的问题，而是一张决策网络。上游决定会改变下游问题的必要性、候选范围和评价标准。例如，目标用户决定核心场景，核心场景决定风险边界，风险边界再约束权限、数据和技术架构。

“resolving dependencies between decisions one-by-one”包含两个不可分割的部分：

- `dependencies` 处理逻辑结构：哪些决定必须先发生，哪些决定受其约束，哪些分支会被上游选择直接排除。
- `one-by-one` 处理交互反馈：每轮只稳定一个决策节点，用最新答案重算下一步，而不是一次性提交一份静态问卷。

只有依赖关系而没有单步反馈，后续问题仍然无法动态调整；只有逐项提问而没有依赖关系，问题顺序仍可能随机，并在前提未定时过早讨论细节。

## How It Works

### 1. 区分事实与决策

能从代码库、文档或环境中查明的事实，由 Agent 主动调查；涉及目标、风险、成本、优先级和不可逆取舍的决定，由人保留裁决权。这避免 Agent 把调查劳动推给用户，也避免 Agent 把价值判断伪装成技术事实。

这一责任边界与 [[wiki/concepts/Agentic Coding Unknowns]] 相连：面对 prompt-map 与真实 codebase-territory 的缺口，Agent 应先判断是可以调查的未知，还是必须澄清的决策。

### 2. 建立决策依赖图

把待定事项视为节点，把“D2 必须以 D1 的结果为前提”视为有向边：

```text
目标用户
  → 核心任务
  → 交互与数据边界
  → 权限和风险控制
  → 技术架构
```

优先处理没有未解决前置条件、且对后续影响最大的节点。形式上，这接近决策的拓扑排序；现实设计存在反馈时，则需要允许下游可行性证据回写并修正上游决定。^[inferred]

### 3. 每次只解决一个节点

每轮提出一个会显著改变范围、架构、成本、风险、体验或可逆性的决策，并附上推荐答案、理由、主要代价以及推荐发生变化的条件。Agent 负责分析与建议，用户负责关键取舍。

用户回答后，立即更新决策图：

- 删除已经失效的分支；
- 缩小仍然成立的候选空间；
- 暴露由新决定产生的问题；
- 检查是否与既有决定冲突；
- 选择下一个已被解锁的上游节点。

这使对话成为闭环控制，而不是开环问卷：

```text
当前理解 → 选择关键问题 → 用户反馈 → 更新模型 → 重算下一问题
```

### 4. 用确认门阻止过早执行

当目标、非目标、关键约束、高影响决策、风险和未决问题已经显式化，双方再确认是否形成足够的共享理解。没有确认，就不进入执行阶段。

这延伸了 [[wiki/concepts/Design-First Collaboration]] 的原则：把分歧留在仍是语言和设计提案的低成本阶段，不让临时假设过早固化成接口、数据结构和代码。

## What Problems It Solves

### 在变化的地基上做决定

如果上游前提尚未稳定，下游决定只能依赖临时假设。一旦上游变化，整片下游设计都会失效。依赖排序让影响范围最大的决定先发生，把后期返工变成前期剪枝。

### 伪问题与无效讨论

某些问题会因为上游选择而自动得到答案或直接消失。例如“不支持多人协作”会使“选择哪种实时同步协议”失去必要性。先解决上游节点，可以避免为不存在的分支投入分析成本。

### 局部合理、整体矛盾的愿望集合

多个问题同时提出时，用户可能分别选择强一致、即时完成、离线可用、零冲突处理和零额外成本，却看不到这些选择不能同时成立。逐项处理会让每个选择的后果成为下一问的上下文，从而更早暴露约束冲突。

### 静态问卷无法适应新信息

前一个答案可能删除旧问题、改变候选项或打开新分支。一次只问一个问题，使后续访谈能够根据最新状态自适应，而不是继续执行已经过时的问题清单。

### Agent 越权或过度打扰用户

事实调查、专业建议和最终裁决被分开后，Agent 既不会什么都问，也不会什么都替用户决定。这是 [[wiki/concepts/Agent Loop Human Judgment Positions]] 在计划阶段的具体落点：人保留价值定义、边界、否决和执行放行责任。^[inferred]

## Why It Works

这个方法有效，是因为它和复杂协作的真实结构相匹配：

1. 信息分散在人与 Agent 之间，需要通过调查、建议和反馈合并。
2. 决策存在依赖，需要先稳定上游变量，再讨论下游实现。
3. 实现会放大误解，需要在修改成本较低时暴露假设和分歧。
4. 每轮反馈都会改变当前模型，因此闭环访谈比一次性开环生成更可靠。

它控制的核心不是“问题数量”，而是一个决定的后果如何向下传播：

```text
错误顺序：下游先定 → 上游改变 → 成片返工
正确顺序：上游先定 → 下游剪枝 → 只研究仍然成立的路径
```

## When to Use

适用于：

- 架构、产品、API、数据和权限设计；
- 大规模重构、迁移和难回滚的工程选择；
- 用户掌握业务目标与风险偏好，而 Agent 掌握代码事实与技术比较的协作；
- 决策错误成本明显高于前期澄清成本的任务。

不适用于：

- 明确的小范围修改；
- 低成本、易回滚、可通过小实验快速获得答案的任务；
- 紧急故障处置中必须先恢复服务的阶段；
- 可以直接从代码或文档查明、根本不需要用户裁决的问题。

## Failure Modes

- 把所有细节都升级为阻塞性决策，导致访谈替代行动。
- 用“共享理解”作为模糊终止条件，过早停止或无限追问。
- 推荐答案不写前提和代价，对用户产生锚定效应。
- 只维护问题序列，不记录已经确认的决定、理由和未决项。
- 把代码中的现状误判为设计意图，把推断伪装成事实。
- 忽视可逆性：低影响、易回滚的决定应允许 Agent 自主处理并记录。^[inferred]

## Quality Check

一次依赖驱动的逐项决策过程应能回答：

- 当前问题的上游目标和约束是什么？
- 下一个问题依赖哪些已确认决定？
- 这个答案会删除、约束或打开哪些分支？
- 哪些内容是已验证事实，哪些是推断，哪些必须由用户决定？
- 推荐方案为什么成立，它在什么条件下会改变？
- 哪些高影响、难逆转的决定仍未确认？
- 用户是否明确放行从讨论进入执行？
