# Use Case

用例不是功能清单，也不是页面、按钮、接口或数据库操作。

用例是一种从外部看系统的需求表达方式：谁在系统边界外，带着什么目标来和系统交互，系统向他承诺什么可完成的价值。

最短定义：

> 用例 = 外部执行者通过一次有边界的交互，从系统获得一个有意义结果。

## 核心问题

分析一个用例时，先问七个问题：

1. 边界：我们正在研究哪个系统或组织？
2. 执行者：谁在边界外与它交互？
3. 目标：这个执行者想完成什么？
4. 价值：成功以后，谁变得更好，为什么？
5. 粒度：这是不是一次坐下来能完成的有意义目标？
6. 范围：哪些属于这个用例，哪些只是前置条件、步骤或实现？
7. 替代：这个问题是否更适合用业务流程、用户故事、JTBD 或 API 契约表达？

## 一页审查清单

### 1. 边界

问：

- 研究对象是组织、软件系统，还是系统内部模块？
- 边界是否清楚？
- 执行者是否都在边界外？
- 我们是在从外部价值看系统，还是从内部结构看系统？

好信号：

- 能画出明确边界。
- 系统是研究对象，不是执行者。
- 用例描述系统对外承诺的价值。

危险信号：

- 内部模块被画成执行者。
- 数据库被画成执行者。
- 用例图看起来像架构图。
- 没有人说得清什么在系统内、什么在系统外。

### 2. 执行者

问：

- 谁发起这次交互？
- 谁在成功后获得价值？
- 执行者是真实角色，还是“用户”“管理员”这种含混标签？
- 它是主执行者还是辅助执行者？

好信号：

- 执行者按职责命名。
- 主执行者有清楚目标。
- 辅助执行者提供数据、支付、身份校验、通知或外部服务。

危险信号：

- “用户”代表所有人。
- “管理员”掩盖了审核员、配置员、运维员等不同职责。
- “系统”被列为执行者。
- 内部服务被误认为外部系统。

### 3. 目标

问：

- 执行者到底想完成什么？
- 这个目标对执行者是否有意义？
- 用例名是否是“动词 + 宾语”？
- 用例名是否使用业务语言，而不是实现语言？

好信号：

- 客户下单。
- 医生开处方。
- 客服处理工单。
- 财务审批报销。

危险信号：

- 订单管理。
- 数据处理。
- 点击提交。
- 用户 CRUD。
- 调用 API。
- 写入数据库。

### 4. 价值

问：

- 用例成功后，世界发生了什么变化？
- 谁因此受益？
- 系统兑现了什么承诺？
- 它支撑哪个业务结果？
- 如果删掉这个用例，会造成什么真实损害？

好信号：

- 价值能用一句话说清。
- 执行者完成后可以“满意地离开”。
- 用例能追溯到业务目标或业务指标。

危险信号：

- 价值只是“数据被保存”。
- 没有任何执行者真正关心。
- 用例存在只是因为有一个页面、菜单或表。
- 功能技术上成立，但业务上空心。

### 5. 粒度

问：

- 这是否能在一次坐下来完成？
- 它是否小到只是一个步骤？
- 它是否大到跨多人、跨多天、跨多个部门？
- 它是用户目标、子功能，还是完整业务流程？
- 是否应该把几个小操作合并成一个有意义的用例？

好信号：

- 有清楚开始和结束。
- 表达一次 actor-system 交互。
- 大到值得说明，小到可以规约。

危险信号：

- 把“登录”当完整用例。
- 每个按钮都是一个用例。
- 一个完整业务流程被压成一个系统用例。
- 用例需要多次访问、多人接力或长时间等待才能完成。

### 6. 范围

问：

- 哪些内容属于这个用例？
- 哪些内容不属于？
- 哪些只是前置条件？
- 哪些属于主成功路径？
- 哪些属于扩展路径或异常路径？

好信号：

- 前置条件没有被误认为目标。
- 登录、权限检查、输入校验通常不作为顶层用例。
- 异常路径被记录，但不抢走主目标。
- 一个用例只围绕一个执行者目标展开。

危险信号：

- 身份认证被当成独立业务目标。
- 错误处理过早变成顶层用例。
- 可选路径被散落到用例图上。
- 一个用例试图解释所有实现细节。

### 7. 与业务用例的关系

问：

- 这是业务用例，还是系统用例？
- 如果是业务用例，研究对象是哪一个组织？
- 如果是系统用例，研究对象是哪一个软件系统？
- 这个系统用例支撑哪个业务用例？
- 它来自真实业务交互，还是来自想象中的功能清单？

好信号：

- 业务用例说明组织对外价值。
- 系统用例说明软件对外价值。
- 系统用例可以追溯到业务流程、业务目标或业务序列图中的外部消息。

危险信号：

- 把“客户获得服务”直接当成系统用例。
- 把一个多日业务流程当成一次软件交互。
- 系统用例没有业务父级。
- 团队从页面出发，而不是从业务交互出发。

### 8. 与用户故事的关系

问：

- 这个内容更适合写成用户故事，还是用例？
- 我们只需要表达意图，还是需要表达交互路径？
- 一个用户故事是否隐藏了多个用例？
- 一个用例是否需要拆成多个用户故事交付？
- 验收条件是否来自主路径和扩展路径？

判断：

- 用户故事更适合计划交付。
- 用例更适合建模交互。
- 两者可以互相支撑，但不能互相替代。

### 9. 与 JTBD 的关系

问：

- 执行者真正想取得什么进展？
- 是什么情境制造了这个需要？
- 这里有什么焦虑、压力、限制或替代方案？
- 不用这个系统，是否也能完成同一个 job？
- 用例是否只捕捉了交互，却漏掉了动机？

判断：

- JTBD 解释为什么用户在乎。
- 用例解释用户怎样从系统获得价值。
- 如果只写用例，容易看见交互，漏掉情境。

### 10. 与工作流的关系

问：

- 这是单个执行者和系统的一次交互，还是多角色协作流程？
- 这个过程是否跨角色、跨部门或跨时间？
- 哪些步骤是人的工作？
- 哪些步骤是系统交互？
- 哪些工作流步骤应该拆成系统用例？

判断：

- 工作流显示交接、顺序和协作。
- 用例显示系统对外承诺的价值。
- 不要把用例图画成流程图。

### 11. 与 API 契约的关系

问：

- 这是用户目标，还是技术接口？
- 执行者是人、组织、外部系统，还是内部服务？
- 同一个用例是否可以用不同 API 实现？
- API 是支撑用例，还是被误当成用例？

好信号：

- API 能追溯到用例。
- 外部系统可以作为辅助执行者。
- 技术端点不替代用户价值。

危险信号：

- 每个接口都是一个用例。
- 用例名听起来像 HTTP 方法。
- 内部微服务被画成执行者。
- API 存在，但没有清楚的外部目标。

### 12. 图的质量

问：

- 系统边界是否清楚？
- 执行者是否在外，用例是否在内？
- 名称是否短、准、有业务意义？
- include 和 extend 是否少而必要？
- 业务人员能否看懂这张图？

好信号：

- 图简单。
- 只显示执行者、系统边界和目标级用例。
- 不展示内部设计。
- 能帮助讨论，而不是炫技。

危险信号：

- 图很聪明，但没人看懂。
- include 和 extend 到处都是。
- 图里混入页面、表、模块、API 和流程。
- 图试图一次说完所有事情。

## 快速评分

每项 0 到 2 分：

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 边界清晰度 | 不清楚或混乱 | 部分清楚 | 明确稳定 |
| 执行者质量 | 含混或错误 | 部分清楚 | 都在边界外，且角色具体 |
| 目标清晰度 | 多数是功能或操作 | 目标和操作混杂 | 执行者目标清楚 |
| 价值清晰度 | 没有价值 | 价值隐含 | 价值明确 |
| 粒度 | 过小或过大 | 不均衡 | 一次坐下来完成一个有意义目标 |
| 业务可追溯性 | 无业务来源 | 关联较弱 | 清楚支撑业务价值 |
| 范围纪律 | 混合目标、步骤和实现 | 有少量混合 | 分离清楚 |
| 图的简洁性 | 混乱 | 可用但噪音多 | 简单可读 |

分数解释：

- 0-5：还不是用例模型，可能只是功能清单、流程图或架构草图。
- 6-10：部分有用，但需要清理边界、执行者或粒度。
- 11-14：足够用于讨论和评审。
- 15-16：很强，可以进入用例规约。
