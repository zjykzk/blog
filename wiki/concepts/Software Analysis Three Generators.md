# Software Analysis Three Generators

软件分析是一门看着特别乱的手艺。

翻书架：需求工程、用例建模、领域驱动设计、事件风暴、用户故事地图、业务流程再造、数据流图、状态机、UML 九种图、C4 模型、ArchiMate、Use Case 2.0、Jobs-to-be-Done……每一派都说自己抓到了本质，每一派都有一套词汇表、一套图、一套流程。新人进来第一个反应是：这些到底是同一件事的不同说法，还是真的在干不同的事？

把这些派别摆开看，吵架的焦点永远在同一个地方：谁是老大？价值从哪来？边界在哪？——翻来覆去，就这三个词。

所以这一篇的赌注是：软件分析这一整个领域，只靠三根生成器就撑起来了。

## 三根

### 组织（Organization）

软件从来不是为"用户"写的，是为某个组织写的。

- 这个组织有个老大，老大有他的痛，有他愿意掏钱解决的度量指标。
- 愿景这件事翻译成大白话就是：找到这个老大，搞清楚他到底想让哪个数字变好。
- 没有组织，就没有"谁受益"这个原点，后面全是空转。

**判据**：任何需求都能追溯到某个老大的某个度量上。追不回去的需求是孤魂野鬼。

### 交换（Exchange）

价值不是抽象的"用户满意度"，而是一次具体的交换。

- 某个人坐在某个界面前，在分钟级的时间里，完成一件对他有用的事，然后走人。
- 业务用例是组织和组织外部执行者之间的交换，系统用例是系统和系统外部执行者之间的交换。
- 用例的粒度之所以永远在吵，是因为粒度就是交换的单位——一次交换完成一次，就是一个用例。

**判据**：这一次交互能不能独立地、完整地给执行者兑现一份价值？能兑现就是一个用例，不能就不是。

### 边界（Boundary）

"这里是我们要建的东西、那里不是"的那道线。

- 业务建模阶段，边界是组织；需求建模阶段，边界是待开发系统。
- 这根线最容易被忽略，因为它是隐形的——但每次分析失灵，几乎都是边界先崩：把系统内部实现画进了业务序列图，把组织外部的业务执行者当成了系统执行者，把产品经理的 UI 想象当成了用户的真实目标。

**判据**：线两边的东西是否有清晰的责任归属？跨线的交互是否被明确地画成了"消息"而不是"内部实现"？

三根：**组织、交换、边界**。

## 四条判据（为什么是这三根）

### 生成性

挑几个看着最不一样的现象，用这三根回推：

- **为什么需求文档总是写得像愿望清单？** 因为缺了组织这根——没锁定老大，所有人都是利益相关者，每个人的愿望都平等，最后就是平等地没用。
- **为什么 UML 有九种图？** 因为交换这根有结构：谁和谁换（用例图）、换的过程长什么样（序列图）、换完系统变成什么状态（状态图）、换的对象长什么样（类图）。九种图是交换的九个切面，不是九种平行的事。
- **为什么"这个功能到底算不算一个用例"永远有人吵？** 因为边界没锁死。边界一动，用例的粒度就动。系统边界选在 API 层还是选在 UI 层，同一个"登录"可能是一个用例，也可能是三个。
- **为什么敏捷开发里"用户故事"经常退化成任务清单？** 因为三根都松了：组织层的老大没出场、交换的价值单位被拆成了实现步骤、边界被开发任务边界取代。于是"故事"只剩"任务"。

### 最小性

拆一根看看：

- **拆掉组织**：剩下交换和边界。能画出精美的用例图和清晰的系统边界，但不知道为什么要画。这是很多技术团队的日常——模型漂亮，项目失败，因为没人追问过老大要什么。
- **拆掉交换**：剩下组织和边界。知道老大是谁、知道系统在哪一圈，但不知道系统具体干什么。这是很多战略咨询报告的日常——蓝图宏大，落不了地。
- **拆掉边界**：剩下组织和交换。知道老大要什么、知道哪些交换有价值，但分不清哪些归这个系统做、哪些归别的系统或人做。这是很多"大而全"项目的日常——什么都想管，什么都管不好。

三根都不能少。少一根，现象就塌。

### 独立性

找反例：

- 同一个组织（老大不变），可以有完全不同的交换设计。银行给老大"降低坏账率"这个数字，既可以做"贷前风控系统"，也可以做"贷后催收系统"——组织不动，交换全变。
- 同一套交换（用例清单不变），可以切不同的边界。"用户提交订单、系统扣库存、系统通知物流"这三个交换，可以全做进一个单体系统，也可以切成三个微服务——交换不动，边界全变。
- 同一条边界（系统范围不变），可以服务不同的组织。一个 CRM 系统卖给 A 公司和卖给 B 公司，边界一样，但老大不同、关心的度量不同，于是优先级和默认配置完全不同。

两两独立，都能举出真实案例。

### 预测力

用这三根推一个原始清单里没有的现象：

**AI Agent 时代的软件分析会变成什么样？**

- 组织这一根不变——总有个老大要为某个指标负责。
- 边界这一根也不变——Agent 是不是"我们要建的东西"总要划一道线。
- 但交换这一根会变：以前的交换单位是"人坐下来几分钟完成一件事"，Agent 时代的交换单位变成了"人下一个意图，Agent 自己跑几分钟到几小时，回来交付一个结果"。

预测：Agent 时代的"用例"会从"分钟级人机对话"膨胀成"小时级目标委托"。用例图不会消失，但执行者栏里会出现"另一个 Agent"这种新类型。边界争议会变得更激烈，因为 Agent 可以在运行时动态调用其他 Agent，系统边界第一次变成了流体。

这个预测已经在 2025-2026 的实践里兑现了：Anthropic 的 MCP、OpenAI 的 Agent SDK 吵的全是"Agent 算不算一个独立的系统执行者"——这正是"边界 × 交换"这两根生成器在新介质下的重新组合。

## 三根各需的思维方式

三根生成器各自要求一种根本不同的思维。用一根的脑子去想另一根的事，分析就会失真。

### 组织需要「经营思维」

本质：先锁定谁为结果负责，谁愿意为改进买单。

- **老大识别能力**：看清目标组织里谁真正对结果负责，谁不是热闹型干系人。
- **度量感**：把模糊的不满压成可改进的指标，如转化率、坏账率、交付周期、投诉率。
- **优先级判断**：知道哪些需求直接打在老大的痛点上，哪些只是局部愿望。
- **价值归因**：能回答"这件事做成了，到底帮谁把哪个数字拉动了"。

思维类型：**经营性 / 结果性 / 向上看的思维**。看的不是功能，而是责任和度量。

典型误区：把"很多人都想要"误当成"值得做"。结果列出一堆需求，却没有一个能追到老大的改进指标。

### 交换需要「价值兑现思维」

本质：把价值看成一次可完成、可感知、可交付的交换。

- **执行者视角**：站在系统外问"谁来发起这次交互，为什么愿意发起"。
- **完成感判断**：分清什么叫一次完整兑现，什么只是过程中的半步。
- **粒度感**：能把用例切在"一次坐下来换到一个结果"的尺度上，不切成任务，也不糊成一大片。
- **价值闭环感**：能检查一次交换有没有明确起点、明确结果、明确受益者。

思维类型：**交易性 / 兑现性 / 向外看的思维**。看的是这一次来回到底换成了什么。

典型误区：把价值单位拆成实现步骤。于是"提交表单""写数据库""发通知"都被写成需求，最后只剩任务，没有用例。

### 边界需要「责任划界思维」

本质：明确"这件事归谁做，不归谁做"。

- **内外感**：时刻分清系统外、系统内，组织外、组织内。
- **责任分配感**：知道哪些责任该落在待开发系统，哪些该留给人、别的系统、别的组织。
- **消息意识**：跨边界只画交互消息，不偷画内部实现。
- **克制**：忍住把所有合理的事都装进当前系统的冲动。

思维类型：**法理性 / 划界性 / 向内外分责的思维**。看的是线画在哪里，责任停在哪里。

典型误区：边界没锁死就开始谈功能细节。结果一会儿把外部执行者画成系统内部角色，一会儿又把内部算法说成外部价值。

### 汇总

| 生成器 | 思维基元 | 主要工具 | 主要敌人 |
|--------|---------|---------|---------|
| 组织 | 老大 vs 热闹 | 愿景、度量、优先级 | 谁都重要 |
| 交换 | 发起者 vs 结果 | 用例、场景、完成定义 | 只有任务 |
| 边界 | 里面 vs 外面 | 边界、责任、消息 | 什么都想管 |

三根都强的分析师很少见。多数人是一根特别强，另外两根靠方法和团队补：

- 组织强，像产品负责人或业务顾问。
- 交换强，像需求分析师或用例建模者。
- 边界强，像架构师或系统分析师。

## 与《软件方法》第 2/3/4/5 章的映射

如果用"组织、交换、边界"来重读《软件方法》，第 2 到第 5 章其实是在围着这三根轮流锁定。

| 章节 | 主要产物 | 主要锁定哪一根 | 回答的问题 |
|------|----------|----------------|------------|
| 第 2 章 愿景 | 愿景文档 | 组织 | 老大是谁？他想让哪个指标变好？ |
| 第 3 章 业务用例 | 业务用例图、业务执行者清单 | 交换 | 目标组织对外承诺了哪些价值交换？ |
| 第 4 章 业务序列图 | 现状/改进后业务序列图 | 边界 + 交换 | 组织内部如何协作完成这次交换？哪些改进该交给待开发系统？ |
| 第 5 章 系统用例 | 系统用例图、系统执行者清单 | 边界 + 交换 | 锁定待开发系统后，系统对外提供哪些交换？ |

再压一层，可以看到四章不是平铺展开，而是一个收紧边界、提纯交换的过程：

- **第 2 章先锁组织**：不先找到老大，后面的用例都会漂。
- **第 3 章先看组织对外怎么换**：从组织外视角识别业务执行者和业务用例。
- **第 4 章进入组织内部**：看业务工人和业务实体怎样协作，把"现状怎么做"和"改进后怎么做"分开。
- **第 5 章再锁系统边界**：从改进后的业务协作里，提炼"外部到待开发系统"的消息，得到系统用例。

所以第 2 到第 5 章，不是四种并列画法，而是一条分析收敛链：

**先问谁受益，再问换什么，再问线画到哪里。**

如果用三根来概括它们的重心移动，就是：

- **第 2 章**：组织
- **第 3 章**：交换
- **第 4 章**：交换 × 边界
- **第 5 章**：边界 × 交换

## 业务建模、软件分析、软件设计的关系

这三者不是并列关系，而是逐层收敛关系：从组织到系统，再到系统内部实现。

如果单独看"业务建模"这个动作，见 [[wiki/concepts/Business Modeling in Software]]。那一篇把它压成了三根：边界、协作、改进。

如果往代码层继续收缩，见 [[wiki/concepts/Domain-Driven Design]]。那一篇把 DDD 压成了三根：领域、语言、边界，并把实体、值对象、聚合、领域服务、仓储、工厂、领域事件接回这三根。

| 层次 | 关注对象 | 核心问题 | 典型产物 |
|------|----------|----------|----------|
| 业务建模 | 目标组织（系统还没上场） | 组织怎么创造价值？对外承诺哪些交换？ | 愿景、业务用例、业务序列图 |
| 软件分析 | 待开发系统（从业务里切出来） | 系统该承接哪些交换？边界画到哪里？ | 系统用例、系统边界、执行者清单 |
| 软件设计 | 系统内部结构（边界内） | 系统内部怎么拆、怎么连、怎么流转？ | 模块划分、接口契约、状态流转设计 |

如果按三根看它们的重心：

- **业务建模**：组织 + 交换
- **软件分析**：交换 + 边界
- **软件设计**：边界内的划线 + 接线 + 流转

最容易犯的错是层次穿透：

- 在业务建模里提前讨论数据库表和 API 细节。
- 在软件分析里把用例写成开发任务清单。
- 在软件设计里反向改写业务边界，把系统能力当成业务目标。

可以压成一句话：

**业务建模看组织，软件分析看系统边界，软件设计看系统内部。**

## 收束

软件分析看着有一百种方法论，骨子里就三个问题：

**谁是老大？这次交换换什么？线画在哪里？**

三个问题答清楚，任何一派的方法论都是这三根的某种排列组合。三个问题有一个没答，再漂亮的图也撑不住。

## 与"软件设计三根"的对偶

软件分析回答"要建什么"，软件设计回答"怎么建"。两者各有三根，互为上下游：

| 分析（要建什么） | 设计（怎么建） | 衔接 |
|------------------|----------------|------|
| 组织 | —— | 老大的度量决定设计优先级 |
| 交换 | 流转 | 一次用例交换在系统内部对应一串状态流转 |
| 边界 | 划线 | 系统外边界锁死后，内部的模块划线才有意义 |
| —— | 接线 | 分析不管接线；接线是纯设计决定 |

见 [[wiki/concepts/Software Design Three Generators]]。

## 方法论含义

- 看到陌生分析术语，先问它在回答哪一个问题：组织？交换？边界？
- 一个都对不上，这个术语多半是包装。
- 三根彼此独立又缺一不可——分析失灵时先检查是哪一根松了。
- 与软件设计三根（划线、接线、流转）接力：分析先锁组织/交换/边界，设计再谈划线/接线/流转。
