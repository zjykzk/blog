# Factual Memory

Factual memory is the first layer of a [[wiki/concepts/Company Brain]]: the company ability to answer what exists, what happened, where the source is, who owns it, when it changed, and how a thing works.

## What It Stores

The source defines factual memory across the traces where work actually happens:

- documents, product specs, roadmap documents, and decisions once they become artifacts
- tickets, support threads, incidents, dashboards, and open issues
- CRM notes, customer calls, renewal notes, customer commitments, and account history
- emails, meetings, messages, local notes, code, and design tools

This is not merely a shared drive, wiki, or enterprise-search chat box. Factual memory has to preserve facts across tools while staying useful in the moment work is happening.

## Boundary with Personal Memory

The source distinguishes the individual brain from the company brain. Individuals have drafts, private notes, half-formed opinions, local context, and working memory; companies have official documents, shared decisions, customer commitments, policies, projects, and operating knowledge.

A useful [[wiki/concepts/Organizational Memory]] must let individual work become shared work and shared work become institutional memory without pretending every private thought is a company record.

## Required Metadata

Factual memory needs attribution and trust signals:

- who created or modified an artifact
- who owns it now
- which person or team made it official
- whether it is current or contradicted by newer information
- whether a user can see the underlying source
- whether an answer is safe for a customer conversation or only a working hypothesis

This makes factual memory closer to governed memory infrastructure than to passive retrieval. ^[inferred]

## Experience Shape

The same factual memory should produce different answers for different roles. An IC may need implementation details and owners; a manager may need blockers, risk, and scope; a CEO may need the issue shape across customers and strategy.

The source argues that personalization is part of memory, not decoration, because factual memory must understand who is asking, what they are allowed to know, and what they are trying to do.
