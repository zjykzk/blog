# Knowledge System Construction

知识体系构建不是把资料放整齐，而是把经验外化成可复用资产，把资产组织成可导航网络，再通过实践、输出和共同体反馈持续更新。

## Context

个人知识体系经常失败在两个方向：一种是只收藏资料，缺少问题牵引和输出闭环；另一种是只做目录分类，误以为层级越完整，理解越深入。更稳的做法是把学习方法论、知识网络、知识管理和认知工程合成一个循环系统。^[inferred]

## Finding

知识体系的最小闭环是：

> 问题牵引输入 → 隐性经验外化 → 概念压缩 → 语义链接 → 场景复用 → 输出验证 → 审计重构。

这个闭环把 [[wiki/topics/Learning Methodology]] 的“从问题进、输出即学习”，[[wiki/concepts/Knowledge as Network]] 的“知识是网不是塔”，以及 [[wiki/concepts/Cognitive Engineering]] 的“人 + 工具 + 信息结构 + 反馈回路”合在一起。知识体系不是静态仓库，而是一个持续工作的认知系统。^[inferred]

## Reasoning

### 1. 从问题进，不从知识进

知识体系应先回答“我想解决什么问题”，而不是先追求覆盖完整学科。问题会决定该捕获什么、忽略什么、连接什么，以及什么时候停止补课。

这延续 [[wiki/topics/Learning Methodology]] 的基本动作：每学一段内容，先写下“我为什么要学这个？我想回答什么问题？”没有问题牵引，知识库容易退化成收藏夹。

### 2. 把知识当网，不当塔

[[wiki/concepts/Knowledge as Network]] 强调知识“局部有序，全局无层”。构建知识体系时，目录只能解决定位问题，不能自动生成理解。

真正重要的是语义连接：

- 上游：这条知识从哪里来？
- 平级：它和什么相似、冲突、互补？
- 下游：它能用于什么问题、文章、决策或行动？

Zettelkasten 的经验可以补强这一点：链接本身要说明关系，而不是只堆 wikilink。一个好链接应回答“为什么这两条知识有关”。^[inferred]

### 3. 用 SECI 处理隐性知识和显性知识

Nonaka 的 SECI 模型把知识创造看成隐性知识和显性知识之间的循环：

- Socialization：隐性知识通过共事、观察、对话传播。
- Externalization：把隐性经验写成模型、原则、案例。
- Combination：把分散材料组合成体系。
- Internalization：通过实践重新内化成能力。

用于个人 wiki 时，这个循环可以翻译为：经历 → 卡片/笔记 → 概念页/综合页 → 实践调用 → 新经验回流。它提醒知识体系不要只处理外部资料，也要处理自己的判断、失败、手感和例外。^[inferred]

### 4. 把知识做成资产，而不是资料

Davenport 与 Prusak 一类知识管理经验强调：知识只有能被找到、理解、信任、复用，才算资产。^[inferred]

因此，重要页面至少要保留五个元问题：

- 它解决什么问题？
- 它来自哪里？
- 它适用在哪里？
- 它和哪些概念相连？
- 它上次被验证是什么时候？

这和当前 wiki 的 `summary`、`sources`、`updated`、`lifecycle`、`Related` 等字段相容。它们不是文档装饰，而是知识资产的可用性条件。^[inferred]

### 5. 把知识体系嵌入工作流

很多知识库不是因为内容少而失败，而是因为不进入真实任务。知识体系必须和工作流互相绑定：

- 做项目时查 wiki。
- 解决问题后补 wiki。
- 复盘时重构 wiki。
- 输出文章、方案、教学时引用 wiki。
- 问 AI 时让 AI 先查 wiki。

只有进入工作流，知识才会被现实持续校验；否则知识库会变成死档案。^[inferred]

### 6. 用 CODE 降低日常维护成本

Tiago Forte 的 CODE 可以作为个人知识管理的轻量动作框架：

- Capture：捕获有共鸣或有用的材料。
- Organize：按项目、领域、资源、归档组织。
- Distill：提炼最关键的判断。
- Express：输出成文章、方案、决策或教学。

在这个 wiki 里，关键不是照搬 PARA 文件夹法，而是强化 Distill 与 Express：只有被提炼和表达过的知识，才真正进入长期体系。^[inferred]

### 7. 通过实践共同体保留语境

Wenger 的 Communities of Practice 提醒知识管理不能只保存定义。很多关键知识存在于共同体中的判断、惯例、例外和品味里。^[inferred]

因此，知识体系还要记录：

- 专家如何判断？
- 新手常错在哪里？
- 真实场景中的例外是什么？
- 争议点是什么？
- 哪些知识只在某个共同体里成立？

这能避免知识从实践语境中被抽干，只剩无摩擦的概念。

### 8. 定期做知识审计

知识审计把知识库从“持续堆积”拉回“持续重构”。审计时可以问：

- 哪些主题正在膨胀，需要 synthesis？
- 哪些概念重复，需要合并？
- 哪些页面很久没验证，可能过期？
- 哪些问题反复出现，说明缺一个 map？
- 哪些知识只收藏了，没有被用过？

审计的目标不是追求整洁，而是让知识体系继续服务真实问题。

## Implications

### 页面设计

一个长期知识页不应只包含定义。更好的结构是：问题、核心判断、机制、适用边界、例子、反例、相关链接、更新状态。

这与 [[wiki/concepts/Knowledge Types]] 相连：同一个主题要尽量覆盖描述性知识、解释性知识、程序性知识、判断性知识和情境性知识，而不是只保存事实。

### 维护节奏

知识体系可以分三层维护：

- 日常层：捕获和轻量整理，避免打断当前工作。
- 周期层：把散点合并为概念页、主题页或综合页。
- 审计层：删除、合并、重命名、标记过期、补 map。

### AI 协作

AI 适合帮助做结构化、链接建议、重复检测、摘要提炼和审计提示，但不应替代人的问题选择、判断承担和实践验证。AI 可以提高反馈密度，但知识体系的方向仍由真实问题和行动结果决定。^[inferred]

## Template

构建或重构一个知识主题时，可以用这组问题：

1. 这个主题正在回答什么问题？
2. 目前有哪些原始材料和个人经验？
3. 哪些经验需要外化成概念、原则或案例？
4. 哪些概念是核心节点？
5. 它们之间是什么关系：因果、类比、冲突、层级、流程、约束，还是反馈？
6. 这些知识能在哪些场景中复用？
7. 如何通过输出、项目或教学验证它？
8. 下一次审计时要检查什么？
