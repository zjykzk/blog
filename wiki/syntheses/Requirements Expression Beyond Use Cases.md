# Requirements Expression Beyond Use Cases

## Context

用例捕获的是系统对外提供的行为：谁在系统边界外发起交互，系统怎样响应，最后给执行者兑现什么结果。它擅长表达“系统做什么”，尤其适合系统用例、主成功场景、扩展场景和异常场景。

但需求不只有行为。很多真正决定架构、设计和验收的内容，不是一次执行者—系统交互，而是质量目标、业务规则、合规限制、数据约束、界面体验、外部集成、运行维护和项目条件。把这些都塞进用例，会让用例规约变成杂物间；完全不写，又会让关键架构约束在实现后期才暴露。

## Finding / Decision

需求表达应采用“用例为主线，其他需求按关注点分层补充”的结构。

```text
需求
  ├─ 行为需求：用例 / 场景 / 用户故事 / 流程
  ├─ 质量属性：性能、可用性、安全性、可维护性、可观测性
  ├─ 约束条件：法规、技术栈、预算、时间、部署环境、组织能力
  ├─ 业务规则：权限、计费、风控、审批、状态转换、不变量
  ├─ 数据需求：数据对象、字段、生命周期、保留、归属、质量
  ├─ 界面与交互：信息架构、表单、反馈、可访问性、错误提示
  ├─ 外部集成：第三方系统、协议、消息、SLA、失败策略
  └─ 运维与治理：监控、审计、备份、恢复、发布、权限治理
```

用例仍然是行为需求的骨架；其他需求不应伪装成用例，而应以各自最能被检查和验证的形式表达。

## Reasoning

### 行为需求：用例、场景和用户故事

行为需求回答：系统在边界上对谁提供什么服务。

适合表达为：

- 用例图：列出系统外执行者和系统对外服务。
- 用例规约：写触发者、前置条件、主成功场景、扩展路径、后置条件。
- 用户故事：表达角色、目标和价值。
- 业务流程 / 活动图：表达多角色、多步骤协作。

判断标准是：它是否描述一次可完成、可感知、可交付的价值交换。能构成交换的，进入用例；只是实现步骤、页面按钮或内部任务的，不应冒充用例。

### 质量属性：用质量场景表达

质量属性回答：系统要以什么质量完成行为。

“系统要高性能”“系统要安全”“系统要可用”都太空，应改写成质量场景：

```text
在什么刺激下，
哪个部分受到影响，
系统在什么环境中，
必须给出什么可度量响应。
```

例子：

- 性能：在工作日 9:00-10:00 高峰期，订单查询接口 P95 延迟不超过 300ms。
- 可用性：单个应用实例故障时，5 分钟内自动摘除并恢复容量。
- 安全性：普通运营不能查看非负责店铺的订单导出结果。
- 可观测性：每一次导出任务必须记录发起人、参数摘要、任务状态和失败原因。

质量属性往往直接影响架构，因为它们决定缓存、异步、分区、限流、隔离、监控和恢复机制。

### 约束条件：用约束清单和决策边界表达

约束条件回答：解决方案不能越过哪些边界。

常见约束包括：

- 法规与合规：数据出境、审计留痕、隐私删除。
- 技术约束：必须运行在现有云平台，必须使用既有数据库。
- 项目约束：上线窗口、预算、人力、团队能力。
- 组织约束：运维团队是否能承接某类系统，安全团队是否要求特定审批。

约束不是功能，也不是质量目标。它更像设计空间的墙：墙在哪里，架构方案的可选集合就在哪里。约束应单独列出，并标明来源、强度、可协商性和验证方式。

### 业务规则：用规则表、决策表、状态机表达

业务规则回答：系统在某些条件下必须如何判断。

它常常散落在用例步骤里，但不应该只埋在步骤中。适合表达为：

- 规则表：列出规则编号、条件、动作、例外。
- 决策表：列出多个条件组合下的结果。
- 状态机：表达对象生命周期和合法状态转换。
- 不变量清单：表达任何时候都不能被破坏的业务事实。

例子：

```text
规则：普通运营只能导出自己负责店铺的订单。
条件：user.role = operator 且 order.shopId 不在 user.shopIds 中。
结果：拒绝导出，返回无权限错误。
```

业务规则可以被用例引用，但不必被重复粘贴到每个用例里。

### 数据需求：用概念模型和数据生命周期表达

数据需求回答：系统需要管理什么信息，以及这些信息如何变化、归属和失效。

适合表达为：

- 领域词表：统一业务术语。
- 概念模型：对象、关系、关键属性。
- 数据生命周期：创建、修改、冻结、归档、删除。
- 数据质量规则：唯一性、完整性、准确性、时效性。
- 数据归属与权限：谁拥有、谁可见、谁可改。

数据需求和数据库设计不同。前者描述业务上必须理解和治理的数据，后者描述技术落地结构。

### 界面与交互需求：用任务、原型和交互状态表达

界面需求回答：人如何理解、操作和恢复。

它不应只写成“做一个页面”。更稳的表达包括：

- 用户任务：用户来到界面要完成什么。
- 信息架构：信息如何分组、排序、显示层级如何组织。
- 交互状态：空状态、加载中、成功、失败、部分成功、无权限。
- 表单规则：字段含义、校验、默认值、错误提示。
- 可访问性与易用性要求。

界面需求通常连接 [[wiki/topics/User Stories]] 和行为用例，但它关注的是人机交互的认知路径，不是系统内部实现。

### 外部集成需求：用接口契约和失败策略表达

集成需求回答：系统如何与外部系统交换信息。

适合表达为：

- 上下游系统清单。
- API / 事件 / 文件契约。
- 鉴权方式、限流、幂等、重试。
- 上游异常和下游不可用时的降级策略。
- SLA、超时、补偿和人工处理路径。

集成需求经常和系统边界混在一起。正确做法是先通过用例锁定系统对外行为，再把跨系统消息、协议和失败策略单独表达。

### 运维与治理需求：用运行手册、审计和控制面表达

运维需求回答：系统上线后如何被看见、控制、恢复和治理。

适合表达为：

- 日志、指标、链路追踪。
- 告警规则和响应等级。
- 备份、恢复、演练。
- 发布、回滚、灰度。
- 审计、权限、配置变更记录。

这类需求经常被误认为“后期再说”，但它们决定系统是否可运营。对关键系统而言，运维和治理需求应在架构阶段就进入需求矩阵。

## Implications

一个实用的需求包可以这样组织：

```text
1. 愿景：目标组织、老大、业务指标
2. 范围：组织边界、系统边界、外部执行者
3. 行为：用例图 + 用例规约 + 用户故事
4. 规则：业务规则表 + 决策表 + 状态机
5. 数据：领域词表 + 概念模型 + 数据生命周期
6. 质量：质量属性场景 + 可度量验收标准
7. 约束：法规、技术、项目、组织约束
8. 界面：任务流、原型、交互状态、错误提示
9. 集成：接口契约、消息、失败策略、SLA
10. 运维：监控、审计、备份、发布、权限治理
```

最关键的判断是：不要问“所有需求能不能都写进用例”，而要问“这个需求最自然的可验证表达形式是什么”。

- 能构成执行者与系统之间一次价值交换的，写成用例。
- 描述系统质量的，写成质量属性场景。
- 描述不可越过边界的，写成约束。
- 描述业务判断的，写成规则表、决策表或状态机。
- 描述数据事实的，写成概念模型和生命周期。
- 描述人与界面协作的，写成任务流、原型和交互状态。
- 描述系统间协作的，写成接口契约和失败策略。
- 描述上线后治理的，写成运维、审计和控制要求。

这样做能保持 [[wiki/concepts/Software Analysis Three Generators]] 中“组织、交换、边界”的主线，同时补足 [[wiki/topics/Requirement to Architecture Mapping]] 所需要的架构输入。
