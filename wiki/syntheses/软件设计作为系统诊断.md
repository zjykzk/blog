# 软件设计作为系统诊断

## Context

软件设计常被误解成“选择正确的设计模式”或“遵守若干设计原则”。这种理解停留在表层：看到 if/else 膨胀，就问是否应该使用 Strategy；看到类很大，就说要符合单一职责；看到接口混乱，就想套 Adapter 或 Facade。

更有解释力的看法是：软件设计是一种系统诊断。设计者面对的不是孤立代码现象，而是一组会反复生成复杂性、耦合、失真和不一致的结构。[[wiki/concepts/Iceberg Model]] 提供了下潜路径，[[wiki/concepts/Software Design Three Generators]] 提供了软件结构的基本切面，[[wiki/sources/设计模式 设计原则与系统思维 Source Guide]] 保存了这套理解的原始教学讨论。

## Finding

软件设计问题不应直接从“事件”跳到“模式”，而应沿着系统诊断链条下潜：

```text
代码事件
  -> 重复模式
  -> 生成结构
  -> 设计心智模型
  -> 杠杆点
  -> 可能的设计模式或架构改造
```

在这条链上：

- 设计模式更接近 Iceberg Model 的“模式层”：它们是反复出现的局部设计问题沉淀出的可复用结构。
- 设计原则更接近“结构层”的判断工具：它们用来诊断职责、边界、依赖方向、抽象稳定性、信息流和状态流转是否合理。
- 信息流和状态流转补上运行时结构：前者处理“事实如何变成可用认知”，后者处理“对象如何沿时间合法变化”。
- 高杠杆设计不是更换写法，而是改变边界、依赖方向、反馈通道、状态机和不变量位置。

一句话：

> 设计原则帮助看清结构，设计模式帮助改造局部结构，系统思维帮助判断应该下潜到哪一层再动手。

## Reasoning

### 1. 事件层：不要把代码现象直接等同于设计答案

事件层看到的是可见症状：

- 某个函数 if/else 很多。
- 某个类越来越大。
- 某个接口被很多调用方误用。
- 某个任务经常卡在 RUNNING。
- 某个状态在异常路径下变得不可修复。
- 某个系统出错后用户、开发、运营都不知道发生了什么。

这些现象是诊断入口，不是设计答案。直接从“if/else 多”跳到“用 Strategy”，会把设计模式变成条件反射。系统诊断要求先问：这个现象是否反复出现？反复出现的原因是否来自同一个结构？

### 2. 模式层：设计模式是重复设计问题的局部结构答案

设计模式不是背诵清单，而是对重复局部问题的结构回应。[[wiki/topics/Software Development Thought Lineage]] 中也把设计模式定位为：局部设计问题反复出现后，经验需要被复用，于是形成稳定结构解法。

典型对应关系：

| 反复出现的问题 | 可能的模式 | 真正要诊断的变化 |
|---|---|---|
| 算法或规则经常替换 | Strategy | 变化点是否应从流程中分离 |
| 创建逻辑复杂且多处重复 | Factory / Builder | 创建知识是否泄漏到调用方 |
| 接口不兼容 | Adapter | 边界两侧的契约是否稳定 |
| 状态变化要通知多方 | Observer / Event | 信息流是否需要解耦传播 |
| 请求要排队、记录、撤销 | Command | 行为是否需要被对象化和调度 |

模式层的关键问题不是“这个模式叫什么”，而是：

```text
这类局部问题是否反复发生？
这个模式是否正好改变了生成问题的结构？
它降低了未来变化成本，还是只是增加了抽象名词？
```

### 3. 结构层：设计原则是结构诊断工具

设计原则的价值不在口号，而在结构判据。[[wiki/maps/Software Design Map]] 把 SOLID、GRASP、KISS、YAGNI、DRY、关注点分离等放在基础设计原则层；这些原则用于判断结构是否能承受变化。

结构层要问：

- 职责是否混在一起？
- 稳定部分和变化部分是否分离？
- 依赖方向是否让低层细节污染高层业务？
- 接口是否迫使调用方知道太多？
- 重复代码是否代表同一份知识被多处表达？
- 抽象是否服务真实变化点，还是提前服务假想需求？
- 信息是否到达正确决策点？
- 状态是否沿合法路径推进并保持不变量？

因此，“原则生成模式 / 模式服务原则”比“原则 vs 模式”更准确。原则定义好结构的判断方向，模式提供某些场景下的结构实现方式。

### 4. 心智模型层：软件设计背后是对变化和责任的信念

很多设计分歧不是技术分歧，而是心智模型不同：

- 软件最大的敌人是复杂性失控，还是代码量不够少？
- 需求变化是异常，还是常态？
- 抽象是越多越好，还是只应服务真实变化点？
- 模块边界是代码分组，还是责任边界？
- 依赖方向只是 import 方向，还是变化传播方向？
- 测试困难是测试技巧问题，还是边界设计问题？

如果心智模型停留在“把这次需求写完”，设计会倾向于局部堆叠。如果心智模型转向“变化会继续发生，边界要吸收变化”，设计才会把责任、协作、信息流和状态流转作为一等对象。

### 5. 杠杆点：高杠杆设计改变生成结构

[[wiki/concepts/Leverage Points]] 与 [[wiki/topics/Thinking in Systems]] 的核心启发是：事件层修补通常只能缓解症状，结构层和心智模型层改造才可能改变生成器。

在软件设计中，高杠杆点通常是：

- 改边界：把一起变的放在一起，不一起变的分开。
- 改依赖方向：让业务核心依赖抽象，而不是依赖外部细节。
- 改信息流：让失败、进度、风险、反馈到达能行动的人或模块。
- 改状态流转：定义状态机、终态、幂等、补偿、不变量和异常路径。
- 改反馈：让测试、监控、告警、日志、用户反馈成为设计的一部分。
- 改目标：从“这次写完”改成“后续同类变化更便宜”。

## Diagnostic Template

遇到一个设计坏味道时，不先问“该用哪个设计模式”，而按下面路径诊断。

### 1. 事件：现在看见了什么？

```text
哪个代码现象最明显？
它是大类、长函数、重复代码、条件分支、循环依赖、接口混乱、状态异常，还是观测缺失？
```

### 2. 模式：它是否反复出现？

```text
类似问题过去出现过吗？
它是否总在同一类变化发生时出现？
它是规则变化、渠道变化、对象创建变化、协议变化、状态变化，还是反馈缺失？
```

### 3. 结构：什么结构在生成它？

```text
哪个职责被混在一起？
哪个边界画错了？
哪个依赖方向让变化扩散？
哪个接口暴露了不该暴露的知识？
信息有没有到达正确决策点？
状态有没有沿合法路径流转？
异常路径、重试、补偿、终态是否定义清楚？
```

### 4. 原则：哪些原则能作为判据？

```text
单一职责：这里是否有多个变化原因？
开闭原则：新增变化能否通过新增结构完成，而不是修改旧流程？
依赖倒置：高层策略是否依赖低层细节？
接口隔离：调用方是否被迫知道或实现无关能力？
DRY：重复是否代表同一知识被多处维护？
YAGNI：抽象是否服务真实变化，而不是假想变化？
KISS：复杂结构是否真的降低了整体复杂度？
```

### 5. 杠杆点：真正应该动哪里？

```text
是重新划边界？
是改接口契约？
是引入事件或回调改变信息流？
是显式化状态机？
是增加幂等、事务、补偿或终态？
是改变依赖方向？
是补测试、监控和反馈回路？
```

### 6. 模式：是否有成熟结构可用？

```text
如果需要隔离算法变化，是否使用 Strategy？
如果需要封装创建知识，是否使用 Factory / Builder？
如果需要兼容外部协议，是否使用 Adapter / Anti-Corruption Layer？
如果需要广播状态变化，是否使用 Observer / Domain Event？
如果需要封装行为并排队/重试/撤销，是否使用 Command？
如果需要隔离核心与外设，是否使用 Port / Adapter / Hexagonal Architecture？
```

## Worked Example: if/else 膨胀

```text
事件：if/else 膨胀。
模式：不同支付渠道、折扣规则、导出格式等变化反复出现。
结构：稳定流程和变化规则混在同一个函数里。
原则：单一职责要求流程与规则分开；开闭原则要求新增规则不改旧流程；依赖倒置要求流程依赖抽象规则接口。
心智模型：变化不是例外，而是系统生命周期的一部分。
杠杆点：把变化点抽成可替换协作者。
可能模式：Strategy / Factory / Plugin。
```

这个例子说明：Strategy 不是因为 if/else 多才出现，而是因为“同一稳定流程中的规则变化会持续发生”。如果变化不会持续发生，或者抽象成本高于变化成本，简单 if/else 可能更符合 KISS / YAGNI。

## Information Flow and State Transition

信息流和状态流转是结构层中最容易混淆的两个运行时切面。

信息流处理“事实如何变成可用认知”：

```text
事实发生
  -> 被观察
  -> 被编码成信息
  -> 被传递
  -> 被理解
  -> 影响判断/行动
```

状态流转处理“对象如何沿时间合法变化”：

```text
对象处于状态 A
  -> 触发条件出现
  -> 执行业务规则
  -> 状态变成 B
  -> 保持不变量/一致性
```

二者的区别：

| 维度 | 信息流 | 状态流转 |
|---|---|---|
| 核心问题 | 谁知道什么，信息如何影响判断 | 什么对象从什么状态变成什么状态 |
| 主要风险 | 不知道、知道太晚、知道错、反馈不到位 | 非法跳转、重复执行、中间失败、状态不一致 |
| 设计工具 | 日志、监控、指标、事件、报表、trace、告警 | 状态机、事务、幂等、补偿、重试、事件溯源 |
| 结构位置 | 接线、反馈通道、可见性 | 流转、时间约束、不变量 |
| 判断标准 | 信息是否到达正确决策点 | 状态是否沿合法路径推进 |

两者经常串联：支付成功会产生 `PaymentSucceeded` 事件，这是信息流；订单收到事件后从 `WAITING_PAYMENT` 变成 `PAID`，这是状态流转。消息丢失导致订单不推进时，信息流断裂和状态流转停滞同时发生。

## Implications

### 对代码 review

代码 review 不应只检查“有没有用模式”或“是否符合某原则”，而应追问：这个设计是否改变了生成问题的结构？如果只是把复杂性从一个类搬到另一个类，模式并没有起作用。

### 对架构 review

架构 review 应把边界、依赖方向、信息流和状态流转作为显式议题。很多架构风险不是组件名不对，而是状态没有终态、信息没有反馈、责任边界无法承担失败恢复。

### 对需求分析

需求分析不只要列功能，还要识别：哪些业务状态会变化，哪些角色需要知道变化，哪些反馈必须回到系统，哪些不变量不能被破坏。这连接到 [[wiki/topics/Requirement to Architecture Mapping]]。

### 对设计模式学习

学习设计模式时，应从“它解决哪类反复问题”和“它改变了哪种结构”入手，而不是从类图记忆入手。一个模式的适用性取决于它是否能降低当前结构的变化成本。
