# 类和对象结构图的建模作用

## Context

面向对象分析与设计中的类和对象结构图，容易被误解为“提前列出代码里要写哪些 class”。这种理解会把 OOAD 降级成代码组织技巧，忽略它在需求、领域、责任和协作之间承担的建模作用。

更准确的问题不是“类图怎么画”，而是：业务世界中的哪些对象值得成为稳定责任容器，它们分别维护什么规则、承担什么行为、与谁发生关系，并能否支撑后续协作和变化。 ^[inferred]

## Finding

类和对象结构图解决的是从业务理解到内部责任结构的收敛问题： ^[inferred]

> 它把业务世界压成一组稳定的责任容器及其关系，让团队看清“谁负责什么、谁依赖谁、谁和谁协作、哪里可能变化”，从而把需求行为落到可实现、可维护、可演化的软件结构上。 ^[inferred]

因此，类和对象结构图不是代码类清单，而是责任结构图。类回答“这一类东西应该长什么样、能做什么、对外承诺什么”，对象回答“某个具体个体现在是什么状态、正在响应什么消息、如何和别的对象协作”。 ^[inferred]

## Reasoning

### 1. 先解决“领域里有什么”

在 [[wiki/topics/UML Diagrams in Software Development]] 中，类图和对象图位于领域建模和对象设计环节，用来回答业务里有哪些核心概念，以及这些概念之间是什么关系。

这一步先服务理解，而不是先服务代码： ^[inferred]

- 订单、商品、支付、退款申请这些概念是否被识别出来；
- 它们之间是一对一、一对多、包含、引用，还是生命周期绑定；
- 哪些对象应该成为稳定领域对象，哪些只是临时数据；
- 哪些关系是业务事实，哪些只是某种实现便利。^[inferred]

如果没有这张结构图，系统容易退化成“页面字段 + 数据库表 + service 流程代码”，而不是一个拥有清晰业务对象的系统。^[inferred]

### 2. 更关键地解决“谁负责什么”

[[wiki/topics/面向对象分析与设计]] 把 OOAD 的三根力压成“责任、协作、抗变”。其中类图不是起点，而是责任切分后的投影。

所以类图真正要表达的不是： ^[inferred]

- 这个类有哪些属性；
- 这个类有哪些方法；
- 数据库表应该怎么包装成对象。

它真正要表达的是： ^[inferred]

- 这个责任应该归谁； ^[inferred]
- 这个对象应该维护哪些规则； ^[inferred]
- 外部对象不应该直接碰哪些内部状态； ^[inferred]
- 哪些行为应该属于这个对象，而不是散落到 service 或 controller 里。 ^[inferred]

例如 `Order` 不只是有 `id/status/amount` 的数据包。它应该维护订单状态是否合法变化、是否允许支付、取消、完成等责任。否则 `Order` 只是 DTO，真正责任全挤进 service，OOAD 的建模价值就丢了。^[inferred]

### 3. 给协作图提供“球员名单”

类和对象结构图偏静态，序列图偏动态。

[[wiki/topics/面向对象分析与设计]] 区分类和对象时指出：类提供静态结构，对象形成动态协作。[[wiki/topics/UML Diagrams in Software Development]] 则把类图用于领域建模，把序列图用于一次场景如何协作。

因此，类和对象结构图解决的是： ^[inferred]

> 系统里有哪些能接球的人？

序列图解决的是： ^[inferred]

> 一次业务发生时，球怎么传？

没有类和对象结构图，序列图会变成一堆 service 之间的调用流水账；没有序列图，类图会变成一堆静态盒子，不知道运行时如何动起来。^[inferred]

### 4. 帮助发现“边界是否切错”

类图画出来以后，最重要的不是看图漂不漂亮，而是用它诊断责任结构： ^[inferred]

- 某个类是不是责任太多； ^[inferred]
- 某些类是不是只有数据没有行为； ^[inferred]
- 两个对象之间是不是耦合太深； ^[inferred]
- 某个关系是不是把未来变化绑死了； ^[inferred]
- 继承是不是误用了，导致类型树承担了本该由组合或接口承担的变化； ^[inferred]
- service 是否变成上帝对象，领域对象只是贫血数据。 ^[inferred]

这对应 [[wiki/topics/面向对象分析与设计]] 中的“抗变”：成熟的面向对象设计不是对象多，而是变化来了以后，只有局部需要付出代价。

### 5. 它是从需求行为到代码实现之前的责任收敛点

[[wiki/topics/Software Methodology]] 的主链是：

`愿景 -> 业务用例 -> 业务序列图 -> 系统用例 -> 用例规约 -> 分析设计 -> 代码实现 -> 业务验证`

类和对象结构图大致处在“分析设计”位置。它承接前面的业务目标、系统边界和系统行为，把它们进一步落成内部责任结构。 ^[inferred]

所以它解决的是从“系统要表现出什么行为”到“内部由哪些对象负责实现这些行为”的过渡问题。 ^[inferred]

## Implications

### 使用类和对象结构图时，应优先问责任问题

一张有用的类和对象结构图，应该经得起这些问题： ^[inferred]

- 这张图中的每个对象是否都有明确责任； ^[inferred]
- 每个对象维护的状态是否和它的责任相称； ^[inferred]
- 每个关联关系是否表达真实业务关系，而不是临时实现路径； ^[inferred]
- 业务规则是否落在最有资格维护它的对象上； ^[inferred]
- 运行时协作是否能从这些对象自然展开； ^[inferred]
- 未来变化是否能被某些对象、接口或边界局部吸收。 ^[inferred]

### 类图和序列图应成对校验

类图提供责任骨架，序列图验证这个骨架是否能跑通具体场景。类图如果无法支持关键用例的消息传递，说明责任切分可能有问题；序列图如果总是绕开领域对象、集中进入 service 巨石，也说明类图只是静态装饰。^[inferred]

### 对象图仍有单独补充价值

当前 wiki 已经明确说明类图用于领域建模、类是对象的抽象、对象是运行时责任承担者，但“对象图 / 对象结构图”作为运行时实例快照的作用还没有单独展开。后续可以进一步区分类图的类型结构与对象图的实例结构：前者说明某类对象可能如何组织，后者说明某一时刻具体对象网络实际如何连接。^[inferred]
