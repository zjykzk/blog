# 信息流与状态流转设计原则

## Context

软件中的信息流和状态流转是运行时结构的两条主轴：

- 信息流关心事实、命令、事件、查询和结果如何在模块之间传播。
- 状态流转关心业务对象、任务、资源或流程如何从一个状态进入另一个状态。

如果这两条线设计不好，系统会出现典型失控：数据到处传却没有权威来源，状态到处改却没有合法迁移入口，UI、业务逻辑和数据库逻辑混在一起，一个字段同时表达事实、过程和权限。bug 表面上是“状态不对”，根因却藏在调用链、消息链、重试链和一致性边界里。

这页延续 [[wiki/sources/设计模式 设计原则与系统思维 Source Guide]] 与 [[wiki/syntheses/软件设计作为系统诊断]]，把“信息流 / 状态流转”从区别说明提升为一套可用于设计、review 和故障分析的原则清单。

## Finding / Decision

信息流设计解决“事实如何传播、谁能相信谁”的问题；状态流转设计解决“系统如何合法变化、变化后如何被追踪和恢复”的问题。

好的设计不是使用多少模式，而是让每一次状态变化都有来源、有规则、有边界、有记录、有恢复路径；让每一条重要信息都有权威来源、传播路径、解释语义和反馈去向。

## Reasoning

### 1. 先区分命令、事件、状态和查询

复杂系统的第一个分水岭，是能否把四种信息对象分开：

| 类型 | 含义 | 例子 | 设计责任 |
|---|---|---|---|
| Command | 想让系统做什么 | `PayOrder`、`CancelSubscription` | 表达意图，进入用例入口 |
| Event | 已经发生了什么 | `PaymentSucceeded`、`OrderCancelled` | 表达事实，触发投影或下游反应 |
| State | 当前处于什么阶段 | `PendingPayment`、`Paid`、`Expired` | 表达结果，约束下一步动作 |
| Query | 想知道什么 | `GetOrderDetail`、`ListSubscriptions` | 表达读取需求，不应改变状态 |

混乱常来自把这四者压到一个字段里。例如 `status = paid` 可能同时被用来表示用户点击了支付、支付渠道扣款成功、订单允许履约、发票可以生成。更好的结构是：`PayOrder` 是命令，支付渠道回调是外部事件，`PaymentSucceeded` 是领域事件，`Order.status = Paid` 是状态结果，下游履约是事件订阅或后续命令。

### 2. 状态不是普通赋值，而是合法迁移

状态字段不应被各处代码直接写入：

```text
order.status = "paid"
```

更稳的表达是：

```text
order.markAsPaid(paymentId)
OrderStateMachine.transition(order, PaymentSucceeded)
```

状态变化要回答：谁可以触发，从哪个状态可以变到哪个状态，需要满足什么条件，变化后产生什么事件或副作用，失败时如何处理，是否允许重复触发，是否可回退或只能补偿。

状态不是被“赋值”的；状态是经过规则校验后的“迁移结果”。

### 3. 单一事实源优先于多处推断

一个事实应有一个权威来源。订单是否已支付，不能同时由 `order.status`、`payment.status`、`paid_at`、回调日志和发票状态共同裁判。否则系统会进入多事实源冲突：订单显示已支付，支付记录失败，发票还在待开，用户页面又显示处理中。

合理做法是：

- 明确权威事实源。
- 把其他数据当作投影、缓存、读模型或冗余字段。
- 冗余数据必须能从事实源重建。
- 副本不能偷偷变成核心判断的裁判。

### 4. 信息流方向要稳定

信息流失控时，系统会变成网状互调：UI 写 store，store 触发 API，API 回调改 UI，service 调 repository，repository 又回调 service。表面上功能可跑，实质上没有人能追踪“事实从哪里来、在哪里被改变、谁被通知”。

更稳定的方向是：

```text
UI / API / MQ
  -> Application Service / Command Handler
  -> Domain Model
  -> Repository / Outbox
  -> Event Subscriber / Projection / Integration
```

跨层通信要通过明确接口，不要让下层偷看上层状态，也不要让外部回调直接改核心状态。

### 5. 写路径和读路径可以分离

一个模型很难同时满足写入规则、页面展示、搜索过滤、报表统计、权限裁剪和缓存优化。写模型应保护业务不变量、状态迁移和事务边界；读模型应服务展示、查询效率和用户视角。

这就是 CQRS 的基本判断：

- Command side：改变状态，维护规则。
- Query side：读取展示，优化视图。

小系统不一定需要完整 CQRS，但仍应避免让展示便利性污染核心状态模型，也避免让查询结构破坏写入规则。

### 6. 副作用必须显式化

状态流转常伴随副作用：发短信、开票、扣库存、发货、通知 CRM、写审计日志、调第三方系统。危险做法是把副作用藏在 setter、ORM hook 或 observer 里，让一句状态赋值偷偷触发一串外部动作。

更好的结构是：

```text
状态变化
  -> 产生领域事件
  -> 事件进入 outbox
  -> 订阅者处理通知、发票、履约、积分、分析
```

关键副作用要有幂等、重试、死信、补偿和审计。事务内修改本地状态，事务外处理外部副作用。

### 7. 幂等性是跨边界状态变化的基本功

只要有网络、消息队列、支付回调、定时任务和重试机制，同一个命令或事件就可能被处理多次。设计时要默认：用户会重复点击，HTTP 会超时重试，MQ 会至少一次投递，第三方会重复回调，补偿任务会重复扫描。

状态流转系统要回答：

- 同一个请求重复来，会不会重复扣款？
- 同一个事件重复消费，会不会重复发货？
- 当前状态已经完成，再执行一次会怎样？
- 是否有 `request_id`、`event_id`、`transaction_id` 去重？
- 状态迁移是否天然幂等？

凡是跨进程、跨网络、跨系统的状态变化，都默认会重复发生。

### 8. 一致性边界要先画清楚

状态不是越实时一致越好。设计时要区分：

- 强一致：同一个事务内必须同时成功或失败。
- 最终一致：可以短暂不一致，但最终会收敛。
- 外部一致：和第三方系统无法共享事务，只能靠事件、回调、补偿和对账。
- 读写一致：用户刚写完后马上读取时应该看到什么。

支付场景中，本地订单状态和本地支付记录可以放在同一个数据库事务里；支付渠道扣款和本地数据库不可能真正强一致；发票、履约、积分可能只能最终一致。不能假装所有动作都能被一个事务包住。

### 9. 状态复杂时，用显式状态机替代散落 if/else

当状态超过 3 个，且不同状态允许的动作不同，就应考虑状态机或状态迁移表。

```text
Created        --SubmitOrder------> PendingPayment
PendingPayment --PaymentSucceeded-> Paid
PendingPayment --PaymentTimeout---> Expired
PendingPayment --Cancel-----------> Cancelled
Paid           --RefundRequested--> Refunding
Refunding      --RefundSucceeded--> Refunded
```

状态机的价值不是“显得高级”，而是把状态空间、非法迁移、前置条件和迁移副作用显式化。状态机负责“能不能转、从哪转到哪”；不一定负责全部副作用。

### 10. 时间是一等设计对象

很多状态 bug 本质是时间 bug：订单 30 分钟未支付自动关闭，会员到期，优惠券过期，任务超时重试，审批 48 小时自动升级，支付回调晚于取消操作到达。

状态流转设计必须明确：

- 哪些状态会超时？
- 超时由谁检测？
- 使用定时任务、延迟队列，还是查询时惰性计算？
- 过期状态是存储出来，还是由时间动态推导？
- 迟到事件如何处理？
- 使用本地时间、服务端时间，还是事件发生时间？

凡是有状态，就迟早有时间；凡是有时间，就迟早有乱序、延迟和过期。

## 常用模式的适用边界

### 状态机 State Machine

适合订单、工单、审批、任务调度、支付、物流、用户生命周期等对象。它把状态集合、触发事件、迁移规则、守卫条件和迁移动作集中管理。风险是过度设计，不应把所有业务逻辑都塞进状态机。

### 事件驱动 Event-Driven Architecture

适合一个变化影响多个下游的场景。例如支付成功后要开票、发通知、履约、加积分、更新用户画像。事件驱动让下游解耦，但会引入顺序、重试、死信、幂等和最终一致性问题。

### CQRS

适合写入规则复杂、查询展示复杂、一个模型同时服务读写开始别扭的场景。写侧保护不变量，读侧优化页面和报表。风险是同步延迟和架构复杂度。

### Event Sourcing

适合历史事实本身是核心资产的场景，如审计、金融、协作编辑、复杂业务流程。它保存事件序列，用事件重放得到当前状态。风险是查询复杂、事件版本演化困难、团队理解成本高。Event Sourcing 是强药，不是维生素。

### Saga

适合跨多个服务的长事务。每个正向动作都要有补偿动作，例如创建订单、冻结库存、扣款、创建发货单中某一步失败后，需要释放库存、退款、取消订单。Saga 承认跨服务强事务不可得，用流程和补偿管理失败。

### Outbox Pattern

适合“本地数据库状态修改后必须可靠发布事件”的场景。在同一个数据库事务中更新业务表并写入 `outbox_events`，后台任务再发布消息。它解决数据库更新成功但消息丢失，或消息发出但数据库回滚的双写不一致。

### Inbox / 消费去重

适合 MQ 至少一次投递和消费者可能重复收到事件的场景。消费者用 `event_id + consumer_name` 记录处理状态，已经处理过的事件直接返回成功。生产端用 Outbox 保证发出，消费端用 Inbox 保证重复不伤害。

### Workflow / Process Manager

适合跨对象、跨角色、跨服务、跨时间点的业务过程，如贷款审批、入职流程、复杂履约。实体状态机管理单个对象生命周期，Workflow 管理整个过程走到哪一步、下一步找谁、哪些任务并行、哪些节点超时、如何补偿和恢复。

### Reducer / 单向数据流

适合前端或 UI 状态管理。`State + Action -> Reducer -> New State` 让状态变化可追踪、可测试、可回放，避免组件之间互相改全局状态。前端状态也应遵守状态机思想。

### Observer / Pub-Sub

适合进程内轻量通知或插件扩展。它不适合承载关键业务流程的隐式推进，因为观察者过多会导致副作用藏起来、调用链不清楚、失败处理不可见。

### Pipeline / Chain of Responsibility

适合信息需要经过多个加工阶段的场景，如请求中间件、编译器、ETL、API 网关。信息流是加工过程时用 Pipeline；状态流是生命周期变化时用 State Machine。

### Repository + Unit of Work

适合把领域状态修改和数据库细节分离。Repository 取出和保存聚合，Unit of Work 控制一次业务用例内的事务提交。业务逻辑不应散落在 SQL、Controller 和 ORM callback 里。

## Implications

### 设计时的判断顺序

1. 先判断当前对象是命令、事件、状态、查询，还是派生数据。
2. 再判断它的权威来源是谁，谁可以修改，谁只能读取。
3. 画出信息传播路径：同步、异步、可丢、可重、是否要求顺序、失败如何恢复。
4. 画出状态迁移路径：初始态、终态、合法迁移、非法迁移、权限、超时、补偿和审计。
5. 决定一致性边界：本地事务、最终一致、第三方回调、用户读写一致。
6. 决定副作用承载方式：同步调用、领域事件、Outbox、Saga、Workflow。

### 选择模式的简表

| 现象 | 优先考虑 |
|---|---|
| 只是一个字段变更 | 语义方法封装，不必上状态机 |
| 多个状态且非法迁移很多 | 状态机 |
| 一个变化影响多个下游 | 事件驱动 |
| 跨服务长事务 | Saga / Workflow |
| 写复杂、读也复杂 | CQRS |
| 历史事实比当前状态重要 | Event Sourcing |
| DB 更新后必须可靠发消息 | Outbox |
| MQ 消费可能重复 | Inbox / 幂等去重 |
| 信息经过多段加工 | Pipeline |
| UI 状态需要可预测变化 | Reducer / 单向数据流 |

### 常见反模式

1. `status` 万能化：一个状态字段同时表达支付、履约、取消、失败、权限和用户可见性。
2. Boolean 地狱：`is_paid`、`is_cancelled`、`is_refunded`、`is_expired` 组合出非法状态。
3. 外部回调直接改核心状态：缺少去重、校验、金额核对、状态迁移规则和审计。
4. 隐式状态：只靠 `paid_at`、`cancelled_at`、`shipped_at` 推导复杂生命周期。
5. 贫血状态流转：所有状态规则散落在 Service，实体只是数据袋。
6. 过度事件化：简单同步逻辑被异步事件拆散，调试和一致性成本上升。
7. 没有状态历史：当前值被改了，却无法追问谁改的、何时改的、为什么改、从哪改到哪。

### 一个可复用模板

设计一个业务对象时，可以按下面模板建模：

```text
对象：Order

状态集合：
Created / PendingPayment / Paid / Fulfilled / Cancelled / Refunded / Expired

状态迁移：
Created --Submit--> PendingPayment
PendingPayment --PaymentSucceeded--> Paid
PendingPayment --Cancel--> Cancelled
PendingPayment --Timeout--> Expired
Paid --Fulfill--> Fulfilled
Paid --Refund--> Refunded

命令：
SubmitOrder / PayOrder / CancelOrder / RefundOrder / FulfillOrder

事件：
OrderSubmitted / PaymentSucceeded / OrderCancelled / OrderExpired / OrderRefunded / OrderFulfilled

不变量：
Cancelled 不能支付
Refunded 必须曾经 Paid
Fulfilled 后不能直接 Cancel
PaymentSucceeded 金额必须等于订单应付金额

副作用：
PaymentSucceeded -> 通知、开票、履约
OrderExpired -> 释放库存
OrderRefunded -> 通知财务、更新积分

幂等键：
payment_transaction_id / command_request_id / event_id

一致性边界：
订单状态和本地支付记录：本地事务
支付渠道扣款：外部系统，最终一致
发票和履约：异步事件驱动

审计：
from_state / to_state / trigger / operator / reason / event_id / created_at
```

### 分层落点

信息流和状态流转可以落在五层结构里：

1. 输入层：接收 HTTP、MQ、Webhook、Cron、用户动作，做鉴权、参数校验和反序列化，转成 Command 或 Query。
2. 应用层：处理用例流程，控制事务边界，调用 Repository，写 Outbox，返回结果。
3. 领域层：维护业务规则、不变量、状态迁移和领域事件。
4. 基础设施层：处理 DB、MQ、Redis、HTTP Client、第三方 SDK、缓存和日志。
5. 投影/集成层：消费事件、更新读模型、触发副作用、处理重试和幂等。

这个分层和 [[wiki/concepts/Domain-Driven Design]]、[[wiki/maps/Software Design Map]] 中的应用层/领域层/基础设施层相容：应用层编排流程，领域层维护状态规则，基础设施层处理技术细节，投影层处理信息传播和读侧派生。
