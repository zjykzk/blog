# 编程语言的秩：压缩、裁决与展开

编程语言的根骨架不是语法、类型和编译器，也不是命令式、面向对象和函数式，而是压缩、裁决、展开三道前后相接的工序：把大行为压进小符号，用规则裁定符号的去路，再把符号展开成真实事件。

## Context

编程语言表面上由许多彼此竞争的机制和范式组成：函数、对象、模块、泛型、宏、类型系统、解释器、编译器、垃圾回收、所有权、协程和虚拟机各有一套术语。若只按特性分类，很难解释这些机制为什么出现、不同语言为什么反复争论相同问题，以及一个小机制为什么会牵动整门语言。

这套分析先押着三个基本假设：有限符号可以指挥机器；同一程序在说定的环境里有相对稳定的意思；人的意图至少有一部分能够写成规则。它们是讨论编程语言得以成立的前提，不是三道工序本身。

## Finding

### 压缩：让有限符号承载大于自身的行为

循环把重复的时间序列压成一段结构，函数把操作步骤压进名字，对象把状态和可用动作压进边界，泛型把跨类型重复的算法压成参数化结构，模块把内部知识压到一个较小的入口后面。高级语言首先不是“离机器更远”，而是让人能用有限注意力摆弄远大于工作记忆的行为。

这里的压缩不是减少源码字节，也不等于代码越短越好。它压缩的是描述长度、重复知识和理解一段行为时必须同时展开的细节。局部文字有时会因为良好的抽象变长，但整个系统需要重复表达和重复推理的内容会减少。

### 裁决：为压缩后的符号定准去路

压缩把许多可能性包进一句短话，也因此欠下歧义债。语法裁决符号如何组合，名字解析裁决一个名字指向哪里，类型系统裁决哪些组合被允许，作用域裁决绑定活到哪里，求值规则裁决先做谁、后做谁。

静态语言与动态语言的差别不是一个有裁决、一个没有裁决，而是何时裁决、凭什么证据裁决、失败由谁承担。编译时报错是早裁，运行时发现方法不存在是晚裁，把约定只写在文档中则把一部分裁决移到语言之外。

### 展开：把符号变成机器中的因果事件

纸上的程序不会自行改变世界。解释器、编译器、虚拟机、运行时和操作系统把符号展开成指令、状态变化、内存分配、线程调度与 I/O。展开面对的不是无限的纸面机器，而是有限内存、有限时间、缓存层级、并发交错和故障。

性能、垃圾回收、ABI、JIT、异步运行时和内存模型都从这里长出来。它们不是语言外围的杂务，而是在回答同一个问题：符号怎样在真实机器里成为事件。

## 抽象为什么会产生压缩

抽象和压缩不是两件并列的事。压缩是结果，抽象是取得这个结果的主要机制。

抽象先把多个具体实例放在一起，舍掉当前任务不关心的差异，保留会影响判断和行为的共同结构；随后给这个结构命名，把仍需变化的部分留下参数或实现插口。一个抽象因此可以向后收拢许多具体情况，也可以向前重新生成许多具体行为。

```text
多个具体实例
      |
      | 舍掉无关差异 保留共同结构
      v
一个参数化抽象
      |
      | 实例化 组合 执行
      v
多个具体行为
```

函数抽象压缩重复步骤，数据抽象压缩一组值及其合法操作，控制抽象压缩反复出现的流程，模块抽象压缩内部知识，高阶函数压缩一类控制骨架。函数、对象、接口和模块不是与“压缩”并列的第四根生成器，而是压缩在不同对象上的实现手段。^[inferred]

抽象必然有损，因为完全不丢细节就只是复制原物。好抽象的关键不是“不丢”，而是只丢当前任务不需要的差异，同时保住足以判断、组合和重新展开的结构。它可以概括为一种“任务保真的有损压缩”。^[inferred]

这个说法也划出了抽象与模糊的边界：模糊只是抹掉差异，没有说明保留什么；抽象则必须说清哪些差异被舍掉、哪些性质仍受保证、被舍掉的变化怎样从参数或实现处重新进入。

## Reasoning

三道工序不能互相顶替：

- 只压缩而不裁决，同一句符号可以任意解释，得到的是暗号而不是程序。
- 只裁决而不展开，得到的是一套形式规章或规格说明，不能在机器中发生。
- 只展开而不压缩，人只能逐个安排底层状态，无法养起大型软件。

它们也有明确先后：行为先被折进符号，规则才能对符号下判断；判断定了，机器才能把符号摊成事件。单个程序因此是一条阶段链，而编程活动还会把运行结果送回人的意图，形成下一轮修改。

```text
人的意图
   |
   | 抽象 命名 组合
   v
压缩后的符号
   |
   | 语法 名字 类型 求值规则
   v
裁决后的程序
   |
   | 编译 解释 调度 资源管理
   v
机器中的事件
   |
   | 测试 性能 事故
   v
下一轮意图
```

编程范式可以沿这条链重新理解：

- 函数式把行为压成函数组合，并限制副作用，让裁决和推理少受状态干扰。
- 面向对象把状态和动作压进责任边界，用封装控制哪些细节可以穿过边界；这与 [[wiki/topics/面向对象分析与设计]] 中“类是责任的抽象边界”相接。
- 逻辑式只写关系，把行动次序的展开交给搜索器。
- 系统语言把展开期的资源账提前搬到裁决处，因此出现生命周期、借用检查与所有权。
- 脚本语言推迟许多裁决，换来试验速度，同时让运行时承担更多意外。

所有权和异步函数说明，一个局部机制会同时改动三道工序。所有权重新表达对象关系，把资源释放提前成编译期裁决，再改变运行时资源展开；异步函数把顺序代码压成状态机，由事件决定下一段，再交给运行时分段展开。

## Failure Modes

### 没有抽象：压缩失败

复制粘贴把同一知识散成多份，程序逐渐长到人管不住。这里的问题不是字符多，而是每个副本都要单独理解、修改和验证。

### 错误抽象：把不同事物误压成同一个事物

如果共同结构并不稳定，差异会以开关、条件分支和例外不断返回。坏抽象比没有抽象更危险，因为它既隐藏真实差异，又让调用者误以为差异已经消失。

### 抽象泄漏：被压掉的细节重新影响结果

当性能、正确性或可靠性重新依赖底层细节时，原先隐藏的层会撞回调用者。这与 [[wiki/sources/Law of Leaky Abstractions Source Guide]] 的边界判断一致：抽象可以降低日常理解成本，但不会消灭底层约束。

### 裁决过早或过晚

过早裁决会拒绝本可安全表达的程序，增加表达负担；过晚裁决则把错误推到运行期、用户现场或人工约定中。语言设计的重要问题不是“要不要检查”，而是判决线放在哪里。

### 展开模型与真实机器脱节

若语言模型假装内存无限、并发有唯一顺序、网络调用和本地调用相同，性能与故障就会以意外形式回来。展开层必须承认物理资源、时间顺序和失败模式。

## Implications

判断一门新语言是否真正创新，不能只看标点、关键字或语法外观，而要看它是否改变了行为怎样被压缩、歧义在哪里被裁决、符号如何被展开。若三笔账都没有移动，它多半只是给已有结构换皮。^[inferred]

设计抽象时，可以连续追问五个问题：

1. 哪些具体实例正在被压到一起？
2. 哪些差异被舍掉，为什么当前任务不需要它们？
3. 哪些共同结构必须保留，调用者才能正确判断？
4. 哪些变化通过参数、接口或实现插口重新进入？
5. 展开时，哪些被隐藏的代价可能重新影响正确性、性能或可靠性？

这个视角也连接了“语法是否合法”和“程序能算什么”的区别：前者主要落在裁决，后者需要进入执行语义与计算能力；[[wiki/concepts/可计算性模型]] 对这两层做了更严格的形式化区分。

从认知角度看，抽象压缩只有在还能上下展开时才算形成理解。若一个人只会复述抽象名字，却不能落到实例、解释边界或重新生成行为，他拥有的是标签，不是 [[wiki/concepts/Understanding|可移动的理解结构]]。
