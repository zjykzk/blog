- [[项目/AI]] [[项目/AI]]# 用户故事到架构的标准翻译公式
    你可以在技能里直接加一条规则：
  
      先把用户故事翻译成：入口、用例、业务规则、数据能力、外部依赖、错误语义、领域对象，再映射到分层。
  
    对应关系是：
  
    • 入口 → api
    • 用例 → service/service.go
    • 业务规则 → service
    • 领域对象 → service/<entity>.go
    • 数据能力 → service/*_repo.go
    • 外部依赖能力 → service/*_repo.go
    • 技术实现 → infra/...
    • 协议错误码 → api/error_code.go
    • 业务错误 → service/errors.go
  
    建议把用户故事规范成这个格式
  
      user_story: 作为商家，我希望创建订单时自动校验库存，这样可以避免超卖
      actor: 商家
      goal: 创建订单
      value: 避免超卖
      entry:
        protocol: rpc
        handler: CreateOrder
      business_rules:
	- 下单前必须校验库存
	- 库存不足时拒绝创建订单
	  data_access:
	- 保存订单
	  external_calls:
	- 查询库存
	  domain_objects:
	- Order
	  errors:
	- InventoryNotEnough
	  shared_utils:
	- id generator
	  
	  然后技能输出：
- ## User Story Mapping Report
- ### User Story
	- As a 商家, I want to 创建订单 with inventory validation, so that 超卖 can be avoided
- ### API Layer
	- Handler: CreateOrder
	- Protocol: rpc
	- Service Interface:
		- CreateOrder(ctx, req)
	- Error Codes:
		- ErrCodeInventoryNotEnough
- ### Service Layer
	- Use Case:
		- CreateOrder
	- Domain Objects:
		- Order
	- Business Rules:
		- validate inventory before order creation
		- reject when inventory is insufficient
	- Business Errors:
		- ErrInventoryNotEnough
- ### Repo Interfaces Defined by Service
	- OrderRepo:
		- SaveOrder
	- InventoryRepo:
		- CheckInventory
- ### Infra Implementations
	- infra/mysql/order implements OrderRepo
	- infra/rpc/inventory implements InventoryRepo
- ### Pkg Utilities
	- idgen
	  
	  一个很好记的判断法
	  拿到用户故事后，逐句问：
	  
	  • 这句话是在说怎么接请求吗？→ api
	  • 这句话是在说业务要怎么判断/怎么流转吗？→ service
	  • 这句话是在说系统需要什么外部能力吗？→ service 定义接口
	  • 这句话是在说能力怎么落地实现吗？→ infra
	  • 这句话只是无业务语义的通用工具吗？→ pkg
	  
	  最容易犯的错
	  
	  • 把“库存校验规则”放进 infra：错，规则属于 service
	  • 把“查库存 RPC client”放进 pkg：错，外部依赖实现属于 infra
	  • 把 InventoryRepo 定义在 infra：错，接口应由 service 定义
	  • 把 handler 直接依赖 MySQL repo：错，api 应该只依赖 service interface
	  
	  你可以直接给技能加一句规则
	  
	  •
	  “当输入是用户故事时，先将其归一化为：角色、目标、价值、入口、业务规则、数据访问、外部依赖、领域对象、错误语义，再做分
	  层映射与架构校验。”

- ## Free-Form Requirement Normalization
  
  If the user says something short such as:
  
  • "add create order API"
  • "support querying user quota"
  • "add inventory deduction flow"
  
  First expand it into:
  
  • entry protocol
  • request/response contract
  • business rules
  • persistence needs
  • external dependencies
  • domain objects
  • error cases
  • shared utility needs
  
  If some parts are missing, infer conservatively and explicitly mark them as assumptions.
- ## Example Requirement Mapping
- ### Input
  
    feature: Create Order
    entry:
      protocol: rpc
      handler: CreateOrder
    input:
	- user_id
	- items
	- amount
	  output:
	- order_id
	- status
	  business_rules:
	- amount must be greater than 0
	- inventory must be sufficient
	- single order amount cannot exceed 10000
	  data_access:
	- save order
	  external_calls:
	- deduct inventory
	- publish order-created event
	  domain_objects:
	- Order
	- Item
	  errors:
	- InvalidAmount
	- InventoryNotEnough
	- AmountLimitExceeded
	  shared_utils:
	- id generator
- ### Expected Mapping
  
  • api/
    • defines CreateOrder handler
    • defines service interface used by handler
    • defines API error codes
  • service/
    • defines Order domain object
    • implements CreateOrder use case
    • enforces amount and inventory rules
    • defines OrderRepo, InventoryRepo, EventRepo
    • defines business errors
  • infra/
    • MySQL adapter implements OrderRepo
    • RPC/HTTP adapter implements InventoryRepo
    • MQ adapter implements EventRepo
  • pkg/
    • provides ID generator only if it is a pure utility
- ## Output Requirements
  
  When validation is complete, always return:
  
  1. Pass / Fail status
  2. Requirement Mapping Report
  3. Layer Structure result
  4. Interface Placement result
  5. Dependency Rule result
  6. Requirement Mapping Consistency result
  7. Violation list with:
    • file path
    • violation type
    • reason
    • suggested fix
