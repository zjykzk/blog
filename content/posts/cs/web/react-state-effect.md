## 缘起

React 出现的背景，是前端页面从静态文档逐渐演化为富交互应用。评论、聊天、表单联动、局部刷新这些需求越来越多，前端开发不再只是修改几个 DOM 元素，而是在持续维护一个会变化的界面状态。

在直接操作 DOM 的年代，常见写法是：找到节点、修改文本或样式、绑定事件，然后在不同地方手动同步界面。界面简单时问题不大，但一旦状态变多，代码很容易变成一堆零散的命令：A 变了要更新 X，B 变了要隐藏 Y，C 和 D 同时变了还要补更新 Z。代码可以工作，但很难维护，因为状态和界面之间缺乏一个稳定的抽象。

React 试图解决的核心问题是：不要继续手工告诉浏览器“怎么一步步改页面”，而是直接描述“当前状态下页面应该长什么样”。它最重要的心智模型可以概括成一句话：

> UI = f(state)

也就是说，界面是状态的函数。只要状态确定，界面就应该确定。

## React 的核心概念

如果只保留最关键的几个概念，React 可以概括成：组件、props、state、声明式渲染、单向数据流。

### 组件

组件是 React 组织界面的基本单位。一个按钮、一个搜索框、一张用户卡片、一个评论列表，甚至整个页面区域，都可以被拆成组件。

组件解决的问题是复杂 UI 的组织和复用。没有组件时，页面结构、交互逻辑、样式调整会混在一起；有了组件后，界面可以被拆成职责更清晰的小块，并且可以组合成更大的页面。

### Props

Props 是父组件传给子组件的输入。可以把组件看成函数，props 就像参数。它解决的是组件之间如何传递数据的问题。

Props 的关键价值在于让组件可配置、可复用。组件不必写死，而是根据传入的数据渲染不同结果。

### State

State 是组件内部会变化的数据，例如输入框内容、是否展开、当前选中项、请求返回结果等。State 解决的问题是：当数据变化时，界面如何自动更新。

在 React 中，通常不是手工去改页面，而是更新 state，然后让 React 基于新的 state 重新计算 UI。

### 声明式渲染

React 强调声明式编程。开发者不必一条条描述 DOM 更新步骤，而是直接表达“当前状态下应该渲染什么”。

这种方式解决的是界面更新逻辑过于琐碎、容易遗漏的问题。相比手工维护 DOM 细节，声明式写法更容易阅读，也更容易推理。

### 单向数据流

React 中数据通常从父组件流向子组件。父组件通过 props 向下传递数据，子组件通过回调把事件或意图通知上来。

它解决的是数据流向混乱的问题。谁提供数据、谁负责修改数据会更加清楚，这在复杂界面中尤其重要。

这几个概念里面，最容易被混淆、也是最影响代码质量的，不是组件或者 props，而是 state、effect 以及派生值之间的边界。后面的讨论基本都围绕这条主线展开。

## 为什么 State 和 Effect 不一样

State 和 Effect 经常一起出现，所以很容易被误认为是一类东西。但它们其实解决的是两类完全不同的问题。

State 表示组件“知道了什么”，是描述界面的数据；Effect 表示组件“需要去做什么”，是组件在渲染之后与外部世界发生的同步动作。

可以先记一个最短版本：

- State 是用来描述 UI 的数据。
- Effect 是用来和 React 外部系统同步的动作。

例如，当前计数值、输入框内容、当前 tab 是什么，这些都属于 state，因为它们会直接影响渲染结果。React 看见 state 变化后，会重新渲染组件，得到新的 UI。

而发网络请求、绑定浏览器事件、设置定时器、修改 `document.title`、操作第三方组件实例，这些都不属于“计算 UI”的一部分，而是和 React 外部系统交互，所以属于 effect。

为什么一定要区分？因为 React 想保留一个足够简单的模型：渲染阶段负责根据 state 和 props 计算 UI；effect 阶段负责把 React 和外部环境同步起来。前者本质上是计算，后者本质上是动作。

如果把数据和动作混在一起，代码就会出现几个问题：

1. 不知道哪些值真正参与了渲染。
2. 不知道哪些逻辑会产生外部影响。
3. 不知道一段代码是不是可以安全重跑。
4. 不知道什么时候应该清理副作用。

所以，区分 state 和 effect，不只是为了给概念分类，而是为了让组件保持可预测。React 的理想状态是：渲染尽量纯，副作用尽量少，而且副作用只在该出现的地方出现。

换句话说，state 解决的是“界面应该长什么样”，effect 解决的是“渲染完成后需要和外部世界保持什么关系”。

这也是后面两个问题的起点：为什么很多 effect 其实不需要写，以及为什么派生值不应该被当成真正的 state。

## 为什么你可能并不需要 Effect

React 官方强调 “You might not need an Effect”，本质上是在提醒：`useEffect` 不是一个通用的“只要值变了就执行代码”的工具。

`useEffect` 真正适合做的事情，是在渲染完成后，把 React 和外部系统同步起来，比如：

- 发请求
- 建立和清理事件订阅
- 操作浏览器标题
- 与定时器、本地存储、第三方库同步

如果一个逻辑并没有和外部系统打交道，那它很可能不应该进入 effect。

很多初学者会把 effect 当成一个“响应变化的收纳箱”，只要某个值变了，就写一个 effect 去更新另一个值。但这种写法往往是在用副作用修补本来可以直接表达清楚的数据关系。

一个典型错误是用 effect 计算派生数据，例如把 `firstName` 和 `lastName` 拼成 `fullName`，然后通过 effect 再去 `setFullName`。这样做的问题至少有三个：

1. 增加了一份冗余状态。
2. 会多触发一轮渲染。
3. 容易出现同步错误或短暂的旧值。

更自然的方式是直接在渲染中计算：`const fullName = firstName + " " + lastName`。

另一个常见误用是把由用户事件直接触发的逻辑放进 effect。比如点击提交按钮后弹提示，这种行为更适合写在事件处理函数里，而不是先改一个状态，再让 effect 监听这个状态去做事。因为这是用户动作的直接结果，不是“渲染后与外部系统同步”的问题。

还有一类常见情况，是通过 effect 去“修正”另一个 state，例如列表变化后再用 effect 重新挑选默认项。这通常意味着 state 设计得太多、太散，组件先进入一个不一致状态，再靠 effect 补救。代码虽然能跑，但已经偏离了 React 想鼓励的表达方式。

所以，React 说“你可能并不需要 Effect”，并不是在否定 effect 的价值，而是在强调一个顺序：

1. 先看这个值能不能直接从 props 或 state 算出来。
2. 再看这段逻辑是不是某个用户事件直接触发的。
3. 只有前两者都不是，并且确实在与外部系统同步时，才应该使用 effect。

也可以换成一句更短的话：effect 是留给副作用的，不是留给一切“随着变化而运行的代码”的。

这个判断一旦建立起来，很多组件会明显变简单，因为大量不必要的中间状态和同步逻辑都会消失。

## 为什么派生状态不是状态

派生状态（derived state）指的是那些可以从已有的 props 或 state 直接计算出来的值，例如：

- `fullName = firstName + lastName`
- `isEmpty = list.length === 0`
- `visibleItems = items.filter(...)`
- `totalPrice = items.reduce(...)`

这些值看起来也会变，但它们不是独立事实，而是已有事实的计算结果。

这是理解 React 时最容易模糊的一条边界。因为从直觉上看，派生值“也会随着数据变化”，于是很容易被误放进 `useState`。但从系统建模的角度看，state 和派生值承担的角色不同：

- state 是原始事实，是最小真相来源；
- 派生值是对事实的解释、投影或者汇总。

例如，用户刚输入了什么、当前是否展开、请求结果是什么，这些通常是 state，因为不能单靠别的值直接推出。而 `fullName`、`isEmpty`、`buttonDisabled`、`visibleItems` 这类值，则更像是已有事实的计算结果。

如果把派生值也存成 state，就会制造两份真相来源。比如既保存 `firstName` 和 `lastName`，又保存 `fullName`，那么 `fullName` 就需要依赖额外的同步逻辑才能与前两者保持一致。代码会变得更复杂，而且更容易出现某一份数据滞后一拍的问题。

这类问题常见的坏处包括：

1. 重复存储同一份语义信息。
2. 通过 effect 同步时会增加额外 render。
3. 容易产生中间错误状态或短暂旧值。
4. 数据依赖关系变得更难理解。

所以，一个很重要的原则是：能在渲染时直接计算出来的值，就不要存成 state。只有当一个值需要脱离源值独立演化时，它才应该成为真正的 state。

例如，`user.name` 直接展示时，`fullName` 只是派生值；但如果某个编辑器组件接收 `initialTitle` 作为初始值，并允许用户在本地继续修改，那么这个本地 `title` 就已经不是简单的派生值了，而是一个独立 state。

这也是为什么 `useMemo` 和 `useState` 容易被误用到一起。某个值如果计算昂贵，可以用 `useMemo` 缓存结果，但这仍然不是 state。`useMemo` 解决的是性能上的重复计算，不是数据真相来源的问题。

从这个角度看，React 里一个很重要的设计原则就是：事实应该尽量少，派生应该尽量直接。事实越多，组件越难维护；派生越绕，effect 就越容易被滥用。

所以，派生状态不等于状态。更准确地说，state 是事实，派生值是事实的投影。事实应该存下来，投影应该算出来。

## 结论

理解 React，关键不在于背 API，而在于建立一个稳定的心智模型。

- State 是独立事实，是驱动 UI 的最小真相来源。
- 派生值应该在渲染阶段直接计算，而不是额外存储。
- Effect 只负责与 React 外部系统同步，不应该成为通用逻辑收纳箱。

如果用一句话概括：

> State 决定画什么，Effect 决定做什么。

再进一步说：

> 事实应该存下来，投影应该算出来，外部同步才交给 Effect。

这套边界一旦清楚，很多 React 代码会自然变得更简单，也更容易维护。
