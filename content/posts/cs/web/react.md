## 缘起

React 出现之前，前端开发正在经历一个明显变化：网页不再只是静态文档，而是逐渐变成了富交互应用。聊天、评论、表单联动、局部刷新、动态列表这些需求越来越多，前端需要处理的不再只是展示内容，而是持续维护一个会不断变化的界面。

在这种背景下，直接操作 DOM 的方式开始暴露出问题。开发者往往需要自己记住：某个状态变化以后，要更新哪些节点；某个按钮点击以后，要隐藏哪些区域；某个请求返回以后，要把哪些部分重新渲染。界面简单时，这种方式还可以工作；但当页面复杂以后，代码就会变成大量零散的命令式更新，状态和界面之间也越来越难保持一致。

React 试图解决的，正是这种复杂界面下的维护问题。它不再鼓励开发者直接告诉浏览器“怎么一步步改页面”，而是先描述“当前状态下页面应该长什么样”，再由框架负责把结果更新到页面上。

React 最核心的思想可以概括为一句话：

> UI = f(state)

也就是说，界面是状态的函数。状态确定以后，界面也就确定了。

## React 的核心概念

React 的价值不在于某一个单独 API，而在于一组彼此配合的概念。它们共同解决的是：如何组织复杂 UI、如何让数据和界面保持一致、以及如何管理界面更新。

### 组件（Component）

组件是 React 的基本单位。一个按钮、一个搜索框、一张用户卡片、一个评论列表，甚至整个页面区域，都可以被建模成组件。

组件解决的问题，是复杂 UI 的组织和复用。没有组件时，结构、逻辑和局部交互很容易混在一起；有了组件后，页面可以被拆成很多职责清晰的小块，每一块都只关心自己的输入和输出。

组件的意义不只是“可复用”，更重要的是它给 UI 提供了一种清晰的组织方式。页面不再是一整块模板，而是由很多可以组合的小单元构成。

### JSX

JSX 是 JavaScript 的语法扩展，它允许在 JavaScript 中直接写类似 HTML 的结构。

```jsx
const element = <h1>Hello</h1>;
```

JSX 解决的问题，是界面结构表达不直观。如果只用原始 JavaScript 去描述界面树，会写出很多嵌套的函数调用，阅读成本很高。JSX 让开发者可以更直接地表达“界面长什么样”，从而更自然地把 UI 写成状态到结构的映射。

它的价值不在于“像 HTML”，而在于它让声明式 UI 变得可读、可写。

### 声明式编程

React 强调声明式编程。开发者不必手工描述每一次 DOM 修改的过程，而是直接表达：在当前状态下，应该渲染什么。

例如，与其写一堆“如果登录了就隐藏这个按钮、显示那个区域”的 DOM 操作，不如直接根据状态返回对应的界面结构。

声明式编程解决的问题，是界面更新逻辑容易分散、容易遗漏。它把注意力从“如何一步步更新页面”转移到“当前状态对应什么界面”。这样代码更容易阅读，也更容易推理。

### State

State 是组件内部会变化的数据，例如输入框内容、是否展开、当前 tab、当前计数值、请求返回结果等。

State 解决的问题，是界面如何随着数据变化自动更新。在 React 中，开发者不再直接去改页面，而是修改 state，然后让 React 基于新的 state 重新计算界面。

因此，state 的意义在于建立“数据变化”和“界面变化”之间的稳定关系。只要 state 改变，UI 就会跟着变化。

### Props

Props 是组件之间传递数据的方式，通常由父组件传给子组件。可以把组件看成函数，props 就像函数参数。

Props 解决的问题，是组件如何接收输入以及如何被复用。一个组件如果完全写死，就很难在不同场景中复用；而通过 props，同一个组件可以根据不同输入渲染出不同结果。

### 单向数据流

React 中数据通常从父组件流向子组件。父组件通过 props 向下传递数据，子组件通过回调把事件或意图通知上来。

单向数据流解决的问题，是数据流向混乱。如果任意组件都能随意修改别的组件依赖的数据，系统会变得很难理解。单向数据流让数据来源和修改路径更加清晰，也更容易定位问题。

### 虚拟 DOM

虚拟 DOM 是 React 在内存中维护的一份界面结构表示。当 state 变化时，React 会生成新的虚拟 DOM，与旧版本比较，然后把必要的变化更新到真实 DOM 上。

虚拟 DOM 解决的问题，不只是性能，更重要的是更新管理。开发者不需要自己精确控制哪些节点该改、哪些属性该更新，而是把“最终界面应该是什么样”告诉 React，再由 React 来完成更新。

### 生命周期 / Hooks

真实应用不仅需要渲染 UI，还需要处理请求、订阅、定时器、浏览器 API 等外部交互。早期 React 用类组件的生命周期方法来管理这些逻辑，后来通过 Hooks 把这类能力带入了函数组件。

例如：

- `useState` 用于管理状态
- `useEffect` 用于处理副作用

Hooks 解决的问题，是如何在函数组件中组织状态和副作用逻辑，同时提升逻辑复用能力。它让 React 组件不再局限于“接收输入，返回 JSX”，而是可以更自然地管理行为。

### 受控组件

在表单场景中，React 往往把输入框的值放到 state 中，由 state 统一控制表单内容。

受控组件解决的问题，是表单状态难管理。通过把输入值纳入 React 的状态系统，校验、联动、提交和重置都会更清晰，也更可预测。

### Context

Context 是跨层级共享数据的机制，适合传递主题、语言、当前用户、全局配置等信息。

它解决的问题，是 props 层层透传。如果某个值需要从顶层一路传到很深的子组件，中间很多组件只是帮忙转发，这会让代码变得冗余。Context 让共享数据可以直接被需要它的组件获取。

### 组合优于继承

React 倾向于通过组合来复用界面能力，而不是依赖传统面向对象中的继承层次。

它解决的问题，是 UI 复用方式过于僵硬。通过把布局、卡片、按钮、侧边栏等组件拼接组合起来，界面可以更灵活地构造，也更符合前端实际开发方式。

## React 本质上解决了什么问题

如果把前面的概念再压缩一下，React 主要解决的是三类问题。

### 1. 复杂 UI 的组织问题

通过组件化，把页面拆成可组合、可复用、职责清晰的单元。

### 2. 数据和界面的同步问题

通过 state、props 和单向数据流，让界面始终是数据的映射，而不是靠零散的 DOM 操作来勉强维持一致。

### 3. 界面更新的管理问题

通过声明式渲染和虚拟 DOM，开发者不需要手工维护大量更新细节，只需要描述目标界面。

## React 的位置

React 更准确的定位，是一个用于构建用户界面的库，而不是一个完整框架。

它主要关注 UI 渲染层。路由、状态管理、服务端渲染、数据获取、工程化构建等能力，通常依赖 React 生态中的其他工具一起配合，例如 React Router、Redux、Next.js、TanStack Query、Vite 等。

所以更准确地说，React 提供的是 UI 开发的核心抽象，而 React 生态才构成一个完整的前端应用开发方案。

## 结论

React 的核心贡献，不只是提供了组件这种写法，而是把前端开发从“手工操作页面”转向了“根据状态声明页面”。

它通过组件、state、props、声明式渲染、单向数据流、虚拟 DOM 等概念，共同解决了复杂 UI 的组织、数据同步和更新管理问题。

如果用一句话概括：

> React 的核心不是教你怎么改 DOM，而是让你描述在某个状态下，UI 应该是什么。
