#+title: Agent Memory 的秩
#+date: [2026-06-23 Tue 00:40]
#+filetags: :rank:agent:memory:

谈 agent memory，先把天花板说清：一次 LLM 调用不会自己把上次经历带进来，能影响这次行动的，只有这次上下文里重新出现的东西；上下文有限，世界和用户却一直在变。还得再信一件事：过去有些结构会在未来重来，否则记忆没有用，存档就够了。

这个领域看上去满桌都是东西。短期记忆、长期记忆、情节记忆、语义记忆、程序记忆、用户画像、反思笔记、向量库、知识图谱、会话摘要、技能文件、检查点；再往工程里走，还有 embedding、rerank、时间衰减、去重、冲突消解、权限、来源、遗忘、回滚。若把这些都当成 memory 的组成部分，清单会越列越长，而且每出一种新产品，清单还得添一行。

其实底下只有三个动作：留下痕迹，让痕迹在需要时重新出现，再让现实结果回来改写痕迹。agent memory 的秩是三。

第一根是“留下可寻址的痕迹”。模型眼前的一次观察，本来会随着调用结束而消失。要跨过这道断裂，就得把它变成调用之外仍然存在、以后还能找到的东西。重点不在“存”，而在“有地址”：它属于谁，发生在什么时候，来自哪里，在哪个任务和权限范围内成立，凭什么值得相信。文本文件、数据库、向量库、图谱、事件日志，只是给地址选了不同的材料。没有这根，agent 每次醒来都是新人；有存储却没有地址，则只是把遗忘换成了仓库失火般的堆积。

第二根是“按眼前任务选择性重现”。旧事留在硬盘里不会影响模型，只有被带回当前上下文，才重新有因果力。可上下文装不下全部过去，于是读记忆不是“搜到相似内容”这么简单，而是拿当前目标当线索，在 token、时间和注意力预算里，挑出最可能改变这一步判断的那几条。embedding、关键词、图遍历、query rewrite、recency、importance、rerank、摘要和分层索引，都是这道门的不同门锁。长上下文只把门厅修大，并没有替你决定谁该进来。召回率很高也未必好：十条相关旧事把一条关键约束淹没，记得更多，反而做得更差。

第三根是“让结果反过来改写记忆”。记下来的不是事实本身，只是当时留下的一份判断。用户会变，环境会变，agent 自己也会拿到新证据。一次行动成功或失败之后，旧痕迹要被加固、降权、合并、拆开、改正，或者删除；许多相似情节还会被压成一个事实、一条策略、一项技能。反思、consolidation、语义化、程序化、偏好更新、时间衰减、遗忘和回滚，都从这里长出来。没有这根，memory 不是学习，而是把昨天的误会长期保存。

三根不是并排放着的三根柱子，也不是走完就结束的三段流水线。它们咬成一个环：行动留下痕迹，任务线索把少量痕迹带回眼前，带着这些痕迹做出的行动又产生新结果，新结果再回去改旧痕迹。memory 因而不是“过去的容器”，而是“过去争取下一次行动控制权的制度”。写入规则管谁有资格留下，读取规则管谁此刻有资格发言，改写规则管谁还能继续被相信。

这也解释了那些看似额外的麻烦。多 agent 共享 memory，马上需要 owner、namespace 和权限，因为一条记忆跨过了人的边界；memory poisoning 特别危险，因为错误一旦被召回并指导行动，行动结果又可能被错误的评价器解释为成功，于是坏记忆自己喂大自己；企业 memory 必须保存来源和决策理由，因为一句脱离来路的“事实”一旦进入行动，就会冒充组织共识。真正成熟的 memory 系统迟早会长出审计、回滚和回归测试，不是工程师爱加治理，而是第三根必然要求系统分清“新”与“真”。

沿这三根还能推出一个有点反直觉的结论：最好的 memory 未必存得最多，也未必答得最快。它会主动漏掉无关的过去，会在证据变弱时忘记，会拒绝把一次偶然成功升格成永久规则。人常把遗忘看成 memory 的缺陷；对 agent 来说，不能有根据地忘，才是缺陷。没有遗忘，过去会慢慢占领现在。

所以一句话收口：agent memory，就是让过去留下地址，让有用的过去在当下重新出现，再让现实有权重写过去。
